# v1.1 Research: Sentry React 19 ErrorBoundary capture (issue #17)

## TL;DR

`Sentry.ErrorBoundary` alone does NOT reliably capture render-phase throws on
React 19. Official Sentry docs require pairing it with `Sentry.reactErrorHandler()`
wired into `createRoot`'s `onCaughtError` / `onUncaughtError` / `onRecoverableError`
hooks — without those, React 19 routes caught render errors through its own
default handler (console-only) and Sentry's transport never sees them. Fix: add
the React 19 hooks to `createRoot`, keep `ErrorBoundary` for fallback UI, add a
manual `Sentry.captureException` belt in `onError`.

## Root cause(s)

1. **Missing React 19 root error hooks (primary).** React 19 added
   `onUncaughtError` / `onCaughtError` / `onRecoverableError` on `createRoot`.
   Sentry's official React 19+ guidance is to wire `Sentry.reactErrorHandler()`
   into all three; `ErrorBoundary` is documented as "scoped fallback UI" only
   in 19+, no longer the global capture path. `src/main.tsx` passes NO options
   to `createRoot`, so caught render errors hit React's default handler
   (console-only, no envelope). Matches Phase 6 empirical finding (issues
   WTCS-COMMUNITY-POLLS-2/6: only `setTimeout`/`onerror` mechanisms land,
   never `react.errorboundary`).
2. **Dev-vs-prod rethrow asymmetry (contributing).** In dev/StrictMode React
   rethrows caught errors to the global handler — `globalHandlersIntegration`
   catches that. Production builds don't rethrow, so without `onCaughtError`,
   nothing reports. Phase 6 ran on a Netlify deploy-preview production build.
3. **No belt in current `ErrorBoundary` props (contributing).** No `onError`
   callback calling `Sentry.captureException` directly — no defense in depth.

No v9→v10 ErrorBoundary breaking change (v10 migration only retypes
`componentStack`). This is a React 19 integration gap, not a v10 regression.

## Recommended fix

Replace the `createRoot` + `ErrorBoundary` block in `src/main.tsx`:

```tsx
import { createRoot, type ErrorInfo } from 'react-dom/client'
import * as Sentry from '@sentry/react'

const container = document.getElementById('root')
if (!container) throw new Error('Root container missing')

createRoot(container, {
  onUncaughtError: Sentry.reactErrorHandler((error, info: ErrorInfo) => {
    if (import.meta.env.DEV) console.warn('[sentry] uncaught', error, info.componentStack)
  }),
  onCaughtError: Sentry.reactErrorHandler(),
  onRecoverableError: Sentry.reactErrorHandler(),
}).render(
  <StrictMode>
    <Sentry.ErrorBoundary
      fallback={<AppErrorFallback />}
      showDialog={false}
      onError={(error: unknown, componentStack: string, eventId: string): void => {
        Sentry.captureException(error, {
          tags: { boundary: 'app-root', eventId },
          contexts: { react: { componentStack } },
        })
      }}
    >
      {/* ...existing tree... */}
    </Sentry.ErrorBoundary>
  </StrictMode>,
)
```

The `onError` belt may dupe in dev/StrictMode; Sentry's default `Dedupe`
integration suppresses. `Sentry.init` unchanged.

## Smoke verification

Throw from RENDER, not from an event handler. Add a dev-only component, mount
it behind `?sentry-render-smoke=1` on a public route:

```tsx
// src/components/debug/RenderThrowSmoke.tsx — dev / preview-only
export function RenderThrowSmoke(): never {
  throw new Error('RenderThrowSmoke: deliberate render-phase throw for Sentry verification')
}
```

Pass criteria (all three must hold on a Netlify deploy preview, production
build, NOT dev):
1. `AppErrorFallback` renders.
2. Sentry receives an event with `mechanism.type === 'react.errorboundary'`
   OR `mechanism.type === 'generic'` from `reactErrorHandler` (NOT
   `auto.browser.global_handlers.onerror`).
3. `componentStack` is populated and `tags.boundary === 'app-root'`.

Wire as Playwright e2e: navigate to `/?sentry-render-smoke=1`, assert fallback
copy, then poll Sentry's `search_events` MCP for the matching event by
release SHA.
