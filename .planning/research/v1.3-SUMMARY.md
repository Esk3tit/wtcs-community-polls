# Research Summary — v1.3 Hygiene & Performance

**Project:** WTCS Community Polls
**Domain:** Carry-forward debt closure + aggressive perf-budget pass on a shipped Vite 8 + React 19 SPA (Supabase backend)
**Researched:** 2026-05-14
**Confidence:** HIGH

---

## Executive Summary

v1.3 is a hygiene-and-performance milestone with an unusual characteristic: the majority of its "implement" work has already landed in source code and only requires empirical verification and GitHub issue closure. Issues #17 (Sentry React 19 ErrorBoundary), #19 (Rolldown sourcemap function names), and E2E specs #11/#12/#13 all have committed fixes in `main`; the v1.3 task for these five issues is smoke-verify on a deploy preview, capture evidence, and close the GitHub issues — not new implementation. This reframes roughly half the milestone as "verify + close" work, which changes phase sizing dramatically compared to a typical feature milestone.

The single true high-risk item is DB hygiene: migration 14 must add `SET search_path = ''` to 7 pre-Phase-11 `SECURITY DEFINER` functions, and critically, every function body must be audited for unqualified table references before the migration is applied. `increment_vote_count` has a bare `INSERT INTO vote_counts` (not `public.vote_counts`) that will 42P01-error with `search_path = ''`, silently breaking every vote submission in production. `is_current_user_admin()` gates every admin RLS policy and must receive a body-identical rewrite — logic changes here propagate to all admin-sensitive tables simultaneously. The migration must use `CREATE OR REPLACE FUNCTION` (not `ALTER FUNCTION` alone), be tested locally against a fresh `supabase start`, and be followed by a `submit-vote` smoke round-trip before declaring closure.

The one true performance win for UIDN-02 is lazy-loading `posthog-js` (~180-200 KB out of the critical path). `posthog-js/module.full.js` is 420 KB unminified and is currently eagerly imported at module load even though PostHog is GDPR opt-IN and does nothing until consent. A dynamic `import('posthog-js/react')` inside `ConsentProvider` defers this off the initial parse entirely — no new dependency, just a code change. Two perf work items are explicitly rejected as anti-features: font subsetting (Inter is already a system-font-only stack; adding `@fontsource/inter` would introduce a web-font download where none exists today, hurting LCP) and critical CSS extraction (Tailwind v4 + Vite for an empty-root SPA emits only used utility classes; no above-fold content exists to extract critical CSS from). The effective perf scope is: bundle audit + PostHog lazy-load + `manualChunks` config + wtcs-logo.png WebP conversion + route `defaultPreload: 'intent'`.

## Key Findings

### Recommended Stack

The v1.1/v1.2 stack is locked and unchanged. v1.3 adds exactly one new devDependency: `rollup-plugin-visualizer@7.0.1` for bundle audit. All other v1.3 work is code changes, config changes, SQL migration, and documentation — no new runtime or dev dependencies beyond the visualizer.

Optional hygiene-only version bumps (not required for any v1.3 closure): `@sentry/react` 10.49.0 → 10.53.1, `@sentry/vite-plugin` 5.2.0 → 5.3.0, `@playwright/test` 1.59.1 → 1.60.0, `supabase` CLI 2.85.0 → 2.98.2 (the CLI bump is the most useful — `supabase db lint --linked` at 2.98.2 accurately reflects current advisor WARN-level rules for `search_path` enforcement).

**New tooling for v1.3:**
- `rollup-plugin-visualizer@7.0.1` — bundle treemap/flamegraph audit; Rolldown-compatible; env-gated (`ANALYZE=true`) to avoid slowing every build; must be placed last in `plugins[]` but Sentry plugin also requires last position, so use env-gate to run them mutually exclusive (disable Sentry plugin when `ANALYZE=true`)
- `supabase db lint --linked` (CLI 2.98.2) — verify SECURITY DEFINER advisor warnings clear after migration 14 deploys

**Anti-features (explicitly rejected tooling):**
- `@fontsource/inter` or any web font loader — Inter is already a system-font stack (`font-family: Inter, ui-sans-serif, system-ui, sans-serif`); no `@font-face` declaration exists in `index.css`; no Google Fonts CDN link exists in `index.html`; adding a web font download would hurt LCP
- `vite-imagetools` / `sharp` — native C++ binary (~50 MB install) disproportionate for one 12 KB logo; manual WebP conversion is the correct approach
- `rollup-plugin-critical` / `vite-plugin-critical` — critical CSS extraction for a Vite SPA with empty `<div id="root">` produces no benefit; Tailwind v4 already tree-shakes to only used classes

### Architecture Approach

v1.3 makes no structural changes to the existing five-layer SPA + backend. All six work-streams are modifications to existing integration points. The two dependency clusters are: (A) build-pipeline-sensitive work that must sequence internally (bundle visualizer → visualizer findings inform PostHog lazy-load and manualChunks decisions), and (B) fully independent work that can proceed in parallel (DB hygiene, observability verification, E2E verification, Card migration, planning-doc backfill).

**Precise file map for each work-stream:**
1. **UIDN-02 perf pass** — `vite.config.ts` (visualizer plugin + manualChunks), `src/contexts/ConsentProvider.tsx` + `src/main.tsx` (PostHog lazy-load), `src/components/layout/Navbar.tsx` (WebP image), `src/main.tsx` (defaultPreload)
2. **Observability hygiene** — no new code; smoke-test only against deploy preview
3. **DB hygiene** — `supabase/migrations/00000000000014_security_definer_search_path.sql` (new file); no EF or app code changes
4. **E2E hygiene** — `e2e/tests/*.spec.ts` (verify); `eslint.config.js` (E2E-SCOPE-1 rule)
5. **Card migration** — `src/components/admin/AdminsList.tsx`, `CategoriesList.tsx`, `PromoteAdminDialog.tsx`
6. **Planning-doc backfill** — `.planning/phases/*/` only; no code paths

The plugin ordering constraint in `vite.config.ts` is critical: `tanstackRouter` → `react` → `tailwindcss` → `[visualizer, env-gated]` → `sentryVitePlugin`. Both `rollup-plugin-visualizer` and `sentryVitePlugin` require last position; the resolution is an env flag (`ANALYZE=true`) that swaps visualizer in and disables Sentry upload, never both at once.

**Patterns to follow:**
- Migration 14 uses `CREATE OR REPLACE FUNCTION` (not `ALTER FUNCTION`) for all 7 functions — preserves function OID stability for trigger references and ensures the body is also updated
- `freshPoll` fixture remains test-scoped (not worker-scoped) — prevents UNIQUE constraint violations on parallel Playwright workers
- E2E-SCOPE-1: any locator counting or indexing `suggestion-card` elements must be filtered to `[E2E]`-prefixed titles first

### Critical Pitfalls

1. **`increment_vote_count` 42P01 under `search_path = ''`** — The function body has `INSERT INTO vote_counts` (unqualified). With `search_path = ''`, this fails with `ERROR: relation "vote_counts" does not exist` on every vote submission. `CREATE OR REPLACE FUNCTION` must rewrite the body to `INSERT INTO public.vote_counts`. Test locally with a `submit-vote` round-trip before pushing to production. Emergency recovery path: a new migration with the corrected body; recovery is fast but the window of production outage is real.

2. **`is_current_user_admin()` semantic drift** — This function gates every admin RLS policy across 7+ tables. The migration must be a body-identical rewrite with only the `search_path` value changed from `public` to `''`. Any accidental logic change (simplifying the COALESCE, reverting to single-column check) propagates immediately to all admin-gated RLS policies. Diff the old and new body character-by-character before deploying. Run the TEST-11 12-cell RLS matrix after migration to confirm admin access is intact.

3. **Sentry `dedupeIntegration` collapses render-phase throw evidence** — The current `src/main.tsx` wires both `createRoot.onCaughtError` (via `reactErrorHandler`) and `ErrorBoundary.onError` (explicit `captureException`). For the same error, Dedupe collapses these into one event — which is correct behavior in production, but means the smoke-test verification must use distinct error messages for each scenario tested. Dedupe also applies within a page lifecycle, so re-throwing the same error twice in verification produces only one Sentry event. Use `Sentry.lastEventId()` to confirm capture regardless of whether the transport has flushed.

4. **PostHog lazy-load must preserve consent-gate timing** — `ConsentProvider` currently calls `initPostHog()` at module load (eager). A dynamic `import('posthog-js/react')` inside `ConsentProvider` must not change the consent-gate behavior: PostHog must still initialize to zero-capture state until the user clicks Allow. The refactor needs to confirm that the GDPR opt-IN gate fires before any PostHog page-view events can be sent, not just before initialization of the full module.

5. **Plugin ordering: `sentryVitePlugin` vs `rollup-plugin-visualizer`** — Both require last position in `vite.config.ts` `plugins[]`. The resolution is an env flag: when `ANALYZE=true`, add visualizer and disable Sentry upload; in production builds, Sentry is last. Do not commit visualizer as always-on. Test `npm run build` locally after any plugin array change and verify `dist/` has zero `.map` files (upload-and-delete worked).

- **DB hygiene first (Phase 14):** Highest production risk (vote-submission hot path). Must land independently so a rollback doesn't discard other v1.3 changes. `is_current_user_admin` changes are atomic via a single migration with a clear diff.

### Gaps to Address

These open questions should be resolved at the start of their respective phases, not during requirements:

1. **`rls_auto_enable` function location** — referenced in PROJECT.md as one of the 7 SECURITY DEFINER targets, but not found in on-disk migrations during research. Requires `supabase db dump` or Supabase dashboard Function inspector check before writing migration 14. If the function does not exist, the 7-function count becomes 6 and migration 14 scope shrinks slightly.

2. **ESLint E2E-SCOPE-1 rule presence** — `v1.1-PLAYWRIGHT-FIXTURES.md` prescribed this `no-restricted-syntax` rule for `eslint.config.js`. Whether it was actually added in Phase 8 (v1.1) is unconfirmed. A 10-line check of `eslint.config.js` at Phase 15 start resolves this; if present, skip; if absent, add as part of Phase 15.

3. **PostHog `ConsentProvider` lazy-load implementation detail** — `ConsentProvider` currently calls `initPostHog()` at module load eagerly. The dynamic-import refactor must preserve: (a) the consent gate fires before any PostHog capture events, and (b) the `PostHogProvider` wrapper is still available to the component tree after dynamic import resolves. The refactor pattern (`React.lazy` + `<Suspense>` inside ConsentProvider or a `useEffect`-driven dynamic import) needs a concrete decision at Phase 16 plan-phase — both approaches work but have different timing characteristics.

4. **`index.html` font loading state** — Research confirmed Inter is in the system-font-only stack and no Google Fonts `<link>` appears in committed source. A direct read of `index.html` at Phase 16 start should confirm there is no CDN font link that was added post-research. This is a 30-second check that fully closes the "font work is a no-op" conclusion.

5. **Supabase Image Transformations on the production project** — Research notes that Supabase Storage CDN does not offer automatic format negotiation on the free tier. This means the suggestion-image WebP work-stream (beyond the app-owned wtcs-logo.png) depends on whether the project has image transformations enabled. This affects only suggestion card images (admin-uploaded, Supabase Storage URLs), not the logo. Check the Supabase dashboard Storage settings before deciding whether `?format=webp` query params on Storage URLs are viable.
