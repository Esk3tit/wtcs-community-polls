# Pitfalls Research — v1.2 Admin Visibility Controls

**Domain:** Per-row results-visibility policy retrofit on live voting platform (Supabase + React 19 + shadcn/ui)
**Researched:** 2026-05-11
**Confidence:** HIGH — grounded in codebase state (v1.0 RLS migration, v1.1 audit artifacts, SEED-002 spec, closure evidence)

---

## Critical Pitfalls

### Pitfall 1 [P0]: RLS Leakage Under New Per-Row Visibility Policy (re-validates v1.0 Pitfall 10)

**What goes wrong:**

v1.0 Pitfall 10 established a single RLS rule on `vote_counts`: a user must have cast a vote on that `poll_id` to SELECT its counts. The v1.0 policy is:

```sql
USING (
  EXISTS (
    SELECT 1 FROM public.votes
    WHERE votes.poll_id = vote_counts.poll_id
    AND votes.user_id = auth.uid()
  )
);
```

SEED-002 replaces this with a three-mode enum (`respondents_only` / `public_during` / `public_after_close`). The new policy must branch on `polls.results_visibility` AND `polls.closed_at`. If the new policy is written incorrectly — even in a subtle way, such as wrong operator precedence in the `USING` clause, a missing JOIN to `polls`, or an `OR` where `AND` is required — non-voters can query `vote_counts` directly via the Supabase anon key and see results before they respond (or before close). Because vote_counts is read-only and pre-aggregated, there is no write-path protection to fall back on.

**Why it happens:**

The rewrite must join across two tables (`vote_counts` → `polls` via `poll_id`) and combine two independent conditions with the right boolean logic. The three modes create six distinct (visibility, voted, open/closed) states. Developers test the happy path (voted + public mode) and miss the (not_voted + respondents_only + open) cell or the (not_voted + public_after_close + still_open) cell. Postgres RLS is evaluated at query time, not at row-insertion time, which is correct behavior, but the correctness depends entirely on the `USING` expression covering all six cells without false positives.

**How to avoid:**

Write the policy to cover all six cells explicitly, with each condition expressed and reviewed:

```sql
USING (
  EXISTS (
    SELECT 1 FROM public.polls
    WHERE polls.id = vote_counts.poll_id
    AND (
      -- Mode 1: public_during — visible regardless of vote status while open or closed
      polls.results_visibility = 'public_during'
      OR
      -- Mode 2: public_after_close — only visible once poll is closed
      (polls.results_visibility = 'public_after_close' AND polls.closed_at IS NOT NULL)
      OR
      -- Mode 3: respondents_only (default) — must have voted; open or closed
      (
        polls.results_visibility = 'respondents_only'
        AND EXISTS (
          SELECT 1 FROM public.votes
          WHERE votes.poll_id = vote_counts.poll_id
          AND votes.user_id = auth.uid()
        )
      )
    )
  )
);
```

Write RLS invariant tests that cover every cell in this matrix before landing any UI work:

| visibility | voted | poll status | expected SELECT result |
|---|---|---|---|
| respondents_only | yes | open | ALLOWED |
| respondents_only | no | open | BLOCKED |
| respondents_only | yes | closed | ALLOWED |
| respondents_only | no | closed | BLOCKED |
| public_during | yes | open | ALLOWED |
| public_during | no | open | ALLOWED |
| public_during | yes | closed | ALLOWED |
| public_during | no | closed | ALLOWED |
| public_after_close | yes | open | BLOCKED |
| public_after_close | no | open | BLOCKED |
| public_after_close | yes | closed | ALLOWED |
| public_after_close | no | closed | ALLOWED |

The invariant test must query `vote_counts` with the anon key + an authenticated JWT (without a vote row existing) for each cell. The existing `results-visibility.test.tsx` in Phase 2 (`src/__tests__/`) must be extended to cover the new modes; a new migration-level SQL test should also be run directly (using `supabase db test` or a Deno test against the local Supabase instance) to verify RLS outside the React test harness.

Postgres RLS policies re-evaluate the `USING` expression on every query, not at row-write time. This means there is no stale-policy risk when `results_visibility` is flipped from `respondents_only` to `public_during` mid-flight — the next SELECT automatically uses the new value. This is correct behavior and requires no cache invalidation on the client. However, it also means that any policy bug propagates instantly to all live queries, making the invariant test non-optional.

**Warning signs:**

- A user who has not voted can call `supabase.from('vote_counts').select('*').eq('poll_id', X)` and gets rows back (not an empty array).
- The React test `results-visibility.test.tsx` passes with the old policy in place (because the mock doesn't exercise the new modes) but the new cells are not covered.
- The new EF `reveal-poll-results` updates `polls.results_visibility` but the RLS migration has not yet been deployed — results are still blocked even after reveal.

**Phase to address:**

Phase 11 (SEED-002 schema + RLS phase). The RLS policy and the invariant tests must land in the same migration/PR as the enum column. Do not merge Phase 11 without all 12 matrix cells tested.

### Pitfall 2: Deploy-Window Race on the `results_visibility` Enum Default

**What goes wrong:**

The migration adds `polls.results_visibility TEXT NOT NULL DEFAULT 'respondents_only'`. When the migration lands on the Supabase project, existing `polls` rows (v1.0 and v1.1 data) correctly receive `respondents_only` via the column default. However, there is a deploy-window race: if a `create-poll` Edge Function invocation fires in the gap after the migration applies but before the updated EF that sets `results_visibility` is deployed, the INSERT will succeed (the column has a default) but the EF will not be setting the column explicitly. That row silently takes the default. This is the intended behavior — but the risk is the reverse: if someone manually tests the old EF against the migrated schema and the column default is NOT `respondents_only` (e.g., they set it to `public_during` during dev and forget to reset), rows created in that window get the wrong default in production.

**Why it happens:**

Supabase schema migrations apply independently of Edge Function deploys. The two deploy pipelines (database migration via `supabase db push` and EF deploy via `supabase functions deploy`) are not atomic. The gap between them is typically seconds to minutes.

**How to avoid:**

- Confirm the column default is `respondents_only` in the migration DDL and verify with `SELECT column_default FROM information_schema.columns WHERE table_name = 'polls' AND column_name = 'results_visibility'` after migration.
- Deploy the migration first, verify the column default is in place, then deploy the EF.
- The EF `create-poll` should explicitly set `results_visibility` from the request body (or default to `'respondents_only'` if omitted) — relying purely on the DB default is fragile if the EF body ever passes `results_visibility: undefined` via a JSON key that is explicitly `null`.
- Add a `CHECK (results_visibility IN ('respondents_only', 'public_during', 'public_after_close'))` constraint in the migration (or use a Postgres enum type) to block any invalid value at the DB level.

**Warning signs:**

- Rows in `polls` with a `results_visibility` value that is neither `respondents_only` nor one of the other two valid modes.
- A `create-poll` call that omits `results_visibility` from the JSON body and the EF falls through without setting it.

**Phase to address:**

Phase 11 (SEED-002 schema phase). The migration DDL review must confirm the default and CHECK constraint are present.

### Pitfall 3: Ad-hoc Reveal EF Updates Poll but RLS Appears Stale (It Isn't)

**What goes wrong:**

When the admin clicks "Reveal results now," the `reveal-poll-results` EF fires `UPDATE polls SET results_visibility = 'public_during', results_revealed_at = now()`. The developer assumes they need to "refresh the RLS policy" or "invalidate a cache" to make non-voters see results immediately. They add unnecessary complexity (a `pg_notify` call, a `Cache-Control: no-store` header on vote_counts reads, a forced page reload) based on a misunderstanding.

**Why it happens:**

Postgres RLS is evaluated per query at the time the query runs. The `USING` expression reads `polls.results_visibility` from the live row at the moment `SELECT * FROM vote_counts` executes. There is no policy caching at the RLS layer. The reveal is instant from the database's perspective: the next `vote_counts` query from any non-voter will succeed after the UPDATE commits.

The actual latency that matters is the HTTP polling interval. `useVoteCounts` polls at 8 seconds with a `visibilityState` gate. After the EF commits, the next poll cycle (up to 8 seconds later) will return results to non-voters with `public_during`. This is acceptable and expected.

**How to avoid:**

- Do not add server-sent events, pg_notify, or forced reloads to make the reveal "instant." The 8-second polling lag is by design.
- Confirm that the EF returns the new `results_visibility` value in its response body so the admin UI can optimistically update the card without waiting for the next poll cycle.
- On the user-facing side, if a non-voter's component is not currently polling (e.g., it received a 403 from `vote_counts` and stopped), the component must be designed to retry when the visibility state could have changed. The cleanest pattern: poll `vote_counts` unconditionally at the standard 8-second interval and let RLS determine whether rows are returned; render results if rows are returned, render the pre-vote UI if empty array.

**Warning signs:**

- The EF commits successfully but the admin card does not update; developer adds a `window.location.reload()` to "fix" it.
- The user-facing component caches a "403 = no results" response and stops polling after the first blocked attempt.

**Phase to address:**

Phase 11 (SEED-002 EF) for the server contract; Phase 12 (UI) for the component polling behavior.

### Pitfall 4: One-Way Reveal with No Confirmation — Accidental Irreversible Action

**What goes wrong:**

The admin clicks "Reveal results now" accidentally or misreads which suggestion card they are on. Because the reveal is one-way (SEED-002 specifies no undo), the admin cannot reverse the action. Voters who had not yet responded now see live counts, which introduces the bandwagon effect that the `respondents_only` mode was designed to prevent. Community trust erodes if this happens on a high-stakes suggestion.

**Why it happens:**

Admin UIs often put destructive or irreversible actions in the same visual weight as reversible ones. A simple button with a single click is one misclick from an irreversible state change.

**How to avoid:**

- The "Reveal results now" button MUST use the `variant="destructive"` shadcn Button to signal irreversibility visually (red background, white text in both light and dark mode via the `--destructive` and `--destructive-foreground` tokens in `src/index.css`).
- Gate the action behind a confirmation modal (shadcn `<Dialog>` or `<AlertDialog>`). The modal must:
  - Name the suggestion by title so the admin confirms they are acting on the correct item.
  - Include explicit copy: "This cannot be undone. Once results are revealed, non-voters will see live counts before responding."
  - Use a destructive-styled "Reveal results" confirm button and a neutral "Cancel" button.
  - NOT auto-dismiss on outside click — require explicit Cancel or Confirm.
- The EF response (200) should include `{ results_visibility: 'public_during', results_revealed_at: '...' }` so the admin UI can immediately show a "Results revealed" badge on the card without waiting for a refetch.

**Warning signs:**

- The reveal button is styled with the default shadcn Button variant (primary blue/dark).
- The action fires on a single click with no modal.
- The modal dismiss-on-outside-click is not disabled.

**Phase to address:**

Phase 12 (SEED-002 admin UI). The confirmation modal copy and destructive styling are success criteria for that phase's UAT.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Skip the full 12-cell RLS invariant matrix and test only the 3 "happy path" cells (voted + each mode) | Faster Phase 11 ship | Invisible leak in 9 other cells; only caught by an attacker with network inspector | Never — the matrix is small and the risk is catastrophic |
| Use a `text` column with `CHECK` instead of a Postgres `enum` type for `results_visibility` | Simpler migration, easier to add modes later | No type guarantee; application code must validate the string everywhere | Acceptable if CHECK constraint is added in the same migration (which this project uses) |
| Skip audit log row on visibility flip | Saves one DB write per reveal | No paper trail if admin disputes who revealed what and when | Not acceptable — `audit_log` table already exists from v1.0; the write is one line |
| Single EF for both create-time visibility set and ad-hoc reveal | Fewer files to deploy | Mixed concern in one EF makes it harder to add reveal-specific logic (e.g., rate-limiting reveals) | Acceptable for v1.2 scope if the EF clearly branches on intent via the request body |
| Omit `results_revealed_at` timestamp column | Simpler schema | Cannot audit when a reveal happened, cannot show "Revealed X ago" in admin UI | Not acceptable — SEED-002 explicitly flags this as optional but the audit value is real |
| Ship native `<button>` for the reveal action (like the 4 existing drift sites) | Faster | Adds to the UIDN-03 drift count that v1.2 is supposed to close | Never for new code in v1.2 — use `<Button variant="destructive">` |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| RLS + new enum column | Writing the policy before the column exists in the migration; deploying in wrong order | Add the `results_visibility` column AND the new RLS policy in the same migration file, or in two sequential migrations applied atomically |
| Edge Function + schema | EF references `polls.results_visibility` but migration not yet pushed; EF crashes with column-not-found | Always push migration to Supabase before deploying the EF that depends on the new column |
| React component + RLS | Component calls `supabase.from('vote_counts').select('*')` and treats an empty array as "no results yet" without distinguishing "RLS blocked" from "trigger not yet fired" (new poll, zero votes) | Both states return `[]` — distinguish by checking if the user has voted (from `useVoteSubmit` state or a separate votes query) rather than by interpreting the empty array |
| shadcn `<Button>` + `type` attribute | Replacing a native `<button type="submit">` inside a `<form>` with `<Button>` and forgetting `type="submit"` causes form to not submit (shadcn Button defaults to `type="button"`) | Always pass `type="submit"` explicitly on form-submission Buttons: `<Button type="submit">` |
| shadcn `<Button asChild>` + TanStack Router `<Link>` | Using `<Button variant="link" asChild>` wraps the Link but TanStack Router Link adds its own event handling; nested interactive elements cause a11y violations | Use `<Button variant="link" asChild><Link to="...">text</Link></Button>` — this is the documented shadcn pattern; verify no wrapping `<a>` appears in the DOM |
| `audit_log` table + EF | EF forgets to INSERT an audit row on visibility flip because it's not in the schema migration checklist | Add audit row insert as the last step of `reveal-poll-results` EF before returning 200; use service_role client (same as other admin EFs) |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| `reveal-poll-results` EF omits `requireAdmin` check | Any authenticated user can flip a suggestion to `public_during` | Call `requireAdmin(supabase, req)` from `_shared/admin-auth.ts` as the first line of the EF handler — same pattern as all 14 other admin EFs |
| Trusting `results_visibility` from the client in `create-poll` EF | Client sends `results_visibility: 'public_during'` even if the admin UI only offers `respondents_only` as a default; future RLS bypass if the enum is mishandled | Validate `results_visibility` against the allowlist in the EF body parser before passing to the DB: `if (!['respondents_only', 'public_during', 'public_after_close'].includes(body.results_visibility)) throw 400` |
| Using `results_visibility` in the React component as the sole gate for showing/hiding results | The component can be overridden by a user who patches the JS | RLS is the authoritative gate; the React conditional render is UX only. A non-voter who patches the JS still gets an empty array from `vote_counts` when RLS blocks |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Enum value labels exposed in admin UI ("respondents_only", "public_during") | Non-technical admins (esports organizers) see programmer terminology; increases error rate | Map enum values to admin-readable labels in the UI: `respondents_only` → "Hidden until you respond", `public_during` → "Always visible to everyone", `public_after_close` → "Visible after suggestion closes" |
| Tooltip describes mechanic without mentioning trade-off | Admin chooses `public_during` without understanding bandwagon risk | Tooltip on `public_during` should say: "Community members can see live counts before responding. | Confirmation modal uses generic copy ("Are you sure?") | Admin cannot confirm which suggestion they are acting on | Modal title must include the suggestion title; body copy must name the irreversibility: "Once revealed, this cannot be undone." |

## UIDN-03 Native-Button Refactor Specific Pitfalls

### Pitfall: `type="submit"` dropped during refactor

The 4 drift sites (SearchBar.tsx:22 clear-X, SuggestionForm.tsx:140+163 Back-to-admin link, ImageInput.tsx:108 drop-zone trigger) each use native `<button>` for different reasons. The shadcn `<Button>` defaults to `type="button"`, not `type="submit"`. If any of the native buttons being replaced is inside a `<form>` element and was relying on `type="submit"` behavior (or omitting `type` which defaults to `submit` in HTML), replacing with `<Button>` without `type="submit"` breaks form submission silently.

For these 4 specific drift sites: none appear to be inside `<form>` elements (SearchBar clear-X, ImageInput drop-zone trigger, and SuggestionForm back-link are all non-submit actions). However, verify each site's DOM context before replacing. A `getByRole('button')` query in tests still passes because shadcn `<Button>` renders a `<button>` element — but if `type` is wrong the form submit test would fail. Preserve `type` explicitly on each replacement.

### Pitfall: Existing test selectors that query by text content

Tests using `getByRole('button', { name: '...' })` continue to work after the refactor because shadcn Button renders a real `<button>` element with the same text. However, tests that use `.querySelector('button')` or `container.getElementsByTagName('button')` may return a different count if `<Button asChild>` is used (which renders the child element as the button, not a new one). For the 4 drift sites, `asChild` is only needed for the TanStack Router `<Link>` case (Back-to-admin link). Confirm the test suite passes after each individual site's refactor, not only at the end.

## UIDN-02 Lighthouse Rerun Specific Pitfalls

### Pitfall: Harness fix not applied before rerun declares success

The v1.1 Lighthouse rerun is blocked on two things: (1) the hydration-wait fix in `audit-screenshots.mjs` (Plan 02 defect — the harness captured pre-hydration loading shells for unauth routes), and (2) a v1.2 perf budget improvement. The `audit-mobile.sh` script must be re-run only after both fixes land. Re-running without the harness fix will produce the same 6 F6 DOM-assertion warnings on `/archive` and the same byte-identical unauth PNGs. Re-running without a perf improvement will produce the same P=82-88 scores.

The gate is explicit: `audit-mobile.sh` exits non-zero on any failed Lighthouse route AND on any F6 warning. Both legs must clear. If the perf improvement lands but the harness fix does not, the gate still fails. Plan the harness fix as a prerequisite task inside the UIDN-02 closure phase.

### Pitfall: Lighthouse score variance masking a real regression

Lighthouse mobile scores vary ±5-10pp across runs on the same unchanged codebase. If SEED-002 adds JS to the bundle (new EF invocation code, new UI components, new confirmation modal), the bundle size delta could push `/` from P=82 to P=75-78 — below the "within variance" band. The closure phase for UIDN-02 should note the v1.1 baseline scores and confirm the v1.2 delta is within the ±10pp variance band before declaring PASS. If the delta is larger, investigate bundle size via the `OBSV-02-bundle-delta.md` pattern (Vite build output bytes, per-chunk analysis).

## "Looks Done But Isn't" Checklist

- [ ] **SEED-002 RLS migration:** All 12 matrix cells (4 visibility modes × voted/not_voted × open/closed) have passing invariant tests — not just the 3 happy-path cells.
- [ ] **SEED-002 RLS migration:** The old `vote_counts` policy (v1.0 — voter-exists only) is `DROP POLICY`-ed before the new one is `CREATE POLICY`-ed. Two simultaneous policies on the same table with `OR` semantics between them could produce an overly permissive combined policy.
- [ ] **SEED-002 EF `reveal-poll-results`:** `requireAdmin` is called; audit log row is inserted; `results_revealed_at = now()` is set; the EF returns the updated poll row (not just a 200 OK) so the admin UI can immediately update.
- [ ] **SEED-002 admin UI:** The reveal button uses `variant="destructive"`; the confirmation modal names the suggestion by title; the button is replaced with a badge after a successful reveal.
- [ ] **SEED-002 admin UI:** The visibility selector on the creation form includes human-readable labels (not enum values) and has a tooltip that names the bandwagon trade-off for `public_during`.
- [ ] **SEED-002 user UI:** The component that renders `ResultBars` handles three states: (a) RLS blocked — empty array from vote_counts = do not show results; (b) voted — show results as before; (c) public mode + not yet voted — show results (the component receives results even without a vote row).
- [ ] **UIDN-03 refactor:** All 4 native-button drift sites replaced with `<Button>` using the correct `variant=` and `type=` attributes; test suite passes after each individual replacement.
- [ ] **UIDN-03 refactor:** `<Button asChild>` is only used where the child must be the rendered element (TanStack Router Link case); all other drift sites use plain `<Button>`.
- [ ] **UIDN-02 rerun:** `audit-screenshots.mjs` hydration-wait fix is applied before re-running; the 6 F6 warnings are gone; the 30 unauth PNGs show distinct routes (not byte-identical loading shells).
- [ ] **UIDN-02 rerun:** Lighthouse Performance score for `/` is ≥90, or the delta from the v1.1 baseline is within the ±10pp variance band (documented and accepted with rationale if below 90).

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| RLS leak discovered post-merge (non-voters can read vote_counts) | HIGH | Emergency migration: `DROP POLICY "Vote counts visible to voters" ON public.vote_counts; CREATE POLICY ... (corrected);` — deploy via `supabase db push` against production immediately. No data is corrupted (vote_counts is read-only); the only damage is that vote counts were visible to non-voters during the window. |
| Wrong enum default set in migration (e.g., `public_during` instead of `respondents_only`) | MEDIUM | New migration: `ALTER TABLE polls ALTER COLUMN results_visibility SET DEFAULT 'respondents_only'; UPDATE polls SET results_visibility = 'respondents_only' WHERE results_visibility = 'public_during' AND results_revealed_at IS NULL;` — the UPDATE only corrects rows that were not explicitly revealed (those with `results_revealed_at IS NOT NULL` were intentional). |
| Accidental ad-hoc reveal on wrong suggestion | MEDIUM | No DB rollback available (one-way by design). Mitigation: close the suggestion immediately (using existing `close-poll` EF) to prevent further response changes; communicate to the community via Discord. Log in `audit_log` for accountability. |
| Lighthouse Perf gate still fails after perf budget work | LOW | Per the v1.1 Path-3 precedent: accept Path 3 (defer) with documented rationale and a follow-up token; do not block the v1.2 merge on a sub-90 Perf score if the delta is within variance. Record baseline + new score in the UIDN-02-mobile-evidence.md rerun section. |
| `type="submit"` dropped on a form button during UIDN-03 refactor | LOW | Re-add `type="submit"` to the `<Button>` component; the fix is one attribute; no schema or data change needed. Caught immediately by any form-submission test. |
