## Work-stream 1: UIDN-02 Aggressive Perf-Budget Pass

### 1a. Bundle Splitting

**Context:** `vite.config.ts` already has `tanstackRouter({ autoCodeSplitting: true })`. The current `createRouter(routeTree)` in `main.tsx` has no `defaultPreload`. Route tree has 5 user-facing routes (`/`, `/topics`, `/archive`, `/auth/callback`, `/auth/error`) and a nested admin subtree (`/admin`, `/admin/suggestions/new`, `/admin/suggestions/$id/edit`).

#### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| Confirm `autoCodeSplitting: true` is active and actually splitting admin routes | With `autoCodeSplitting` enabled, TanStack Router splits non-critical route config (component, pendingComponent, errorComponent) into `.lazy.tsx` chunks automatically. The build visualizer (rollup-plugin-visualizer) must confirm admin-route components land in separate chunks, not in the main bundle. | SMALL | `autoCodeSplitting: true` is already set in `vite.config.ts`. Task is audit + verify split is real via bundle visualization, not assumption. |
| Admin route lazy-load verification | `/admin/*` routes are auth-gated (AdminGuard redirects unauthenticated users). All 20-30 concurrent users are non-admin; admin UI code should never load for them. If `autoCodeSplitting` isn't isolating the admin component tree, manually convert to `.lazy.tsx` pattern. | SMALL | Check rollup-plugin-visualizer output. If admin components appear in the initial chunk, use `createLazyFileRoute` for each admin route file. |
| Bundle visualization (rollup-plugin-visualizer) | No perf-budget work should proceed without seeing what's in the chunks. A treemap or sunburst report is the minimum evidence base. | TRIVIAL | `npm i -D rollup-plugin-visualizer` + one-line plugin addition in `vite.config.ts`; generates `dist/stats.html`. Remove from final build output or gate behind `analyze` mode env variable. |
| Vendor chunk identity check | Confirm React, React DOM, and TanStack Router land in a stable vendor chunk (hash stable across app-only changes) so return visitors hit cache. Vite handles this automatically when chunks are stable, but verify the chunk name pattern is predictable. | TRIVIAL | Inspect `dist/assets/` filenames post-build. Stable vendor hash = no work needed. Unstable hash = investigate `manualChunks` or check import order. |

#### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| Granular split of Sentry bundle | `@sentry/react` + `@sentry/browser` contribute ~50–80 KB gzipped. Sentry Replay is already lazy-loaded (per `main.tsx` — `loadSentryReplayIfConsented()`). The core Sentry init stays in main. Worth checking whether `browserTracingIntegration` can be deferred, though tracing is low-value for this app (simple SPA, no backend tracing). | SMALL | Check visualizer for Sentry chunk size. Replay is already deferred — confirm it's not bleeding into the main bundle. Low ROI if Sentry core is <30 KB. |
| PostHog lazy-load behind consent | `initPostHog()` is called unconditionally at module load in `main.tsx`, even for users who haven't consented. If PostHog client is large (it is — ~50–80 KB), deferring initialization until after consent would cut initial parse time for first-visit non-consenting users. | MEDIUM | Requires refactoring `initPostHog()` call into `ConsentContext` effect (after consent state is read). Test that PostHog doesn't miss page-view events. Medium complexity due to timing sensitivity. |

#### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Micro-splitting every component into its own async chunk | Creates waterfall of parallel requests that net-cancels any bundle size win on HTTP/2+. Netlify CDN handles chunked delivery well but excessive chunk count creates overhead from manifest fetching and connection setup. Vite default splitting (one chunk per dynamic import boundary) is the right granularity. | Let `autoCodeSplitting` handle it. If a specific non-route component is >100 KB, defer it with `React.lazy` — otherwise leave it in its route chunk. |
| `manualChunks` exhaustive vendor splitting | Splitting React into its own chunk, shadcn into another, Supabase into another, etc. creates a dependency graph that's fragile to maintain and offers marginal benefit vs route-level splitting alone on an app this size (<300 KB JS total budget). | Route-level splits only. Let Rolldown figure out vendor placement. Override only if visualizer shows a specific 3rd-party library is bloating a route chunk. |
| Suspense wrapper churn (adding Suspense at every lazy import boundary) | TanStack Router's code splitting provides its own `pendingComponent` mechanism. Adding React Suspense wrappers at every lazy import boundary duplicates the loading-state machinery and can produce jarring double-loading flashes. | Use TanStack Router's `defaultPendingComponent` for route-level pending UI. Reserve `React.lazy + Suspense` for non-route component splits like `DebugAuthOverlay` (already done in `__root.tsx`). |

### 1b. Image Optimization

**Context:** App currently uses `img` tags for Discord avatar URLs (external CDN, not under our control) and admin-uploaded suggestion images (Supabase Storage URLs or external URLs). Avatar images are small (64×64px); suggestion images vary.

#### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| `width` + `height` attributes on all `<img>` tags | Prevents CLS (Cumulative Layout Shift). Without explicit dimensions, the browser cannot reserve layout space before the image loads. CLS contributes directly to Lighthouse scores. | SMALL | Audit all `<img>` uses in the codebase. Discord avatars: always 64×64 (set `width={64} height={64}`). Suggestion images: known max-width from container (`max-w-2xl`); use aspect-ratio CSS if dimensions unknown. |
| `loading="lazy"` on below-fold images | Defers off-screen image downloads until scroll. Standard browser optimization, expected by Lighthouse. | TRIVIAL | Add `loading="lazy"` to all `<img>` not in the above-fold hero region. Discord avatars in the admin list are below fold — lazy. Suggestion card images depend on position — lazy by default, `eager` only for LCP candidate. |
| `fetchpriority="high"` on the LCP image | The Largest Contentful Paint element on `/topics` is likely the first pinned suggestion card image (if one exists). Telling the browser to prioritize that fetch reduces LCP. | TRIVIAL | Add `fetchpriority="high"` to the first image in the pinned suggestion list. Only one image per page should have this. |

#### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| WebP/AVIF format migration for suggestion images stored in Supabase Storage | Supabase Storage supports image transformations via the `?width=&height=&format=webp` query parameter transform API (requires storage image transformations, which are available on free tier via the storage CDN). For suggestion images we control (uploaded to Supabase Storage), appending `?format=webp&width=800` gives ~30–50% smaller files vs JPEG/PNG at equivalent quality. | SMALL | Check if Supabase Storage transform API is enabled on this project. If yes, append transform params to Supabase Storage URLs generated in `useUploadImage` hook. External URL images (user-pasted) are not under our control — skip. |
| `srcset` / `sizes` for suggestion images | Serves smaller images to mobile clients, reducing bandwidth and LCP time. Given the `max-w-2xl` container (672px), a 2-stop srcset (400w, 800w) covers most cases. | SMALL | Only worthwhile for suggestion images from Supabase Storage where we can generate multiple-width URLs via transform params. Not applicable to external URLs. |

#### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Installing a Vite image-processing plugin (vite-plugin-imagemin, vite-imagetools) | App images are external URLs (Discord CDN, Supabase Storage, user-pasted external). There are no static images in `src/assets/` that need build-time conversion (only SVG icons and the favicon). Installing a build-time image processing plugin for 0 images is pure complexity overhead. | Use Supabase Storage transform API for hosted images. Accept Discord CDN images as-is. |
| Converting Discord CDN avatar URLs to WebP | Discord CDN supports `?size=N&format=webp` query params but these are external URLs we don't control in the RLS sense. Transforming them would require a proxy. | Accept Discord CDN URLs as-is. The avatars are 64×64px; file size impact is negligible. |
| AVIF as primary format (no WebP fallback) | AVIF has 97%+ browser support as of 2026 but Supabase Storage's transform API supports WebP reliably; AVIF support varies by transform pipeline version. Using AVIF via `<picture>` with WebP fallback is correct but adds markup complexity for marginal gain on a small-community app. | Use WebP via Supabase transform API. If AVIF becomes trivially available in Supabase transform API, add it then. |

### 1c. Font Loading

**Context:** `index.css` sets `font-family: Inter, ui-sans-serif, system-ui, sans-serif` on `body`. Inter is referenced by name but there is no `@font-face` declaration visible in `index.css` — the font is being loaded from Google Fonts (likely via a `<link>` in `index.html`) or is falling through to the system font. The `index.html` content was not read but should be checked.

#### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| Audit current Inter loading mechanism (`index.html` inspection) | Must establish whether Inter is loaded from Google Fonts CDN, a `@font-face` self-hosted declaration, or falling back to the system `ui-sans-serif`. The optimization path differs completely. | TRIVIAL | Read `index.html`. If a Google Fonts `<link>` is present, the font loads cross-origin with DNS lookup + connection overhead. |
| `font-display: swap` on the Inter @font-face | Prevents FOIT (Flash of Invisible Text) on slow connections. The browser shows the fallback font immediately, then swaps when Inter loads. This is expected behavior for body text on performance-focused sites. | TRIVIAL | If Inter is currently loaded via Google Fonts link tag, migrating to self-hosted Fontsource automatically gains `font-display: swap` (Fontsource includes it). If already self-hosted with `@font-face`, verify `font-display: swap` is present. |
| Migrate to Fontsource variable Inter (`@fontsource-variable/inter`) | Self-hosting eliminates the Google Fonts cross-origin DNS lookup and connection time (measurable 100–300ms on mobile). `@fontsource-variable/inter` packages Inter as a WOFF2 variable font with `font-display: swap`. Installation is `npm install @fontsource-variable/inter` + `import '@fontsource-variable/inter/wght.css'` in the app entry. | SMALL | Use the variable font (`@fontsource-variable/inter`) over the static variants (`@fontsource/inter`) — one file covers all weights 100–900 vs separate files per weight. Vite bundles the WOFF2 as a static asset. |

#### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| Latin-only subset (`import '@fontsource-variable/inter/wght-latin.css'` or equivalent) | The WTCS community is English-language; the full variable Inter WOFF2 includes character ranges for Cyrillic, Greek, Vietnamese, etc. Latin subset is typically 40–60% smaller. Fontsource publishes subset variants; use `/wght-latin.css` if available. | TRIVIAL | Check Fontsource's available subsets for `@fontsource-variable/inter`. If `/wght-latin.css` is published, prefer it. Saves ~50–100 KB in WOFF2 size. |
| `<link rel="preload">` for the WOFF2 font file | Tells the browser to fetch the font file during HTML parsing, before the CSS parser encounters the `@font-face` rule. Reduces FOUT window. With Vite, the WOFF2 gets a content-hashed filename; the preload `href` must match. | SMALL | Vite can output a preload hint automatically via `modulePreload`. For font files (not JS modules), a custom plugin or manual `index.html` edit with the hashed URL is needed. Alternatively, the font FOUT on this app (plain body text) is low-visibility — weigh complexity vs benefit. |

#### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| `font-display: optional` | Prevents FOUT entirely by skipping the font if it's not cached — but on first visit Inter never loads, leaving users on the system font forever. For a design system where Inter is part of the visual identity, `optional` defeats the purpose. | Use `font-display: swap`. |
| `font-display: block` | Causes FOIT — invisible text while font downloads. Worse UX than FOUT on slow connections. Standard advice since 2020 is never use `block` for body text. | Use `font-display: swap`. |
| Google Fonts with `preconnect` as a performance fix | Adding `<link rel="preconnect" href="https://fonts.gstatic.com">` reduces but does not eliminate the cross-origin fetch penalty. Self-hosting removes it entirely. Preconnect is a band-aid on top of the wrong architecture. | Self-host with Fontsource. |
| Loading Inter in multiple non-variable weight files | `@fontsource/inter` static variant: separate WOFF2 per weight (400, 500, 600, 700). Four files vs one variable file. HTTP/2 partially mitigates parallel fetch overhead but it's still more bytes and more requests than a single variable font. | Use `@fontsource-variable/inter` (one WOFF2 for all weights). |

### 1d. Critical CSS

**Context:** `index.css` uses Tailwind v4 via `@import "tailwindcss"`. Tailwind v4 uses Lightning CSS as its engine and performs aggressive tree-shaking — only classes referenced in source files are emitted. The output is a single compiled CSS file loaded synchronously via Vite's CSS injection. There are no hand-written CSS rules except the cursor pointer global and the `@theme inline` token block.

#### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| Confirm Tailwind v4 is tree-shaking correctly (no dead class leakage) | Tailwind v4's Vite plugin performs ahead-of-time class detection from source files. Verify the built CSS does not include classes from node_modules or dynamically constructed strings that escaped the scanner. | TRIVIAL | Run `npm run build`, inspect `dist/assets/*.css` size. Compare against a rough expectation (Tailwind v4 for this app: <50 KB uncompressed is healthy). If CSS is >100 KB, investigate with PurgeCSS or a CSS analyzer. |

#### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| CSS inline in `<head>` for above-fold tokens only | For SPAs on Netlify (no SSR), the HTML shell is minimal (`<div id="root">`). The critical CSS concept — inlining above-fold styles — provides almost no benefit here because: (1) the SPA shell renders nothing until JS hydrates; (2) Tailwind's output is already minimal (tree-shaken); (3) the full CSS file is fetched in parallel with JS anyway. The token CSS block in `@layer base` (background, foreground colors) could theoretically be inlined to prevent FOUC, but the FOUC risk is negligible on a Vite SPA where the root div is empty until JS runs. | NOT RECOMMENDED | See anti-feature below. |

#### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Critical CSS extraction for Tailwind v4 SPA | Critical CSS extraction tools (Penthouse, Critical, Critters) work by crawling HTML output and identifying above-fold rules. For a SPA with an empty `<div id="root">`, the crawler sees no rendered content — it would inline zero or trivially few rules. The "critical" CSS for a SPA is the whole CSS file (because every style is applied post-hydration). Extracting critical CSS for Tailwind utility classes is an anti-pattern that adds build complexity with zero measurable benefit in this architecture. | Accept that Tailwind v4 + Vite tree-shaking is the right optimization. Focus effort on font loading and image optimization which have real CWV impact. |
| Splitting Tailwind output into per-route CSS | Dynamic CSS imports per route require CSS-in-JS or CSS modules — neither is in this stack. Tailwind v4's compiled output is a single file; splitting it would require rewriting the build pipeline. ROI is zero for this app size. | Single CSS bundle is correct. |

### 1e. Route Prefetch (TanStack Router)

**Context:** Current `createRouter(routeTree)` in `main.tsx` has no `defaultPreload` set. TanStack Router's `autoCodeSplitting: true` is active. Three preload strategies are available: `intent` (hover/touchstart), `viewport` (Intersection Observer), `render` (immediate on Link mount).

#### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| Enable `defaultPreload: 'intent'` on the router | The most universally recommended TanStack Router preload strategy. When a user hovers a nav link (or touchstarts on mobile), the route's code chunk + loader fire 50ms before navigation. This covers `/topics`, `/archive`, and the admin nav link. Essentially free perceived performance improvement. | TRIVIAL | One-line change to `createRouter({ routeTree, defaultPreload: 'intent' })` in `main.tsx`. |
| Disable preload for `/auth/callback` and `/auth/error` | These routes are never navigated to via `<Link>` — they are redirect targets from Supabase OAuth flow. Preloading them is wasted work. TanStack Router allows per-`<Link>` override via `preload={false}`; since there are no `<Link>` elements pointing to auth routes, this is a no-op (just confirm no accidental links exist). | TRIVIAL | Verify no `<Link to="/auth/...">` components exist in the codebase. |

#### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| `preload="viewport"` on nav links in Navbar | The Navbar is always visible. Using `viewport` preload (Intersection Observer) on `/topics` and `/archive` nav links would trigger preloading the moment Navbar appears — which is immediately on page load. This is slightly more aggressive than `intent` and covers the case where users tap without hovering (mobile). However, since `intent` also covers `touchstart`, the delta is small. Worth doing if Lighthouse TTI is still below threshold after `intent`. | SMALL | Override `preload` prop on Navbar's `<Link>` components to `"viewport"`. Check TanStack Router v1 API — `preload="viewport"` is a documented strategy using Intersection Observer. |
| Explicit `preload={false}` on admin navigation links for non-admin users | The admin route chunk should never be preloaded for non-admin users (even on hover of a hypothetical admin link). Since the admin routes are behind `AdminGuard` and non-admin users never see admin links, this is currently not a risk — but documenting the pattern prevents future regressions if a link is added. | TRIVIAL | Add ESLint comment or code review checklist item: admin `<Link>` components must use `preload={false}` or not exist in the non-admin surface. |

#### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| `defaultPreload: 'render'` (global) | Preloads ALL linked routes immediately on DOM mount, regardless of user intent. On the home page this would trigger preloading `/topics`, `/archive`, and any other linked route simultaneously on every page load. Wastes bandwidth for users who immediately navigate away or are on metered connections. | Use `intent` globally, `viewport` on above-fold nav links. |
| `defaultPreloadDelay: 0` (no debounce) | Fires preload on every hover event with zero delay — triggers for accidental mouse movements. The default 50ms delay exists to debounce hover-jitter. Setting it to 0 turns every cursor movement into a network request. | Keep the default 50ms delay. |
| Preloading suggestion detail routes (`/admin/suggestions/$id/edit`) via viewport | Dynamic route params require the actual param value to preload. You cannot preload a parameterized route without knowing `$id`. Attempting to preload all admin suggestion edit routes would require iterating all suggestion IDs and firing individual preloads — expensive and not TanStack Router's model. | Accept that admin edit routes cold-load. The admin audience tolerates a ~100ms chunk fetch; they are on desktop, not mobile. |

## Work-stream 2: Sentry React 19 ErrorBoundary Capture (#17)

**Context:** Issue #17 was diagnosed in v1.1 research (`v1.1-SENTRY-ERRORBOUNDARY.md`) — render-phase throws are not captured by `Sentry.ErrorBoundary` alone in React 19 because React 19 routes caught errors through its own handler, not through `componentDidCatch`. The fix (wire `Sentry.reactErrorHandler()` into `createRoot`'s `onCaughtError` / `onUncaughtError` / `onRecoverableError`) was researched and documented. **The current `main.tsx` already implements this fix** — `createRoot` has `onUncaughtError`, `onCaughtError`, and `onRecoverableError` all wired to a `taggedHandler` that calls `Sentry.reactErrorHandler`. Additionally, `Sentry.dedupeIntegration()` is pinned explicitly with a comment explaining the contract. This means issue #17 was closed in a prior cycle. The v1.3 task is verification + regression prevention, not the original fix.

### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| Smoke-test render-phase throw capture on a deploy preview (production build) | The fix is in `main.tsx` but was never formally verified on a deploy preview since the v1.1 research doc documents it as an open question. A deliberate render-phase throw (`throw new Error(...)` from a component body, not an event handler) must appear in Sentry with `mechanism.type` from `reactErrorHandler` (not `auto.browser.global_handlers.onerror`). | SMALL | Use the `RenderThrowSmoke` pattern from `v1.1-SENTRY-ERRORBOUNDARY.md`: a dev/preview-only component that throws from render body, mounted behind `?sentry-render-smoke=1`. Run against a Netlify deploy preview, confirm event arrives in Sentry with correct mechanism and `boundary: app-root` tag. |
| Verify `dedupeIntegration` suppresses the `onError`-belt duplicate | With both `createRoot.onCaughtError` (via `reactErrorHandler`) AND `ErrorBoundary.onError` (via explicit `captureException`), React 19 produces two Sentry events for the same error. `dedupeIntegration` is pinned to collapse them. Verify on a deploy-preview run that exactly one event arrives per render throw, not two. | SMALL | Count events in Sentry dashboard for the smoke-test error. If two events appear (different eventIds), `dedupeIntegration` is not working as expected — investigate SDK version. |
| Document the mechanism tags in a closure note | The v1.1 research doc lists two open questions about mechanism type (`react.errorboundary` vs `generic`). Once the smoke test confirms the actual tag, record it in a phase closure note so future debugging doesn't re-derive it. | TRIVIAL | One paragraph in the phase SUMMARY. |

### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| Test `onRecoverableError` path explicitly | React 19's `onRecoverableError` fires for hydration mismatches and automatic error recovery (not relevant for this SPA). Verifying that `reactErrorHandler` on `onRecoverableError` doesn't create noise for non-critical framework warnings would prevent alert fatigue. | SMALL | Trigger a recoverable error scenario in dev (e.g., force a hydration mismatch via `dangerouslySetInnerHTML` in a test component). Confirm Sentry event has `mechanism.handled: true` and is tagged differently than uncaught errors. |

### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Removing `Sentry.ErrorBoundary` in favor of only the `createRoot` hooks | The hooks provide error reporting; `ErrorBoundary` provides fallback UI. Without `ErrorBoundary`, a render-phase throw crashes the entire app with a blank screen. Both are needed: hooks for capture, ErrorBoundary for recovery. | Keep both. The ErrorBoundary `onError` belt + `dedupeIntegration` is correct. |
| Setting `mechanism.handled: false` on `onCaughtError` | `onCaughtError` fires when React catches an error in an ErrorBoundary — the app recovered via fallback UI. Setting `handled: false` would create a false-positive "unhandled" signal in Sentry. The current `taggedHandler('caught')` passes a no-op callback (empty `() => {}`), which correctly signals handled=true. | Keep the current `taggedHandler('caught')` signature (empty cb = handled). |
| Adding Sentry `beforeSend` to filter all `onRecoverableError` events | Blanket filtering removes visibility into legitimate recoverable issues. | Use Sentry's issue-level filtering in the dashboard if noise is excessive. |

## Work-stream 3: Vite/Rolldown Sourcemap Function-Name Preservation (#19)

**Context:** The `vite.config.ts` already contains:

```ts
build: {
  sourcemap: 'hidden',
  rolldownOptions: {
    output: { keepNames: true },
  },
}
```

This is the fix from v1.1 research (`v1.1-VITE-SOURCEMAPS.md`) — `keepNames: true` preserves literal function names in Rolldown's Oxc-minified output (not via a `__name()` helper as described in old Rolldown docs, but by leaving literal `function Name(...)` declarations). The current `vite.config.ts` already has the fix. The v1.3 task is verification + regression prevention.

### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| Run `scripts/verify-sourcemap-names.mjs` verification | The v1.1 research doc provides a script that checks `dist/assets/*.js.map` for named entries and checks bundled JS for literal un-mangled function declarations. This has not been run against the current build (the config was added but verification may not have been formally completed). Run it and record pass/fail in the phase closure note. | TRIVIAL | Script is documented in `v1.1-VITE-SOURCEMAPS.md`. Run `npm run build && node scripts/verify-sourcemap-names.mjs`. Confirm `chunks with literal un-mangled function declarations >= 1`. |
| Confirm Sentry receive un-mangled frames on next production error | The ultimate verification is a live Sentry event showing a function name like `RootLayout` or `handleResponseSubmit` rather than `xR`. If the smoke-test render error from Work-stream 2 is used, check the stack frame function names in Sentry. | SMALL | Use the same smoke-test deploy-preview event from work-stream 2. Confirm frame names are human-readable. |

### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| Check arrow-function-assigned-to-const name preservation | The v1.1 open question: `const handleX = () => {}` may not benefit from `keepNames` if Rolldown doesn't detect the inferred name. If common event handlers (all defined as arrow-const in this codebase) appear mangled in Sentry frames, add `mangle: { keep_fnames: true, keep_classnames: true }` to `rolldownOptions.output.minify`. | SMALL | During smoke-test, trigger an error from a named arrow handler (`const handleSubmit = () => { throw ... }`). Check Sentry frame. If mangled, apply `mangle.keep_fnames`. |
| Version pin comment for `@sentry/vite-plugin` | Current version is v5.2.0. The v1.1 doc notes v6 was not yet released. If v6 has shipped, evaluate whether upgrading is warranted (Sentry changelog). Pin comment in `vite.config.ts` already notes "Sentry plugin must be LAST — tree-shaking landmine." | TRIVIAL | `npm outdated @sentry/vite-plugin`. If major bump available, check release notes before upgrading. |

### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Switching `build.minify` back to `'esbuild'` | `'esbuild'` is deprecated in Vite 8; using it adds a deprecation warning and will be removed in a future Vite version. The Oxc minifier is the strategic path. | Fix function name preservation at the `rolldownOptions.output.keepNames` level (already done). |
| Switching `build.sourcemap` from `'hidden'` to `true` | `sourcemap: true` generates inline sourcemaps or exposes `.map` files publicly (depending on Vite version). `'hidden'` generates separate `.map` files that are then uploaded to Sentry by `@sentry/vite-plugin` and deleted from `dist/`. Exposing sourcemaps publicly leaks source code. | Keep `sourcemap: 'hidden'`. |
| Disabling minification (`build.minify: false`) for Sentry stack readability | Disabling minification is a development-only pattern. Production bundles must be minified for size and performance. Sentry's symbolicator + sourcemaps + `keepNames` is the correct solution for readable production stack frames. | Keep minification enabled. `keepNames: true` solves the problem without sacrificing bundle size discipline. |

## Work-stream 4: DB Hygiene — SECURITY DEFINER + search_path Discipline

**Context:** Supabase's security advisor flags 7 pre-Phase-11 functions for having a mutable `search_path` while marked `SECURITY DEFINER`. The functions are: `update_profile_after_auth`, `handle_new_user`, `validate_vote_choice`, `increment_vote_count`, `is_current_user_admin`, `profile_self_update_allowed`, `rls_auto_enable`.

**Why it matters:** A `SECURITY DEFINER` function executes with the privileges of its definer (typically `postgres`/superuser in Supabase). If `search_path` is not pinned, an attacker who can create objects in a schema that appears before `public` in the search path can shadow the function's referenced tables or functions, hijacking the elevated-privilege execution. This is a known PostgreSQL attack vector documented in the Postgres security manual. On Supabase, authenticated users cannot create schemas by default, but the risk is still flagged by the advisor and should be remediated for clean security posture.

### Table Stakes (must do)

| Practice | Why Expected | Complexity | Notes |
|----------|--------------|------------|-------|
| `ALTER FUNCTION ... SET search_path = ''` on all 7 SECURITY DEFINER functions | The documented Supabase/PostgreSQL fix for the "Function Search Path Mutable" advisor warning. Empty string pins the search path to nothing — all object references inside the function body must be schema-qualified (`public.profiles`, not just `profiles`). | SMALL per function | Migration pattern: `ALTER FUNCTION public.handle_new_user() SET search_path = '';` — repeat for all 7. Then verify each function body uses fully-qualified names (add `public.` prefix where missing). Wrap in a single `supabase/migrations/YYYYMMDD_fix_search_path.sql` migration. |
| Fully-qualify all table/function references inside the affected function bodies | Setting `search_path = ''` causes `SELECT * FROM profiles` to fail with "relation not found" unless rewritten as `SELECT * FROM public.profiles`. Must audit all 7 function bodies and add `public.` prefix where missing. | SMALL per function | This is the primary work beyond the `ALTER FUNCTION` line — the function bodies may reference tables without schema prefix. Use `CREATE OR REPLACE FUNCTION` to update bodies in the same migration. |
| Clear all 7 WARN-level Supabase advisor rows | The advisor runs automatically in the Supabase dashboard. After applying the migration, verify the warnings disappear. | TRIVIAL | Run Supabase advisor from the dashboard or `supabase db lint`. All 7 WARN rows should clear. |

#### Dependency note for `is_current_user_admin` and `rls_auto_enable`

`is_current_user_admin` is called from RLS policies. Setting `search_path = ''` on it may change resolution of `auth.uid()` — which is in the `auth` schema, not `public`. Ensure `auth.uid()` is used (already schema-qualified via `auth.` prefix) or that `SET search_path = 'auth, public, extensions'` is used instead of empty string if the function references multiple schemas. Verify by testing RLS policies after applying the migration.

### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| `SET search_path = 'pg_catalog, public'` instead of `''` for functions calling Postgres built-ins | A strictly empty `search_path` can break functions that call Postgres built-ins (e.g., `gen_random_uuid()` from `pgcrypto`, or `now()` — though `now()` is in `pg_catalog`). Using `SET search_path = 'pg_catalog, public'` maintains security while not requiring every built-in to be schema-qualified. | TRIVIAL | Evaluate per function: does the body call any pgcrypto/uuid functions? If yes, prefer `pg_catalog, public, extensions` (the extension schema on Supabase). If no, `''` is cleanest. |
| Align `11-PATTERNS.md` `vote_counts` skeleton with shipped REVIEW-FIX-H3 form | A doc-only fix but it prevents future implementers from copying the wrong RLS pattern from the patterns doc. | TRIVIAL | Edit `.planning/11-PATTERNS.md` to replace the legacy `admin OR bypass` skeleton with the `service-role-only bypass` form that matches the shipped migration. |

### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Converting SECURITY DEFINER functions to SECURITY INVOKER to avoid the search_path issue | `SECURITY INVOKER` is the default and is safer, but these 7 functions are SECURITY DEFINER for a reason — they need to bypass RLS (e.g., `handle_new_user` writes to `profiles` during Discord OAuth, where the user's auth session doesn't yet exist to satisfy RLS). Changing them to INVOKER would break auth. | Keep SECURITY DEFINER, fix search_path. |
| Applying the fix as a hotfix to individual functions without a migration file | Direct database edits outside the migrations folder break the migration history. The Supabase CLI + migration files are the correct deployment path. | Create `supabase/migrations/YYYYMMDD_fix_search_path.sql` with all 7 `ALTER FUNCTION` + `CREATE OR REPLACE FUNCTION` statements. |
| Adding `search_path = 'public, auth, extensions, pg_catalog, pg_temp'` (too permissive) | Including `pg_temp` in the search path allows session-local temporary objects to shadow production tables — which is itself a hijacking vector. `pg_temp` should never be in a SECURITY DEFINER function's search path. | Use `''` or `'pg_catalog, public'` or `'public, auth, extensions'` (without `pg_temp`). |

#### When to use `toHaveCount` vs `toBeVisible` vs `toHaveText`

| Assertion | When to use | When NOT to use |
|-----------|-------------|-----------------|
| `toHaveCount(N)` | When the exact count is deterministic (e.g., exactly 1 SMOKE-prefixed card after filtering). Always scope to `filter()` first. | On unfiltered lists that include dynamic canonical seed data — count will vary by test run state. |
| `toBeVisible()` | When you need to confirm an element renders (not that it exists in DOM). Use `.first()` only after `.filter()`. | As a count proxy — `toBeVisible` doesn't tell you there's exactly one match. |
| `toHaveText(regex)` | When the text content of a specific element must match. Useful for post-vote assertions on a known card. | On multiple locator matches — will fail if the locator matches more than one element unless `.first()` or similar scoping is applied. |
| `toBeHidden()` | For negative assertions after filtering (e.g., a non-Lineup-Changes card becomes hidden after category filter). | As a sole assertion on a locator that was never visible — cannot distinguish "not-in-DOM" from "failed to load". |

### Differentiators (worth doing)

| Practice | Value Proposition | Complexity | Notes |
|----------|-------------------|------------|-------|
| Fixture isolation via `freshPoll` for all mutable-state specs | The `freshPoll` fixture (test-scoped, service-role insert + cascade DELETE in finally) is the correct pattern for tests that create polls and vote. The `browse-respond.spec.ts` already uses it. Specs that test admin CRUD but create polls via UI (instead of fixture) should use `deletePollById` in `afterEach` for cleanup — this is documented in `poll-fixture.ts`. | SMALL | Audit all 5 specs: which ones create polls via UI and need `deletePollById` teardown? The `admin-create.spec.ts` creates a poll via UI but does not clean it up (leaves an `[E2E]`-prefixed poll in the DB). Add `deletePollById` cleanup or rely on `global-setup.ts`'s sweep. |
| Add `results_hidden` field to `freshPoll` fixture insert | The `freshPoll` fixture was created before `results_hidden` was added (v1.2). The fixture omits `results_hidden` from the insert, so it inherits the column default (`false`). This is correct, but explicitly including `results_hidden: false` in the fixture insert documents the intent and protects against default value changes. | TRIVIAL | Add `results_hidden: false` to the `polls.insert()` call in `poll-fixture.ts`. |

### Anti-Features (don't do)

| Anti-Feature | Why Problematic | What to Do Instead |
|--------------|-----------------|-------------------|
| Worker-scoped fixtures for poll/vote state | Worker-scoped fixtures are created once per worker and shared across tests in that worker. If two tests in the same worker both vote on the same fixture poll, the second vote fails (UNIQUE constraint). Per the fixture comments, test-scoped is intentional. | Keep `freshPoll` as test-scoped (the current implementation is correct). |
| Asserting exact vote counts on shared/canonical seed polls | Canonical seed polls (UUID `b0000…`) accumulate votes across test runs in a shared environment. `toHaveCount` or `toHaveText(/5 total responses/)` against these polls will fail as soon as vote counts change. | Use `freshPoll` fixture for any test that needs a known vote state. Assert only after the test itself has voted. |
| `page.waitForTimeout(N)` between fixture setup and assertion | Arbitrary sleeps are fragile on slow CI. Playwright's auto-retry built into `toBeVisible`, `toHaveCount`, etc. is the correct mechanism. The `freshPoll` fixture inserts data via service-role API before the test begins — no sleep needed to wait for DB write. | Use Playwright assertions with appropriate `timeout` values (already set to 10_000ms in current specs). |
