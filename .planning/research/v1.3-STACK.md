## Work-stream analysis: what is and is not new tooling

### Work-stream 1: UIDN-02 aggressive perf-budget pass

**New devDep needed: `rollup-plugin-visualizer@7.0.1`**

The 728KB `index-BeaKf3YO.js` chunk is the primary Lighthouse bottleneck. From direct dist inspection:

| Chunk | Size (uncompressed) | Status |
|-------|---------------------|--------|
| `index-BeaKf3YO.js` | 728 KB | Main concern — contains React, TanStack Router, PostHog, Sentry core |
| `supabase-Ds4WXhdB.js` | 199 KB | Already split |
| `sentry-replay-Bynx7imk.js` | 139 KB | Already split (consent-lazy loaded) |
| `SuggestionForm-BjE9LSOF.js` | 47 KB | Already split (admin-only route) |
| `admin-BNvjNpIC.js` | 36 KB | Already split (TanStack Router `autoCodeSplitting: true`) |

**PostHog lazy-loading (no new dep — code change only):** `posthog-js` module.full.js is 420 KB unminified. It is eagerly imported in `src/main.tsx` via `import { PostHogProvider } from 'posthog-js/react'`. Since PostHog is GDPR opt-IN (captures nothing until user consents), the JS needs to be initialized at startup to intercept the consent flag — but the full library does NOT need to be in the initial parse. Pattern: use `React.lazy` + dynamic `import('posthog-js/react')` inside `ConsentProvider` (which already holds the consent state). This defers 180-200 KB of parse/compile from the critical path. **This is a code change to `ConsentProvider.tsx` and `main.tsx`, not a new dependency.**

**Vite `build.rolldownOptions` manual chunks (no new dep — config change only):** Rolldown supports `output.manualChunks` under `build.rolldownOptions`. Splitting PostHog and optionally Sentry core into named chunks improves cache granularity — users who return after a PostHog version bump don't re-download the full `index` chunk. Pattern:

```ts
// vite.config.ts addition inside build.rolldownOptions.output:
manualChunks: {
  'vendor-react': ['react', 'react-dom'],
  'vendor-posthog': ['posthog-js'],
},
```

This is a config change; `rolldown-plugin-visualizer` is the verification tool before and after.

**TanStack Router route prefetch (no new dep — config change only):** TanStack Router already has built-in `preload="intent"` on `<Link>` (triggers on hover/touchstart). The Navbar logo link already uses `preload="intent"` (visible in `Navbar.tsx:22`). The global default can be set on `createRouter({ defaultPreload: 'intent' })` in `src/main.tsx`. This defers prefetching all links rather than just the logo link. **No new dependency; one-line change to `createRouter` call.** Confirmed via Context7 `/tanstack/router` guide on preloading.

**`autoCodeSplitting: true` is already active** (confirmed in `vite.config.ts`). This already produces separate chunks for `/admin`, `/admin/suggestions/new`, `/admin/suggestions/$id.edit`, `/archive`, `/topics`, `/auth/callback`, `/auth/error`. Admin routes are already split. The option to add explicit `.lazy.tsx` files exists (TanStack Router's `createLazyFileRoute` pattern) but with `autoCodeSplitting` already enabled, this is redundant — the plugin already performs the equivalent split.

**Image format migration (no new dep — manual conversion + HTML change):** Only one static image is bundled: `src/assets/wtcs-logo.png` (12 KB, 9.9 KB in dist). The public images (`apple-touch-icon.png`, `favicon-32.png`) are not part of the JS bundle. For the logo: manually convert once to `wtcs-logo.webp` (expected ~5-6 KB) + use `<picture>/<source type="image/webp">` in `Navbar.tsx`. No build-time tooling needed for a single image. **Do NOT add `vite-imagetools@10.0.0`** — it requires `sharp` (a native C++ binding, ~50 MB added to `node_modules`) which is disproportionate for one 12 KB file.

**Font subsetting (no work needed):** Inter is declared as a system font stack in `src/index.css`: `font-family: Inter, ui-sans-serif, system-ui, sans-serif`. There is no Google Fonts CDN load, no `@font-face` declaration, no `.woff2` file. The browser uses the system-installed Inter (present on macOS/iOS) or falls back. Adding Inter as a web font would add a font download step and hurt Lighthouse LCP. **Do not add `@fontsource/inter` or any web font loader.**

**Critical CSS (no new dep):** Tailwind CSS v4 via `@tailwindcss/vite` already generates minimal on-demand CSS — only the utility classes used in the source are emitted. The CSS bundle (`index-VHL9purf.css`, 66 KB) is already critical-CSS-like by construction. Inlining it further (e.g., with `vite-plugin-critical`) is not warranted for this project size.

**Vite config changes for the perf pass (summary):**

```ts
// vite.config.ts — additions inside build:
build: {
  sourcemap: 'hidden',
  rolldownOptions: {
    output: {
      keepNames: true,         // existing — issue #19 fix
      manualChunks: {          // NEW — split PostHog out of index chunk
        'vendor-react': ['react', 'react-dom'],
        'vendor-posthog': ['posthog-js'],
      },
    },
  },
},
// Plugins: add visualizer as the LAST plugin (after sentryVitePlugin) during ANALYZE runs:
// ...(process.env.ANALYZE ? [visualizer({ filename: 'stats.html', gzipSize: true, template: 'treemap' }) as PluginOption] : []),
```


### Work-stream 3: DB hygiene (SECURITY DEFINER + `search_path = ''`)

**No new npm tooling needed.** The fix is a SQL migration (migration 14) that adds `SET search_path = ''` to 7 pre-Phase-11 functions. The `supabase` CLI (already a devDep) is the only tool needed.

**Supabase CLI bump (2.85.0 → 2.98.2):** The current pinned version is 13 minor releases behind. The Supabase advisor (`supabase db lint`) has received rule improvements in this range. Bumping to 2.98.2 ensures `supabase db lint --linked` accurately reflects current advisor WARN-level findings (the 7 functions) and reports clean once the migration lands. This is the only practical reason to bump — the migration pattern itself (`CREATE OR REPLACE FUNCTION … SET search_path = ''`) is the same in both versions.

**The 7 target functions and their current search_path state:**

| Function | Migration | Current state |
|----------|-----------|---------------|
| `profile_self_update_allowed` | 002/003/004 | `SECURITY DEFINER`, NO `search_path` clause |
| `update_profile_after_auth` | 002/003 | `SECURITY DEFINER`, NO `search_path` clause |
| `handle_new_user` | 002 | `SECURITY DEFINER`, NO `search_path` clause |
| `validate_vote_choice` | 002 | `SECURITY DEFINER`, NO `search_path` clause |
| `increment_vote_count` | 002 | `SECURITY DEFINER`, NO `search_path` clause |
| `is_current_user_admin` | 005/009 | `search_path = public` — needs change to `= ''` |
| `rls_auto_enable` | (referenced in PROJECT.md) | Could not locate in migrations — confirm exists in Supabase dashboard |

Note: migrations 005 and 009 use `SET search_path = public` (not `= ''`). The Supabase advisor WARN fires specifically for `search_path = public` (without quoting) because it leaves the search path predictably attackable. The correct form is `SET search_path = ''` (empty string, meaning fully qualify all schema references). Migration 14 must `CREATE OR REPLACE FUNCTION` all 7 with `SET search_path = ''` and update all internal object references to fully qualified names (e.g. `public.profiles` instead of `profiles`).

**Pattern for migration 14 (follows existing convention):**

```sql
-- Migration 14: add SET search_path = '' to all 7 pre-Phase-11 SECURITY DEFINER functions
-- Note: SET search_path = '' requires fully-qualified references inside function body.

CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = ''
AS $$
BEGIN
  -- All references must be schema-qualified: public.profiles, auth.users, etc.
  ...
END;
$$;
```

**Edge Function impact:** None. The 17 Edge Functions run on Supabase's Deno runtime and call the affected functions via the `service_role` client. The `search_path` change is transparent to callers — it only affects what names the functions themselves resolve internally (preventing schema-injection attacks if a `public` schema object is shadowed).

## Vite Plugin Slots (full picture post-v1.3)

The `vite.config.ts` plugin array has strict ordering constraints:

```ts
plugins: [
  tanstackRouter({ target: 'react', autoCodeSplitting: true }),  // MUST be first
  react(),                                                         // MUST be after router
  tailwindcss(),
  sentryVitePlugin({ … }),                                         // MUST be last (tree-shaking)
  // During ANALYZE runs only (env-gated, not committed as always-on):
  // visualizer({ gzipSize: true, template: 'treemap' }) as PluginOption,
],
```

`rollup-plugin-visualizer` must be placed LAST per its own docs ("place it last in the `plugins` array so it sees all module sizes after other transforms have run"). The `sentryVitePlugin` also has a LAST requirement (documented in `vite.config.ts` comment: "tree-shaking landmine"). These two cannot both be last; the practical resolution is to use an env flag so only one is active at a time:

```ts
// Only add visualizer when ANALYZE=true — it must be last in that conditional path
...(process.env.ANALYZE === 'true'
  ? [visualizer({ filename: 'stats.html', gzipSize: true, brotliSize: true, template: 'treemap' }) as PluginOption]
  : [sentryVitePlugin({ … })]),
```

Or run Sentry upload disabled (set `disable: true`) when running `ANALYZE=true vite build` — both can be in the array if Sentry plugin is `disable: true`, since a disabled Sentry plugin is a no-op.
