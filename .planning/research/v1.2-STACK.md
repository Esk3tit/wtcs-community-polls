## Key Technical Decisions

### Decision 1: Postgres native enum vs check-constraint vs text+app-validation

**Recommendation: Postgres native enum.**

Three options exist for `results_visibility`:

| Approach | Pros | Cons |
|----------|------|------|
| `CREATE TYPE results_visibility_enum AS ENUM (...)` | DB-enforced, readable strings, 4-byte storage, extensible via `ALTER TYPE … ADD VALUE` without table rewrite, type-safe in queries | `RENAME VALUE` requires Postgres 10+ (available), removing a value requires migration gymnastics |
| `CHECK (results_visibility IN ('respondents_only', 'public_during', 'public_after_close'))` | No type object in schema | No DB-level type; adding values requires `ALTER TABLE … DROP CONSTRAINT … ADD CONSTRAINT` (table scan on Supabase free tier is fine at <500 rows but philosophically weaker) |
| `TEXT` + app-layer validation | Flexible | No DB enforcement; value can drift if the EF layer is bypassed |

The enum wins. The three values are a closed set with clear semantics. `ALTER TYPE … ADD VALUE` (without `IF NOT EXISTS` guard) is safe in a new migration when the value does not already exist. The Supabase JS client surfaces enum values as plain strings, so the React client receives `"respondents_only"` / `"public_during"` / `"public_after_close"` without any special decoding. **Confidence: HIGH** (Context7 /websites/supabase confirms `CREATE TYPE … AS ENUM` as the standard Supabase pattern; official docs at https://supabase.com/docs/guides/database/postgres/enums verified).

**Migration shape (migration 10):**

```sql
-- 1. Create the enum type
CREATE TYPE public.results_visibility AS ENUM (
  'respondents_only',
  'public_during',
  'public_after_close'
);

-- 2. Add column with default (zero behavior change for existing rows)
ALTER TABLE public.polls
  ADD COLUMN results_visibility public.results_visibility
    NOT NULL DEFAULT 'respondents_only';

-- 3. Optional audit timestamp for ad-hoc reveal
ALTER TABLE public.polls
  ADD COLUMN results_revealed_at TIMESTAMPTZ;
```

`DEFAULT 'respondents_only'` is safe: all 10 existing migrations and any in-flight production rows receive the v1.0 RSLT-05 behavior untouched.

### Decision 2: RLS rewrite on `vote_counts` SELECT policy

**Current policy** (migration 05, Section 3):

```sql
CREATE POLICY "Vote counts visible to voters or admin"
  ON public.vote_counts
  FOR SELECT
  TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM public.votes
      WHERE votes.poll_id = vote_counts.poll_id
      AND votes.user_id = auth.uid()
    )
    OR public.is_current_user_admin()
  );
```

**Required behavior matrix (6 cells)**:

| `results_visibility` | User voted? | Poll closed? | Can see `vote_counts`? |
|---|---|---|---|
| `respondents_only` | Yes | Either | YES |
| `respondents_only` | No | Either | NO |
| `public_during` | Either | No (active) | YES |
| `public_during` | Either | Yes (closed) | YES |
| `public_after_close` | Either | No (active) | NO |
| `public_after_close` | Either | Yes (closed) | YES |

The `public_during` cell for non-voters (unvoted, active poll) is the new surface vs v1.0. The `public_after_close` cell for non-voters (unvoted, closed poll) is also new.

**Recommended new policy** (to replace the existing one in migration 10):

```sql
DROP POLICY IF EXISTS "Vote counts visible to voters or admin" ON public.vote_counts;

CREATE POLICY "Vote counts visibility by policy"
  ON public.vote_counts
  FOR SELECT
  TO authenticated
  USING (
    -- Admin bypass: always allowed
    public.is_current_user_admin()

    -- User has voted on this poll: always see their own results
    OR EXISTS (
      SELECT 1 FROM public.votes
      WHERE votes.poll_id = vote_counts.poll_id
        AND votes.user_id = auth.uid()
    )

    -- public_during: visible to all authenticated users while active
    OR EXISTS (
      SELECT 1 FROM public.polls
      WHERE polls.id = vote_counts.poll_id
        AND polls.results_visibility = 'public_during'
        AND (polls.status = 'active' AND polls.closes_at > now())
    )

    -- public_after_close: visible to all authenticated users once closed
    OR EXISTS (
      SELECT 1 FROM public.polls
      WHERE polls.id = vote_counts.poll_id
        AND polls.results_visibility IN ('public_during', 'public_after_close')
        AND (polls.status = 'closed' OR polls.closes_at <= now())
    )
  );
```

Implementation notes:
- The `polls_effective` lazy-close view is used by read paths in the React client. The RLS policy must handle closure detection itself (compare `closes_at <= now()` or `status = 'closed'`) because RLS runs on the base `polls` table, not through the view.
- The admin branch preserves the existing `is_current_user_admin()` call — no change to that SECURITY DEFINER function.
- Both new branches use `EXISTS` subqueries joining to `public.polls` — consistent with the existing voter-check `EXISTS` pattern and avoids a direct `JOIN` in the `USING` clause (which can have subtle semantics in RLS).
- PITFALL: the `public_during` branch must check both `status = 'active'` AND `closes_at > now()` to match `polls_effective` lazy-close semantics — a poll that has expired but whose cron hasn't swept it yet should be treated as closed. Use `polls.closes_at > now()` as the definitive active gate.
- RLS invariant Vitest tests for all 6 matrix cells should be added under `src/__tests__/integrity/` (consistent with existing `rate-limit-edge-function.test.ts` placement).

### Decision 3: One-way ad-hoc reveal — state-machine vs flag-with-timestamp

**Recommendation: flag-with-timestamp (`results_revealed_at`)**, not a separate state-machine.

Three options:

| Approach | Pros | Cons |
|----------|------|------|
| State-machine (new `status` value or separate `reveal_status` enum) | Explicit modeling | Over-engineered for a single one-way transition; conflicts with existing `status` check-constraint |
| `results_visibility` mutation (EF sets column to `public_during`) | Simple, consistent with creation flow | No audit trail of when reveal happened; admin can't see in the DB log that a specific moment triggered it |
| `results_revealed_at TIMESTAMPTZ` audit column + EF writes both `results_visibility = 'public_during'` and `results_revealed_at = now()` | Simple mutation + audit trail for free; RLS policy only needs to read `results_visibility` | Slightly more EF work |

The third option wins. The `reveal-poll-results` EF does two writes: sets `results_visibility = 'public_during'` and `results_revealed_at = now()`. RLS reads only `results_visibility` (no `results_revealed_at` in the policy USING clause) — simpler policy, audit column is purely a record. One-way is enforced at the EF layer: the EF checks `results_visibility != 'public_during'` before writing and returns 409 if already revealed.

The `results_revealed_at` column was already proposed in SEED-002 as "Optional"; given the one-way requirement, it should be non-optional — it's the only audit record of when the reveal happened.

### Decision 4: `reveal-poll-results` Edge Function shape

Follows the standard 16-EF pattern exactly:

```typescript
// supabase/functions/reveal-poll-results/index.ts
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2.101.1'
import { getCorsHeaders } from '../_shared/cors.ts'
import { requireAdmin, adminCheckResponse } from '../_shared/admin-auth.ts'

// POST body: { poll_id: string }
// Guards: requireAdmin → poll existence check → idempotency check (already public_during?) → write
// Returns: { success: true, results_visibility: 'public_during', results_revealed_at: <ISO> }
```

The EF writes with the **service-role client** (bypasses RLS), same as all 16 existing EFs. No new `esm.sh` imports needed.

One design decision: whether to fold this into `update-poll` or make it a standalone EF. **Standalone is correct** because:
1. `update-poll` guards against edits when votes exist — reveal is only meaningful when votes exist. The guard semantics are inverted.
2. A distinct function name makes the audit log clear.
3. It follows the single-responsibility convention the 16 EFs already use.

### Decision 5: `create_poll_with_choices` RPC signature extension

The existing `create_poll_with_choices` RPC (migration 05) does not include `results_visibility`. Two options:

| Approach | Recommendation |
|----------|---------------|
| Add `p_results_visibility results_visibility` parameter to the RPC (migration 10) | YES — keeps the atomic poll+choices creation in one SECURITY DEFINER transaction |
| Have the `create-poll` EF do a separate UPDATE after the RPC | NO — two-phase writes are non-atomic; the creation EF already uses the RPC for atomicity |

Migration 10 should use `CREATE OR REPLACE FUNCTION public.create_poll_with_choices(...)` with the new parameter and a default of `'respondents_only'`. The EF body must be updated to accept and forward `results_visibility` from the request body.

### Decision 7: UIDN-02 — Playwright Pass-A authenticated fixture for UIDN-03

UIDN-02 deferred a Pass-A screenshot capture of authenticated `/topics` and `/archive` because Phase 8's `loginAs` uses local Supabase fixture sessions (not production). The question raised in the milestone context is whether Phase 8's `freshPoll` fixture is sufficient for UIDN-03's authenticated Pass-A, or whether a new fixture is needed.

**Answer: the existing `loginAs` helper in `e2e/helpers/auth.ts` is sufficient. No new fixture needed.**

Rationale:
- `loginAs(page, fixtureUsers.adminUser.id)` injects a session via `signInWithPassword` against local Supabase fixture users. This is exactly what the audit-screenshots harness's Pass-B already uses for `/admin/suggestions/*` routes.
- The UIDN-03 Pass-A gap is not about fixture capability — `loginAs` already works for auth'd screenshots. The gap was that `audit-screenshots.mjs` did not call `loginAs` before navigating to `/topics` and `/archive` in its unauth pass.
- The fix for Pass-A is adding an auth'd screenshot pass in `audit-screenshots.mjs` using `loginAs` for those routes — the same pattern already implemented for Pass-B (`/admin/suggestions/*`).

The hydration-wait fix in `audit-screenshots.mjs` is separate from the auth question. The F6 warnings (6 DOM-assertion failures on `/archive`) were caused by a `waitFor` timing issue — the harness screenshot fired before React hydration completed. The fix is replacing the current `page.goto(url)` + immediate screenshot with `page.goto(url)` + `page.waitForLoadState('networkidle')` (or `waitForSelector('[data-testid="…"]')`).

**No new Playwright API beyond 1.59.1 is needed.** `waitForLoadState('networkidle')` has been in Playwright since v1.0.

### Decision 8: UIDN-02 — Lighthouse perf budget enforcement in CI

The v1.1 Lighthouse run showed Performance misses of 2–8pp on all 5 routes (A11y, BP, SEO all pass). The question: does v1.2 add formal CI enforcement of Lighthouse budgets?

**Recommendation: re-run the existing `audit-mobile.sh` harness locally; do NOT add a Lighthouse CI step for v1.2.**

Rationale:
1. The Lighthouse free-tier Supabase project is a production URL (`polls.wtcsmapban.com`). CI runs against the local preview build (`localhost:4173`), not production — so a CI Lighthouse gate would need a local-server flag set, adding complexity.
2. Performance scores of 82–88 are primarily due to render-blocking Discord OAuth script and SPA hydration overhead — not fixable by v1.2 changes. Adding a hard CI gate at ≥90 would fail on main before any v1.2 code lands.
3. The correct flow is: ship v1.2 → re-run `audit-mobile.sh` against production → check if the SEED-002 UI additions cause a further regression → if regression is clear, diagnose and fix; if scores hold, flip the Key Decision row.
4. Lighthouse score variance (±5-10pp per run, per the v1.1 research) makes hard CI gates on performance scores unreliable without multiple-run averaging and statistical CI frameworks (e.g., `lighthouse-ci` with `assert` + `budgets.json`), which is out of scope for $0 budget.

If a future milestone adds `lighthouse-ci`, the correct approach is `@lhci/cli` (the official Lighthouse CI package) with a `lighthouserc.json` that targets `localhost:4173` after `npm run preview`. But that is v1.3 scope at earliest.

## What NOT to Add

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| `zod` or `yup` for EF body validation | The 16 existing EFs all validate bodies with manual TypeScript guards (see `create-poll/index.ts`). Adding a validation library breaks the pattern and adds a Deno import. | Manual `typeof body.field === 'string'` guards, same as existing EFs |
| `@lhci/cli` (Lighthouse CI) in v1.2 | Score variance + local-vs-prod gap makes hard CI gates unreliable without multi-run averaging. Adds CI complexity for questionable signal. | Manual rerun of `audit-mobile.sh` against production after each milestone |
| `React Query` / `SWR` for results polling | The existing `useSuggestions` / vote-count hooks use hand-rolled `useState` + `useEffect` with 8s HTTP polling. Introducing a caching library for one new query type (conditional `vote_counts` fetch) is stack drift. | Extend the existing hook pattern — add a `results_visibility` field to the poll query result and conditionally fetch `vote_counts` based on it |
| Toggle-back reveal (admin can un-reveal) | One-way reveal is the correct UX: voters who already saw results can't un-see them; a toggle-back creates a trust gap. SEED-002 explicitly recommends against this. | One-way `reveal-poll-results` EF that returns 409 if already revealed |
| New shadcn components beyond what's already vendored | UIDN-03 fixes are pure substitutions of existing `<Button>` component. No new shadcn primitives are needed. | Use `<Button variant="ghost" size="icon">`, `<Button variant="link">`, `<Button asChild>` — all already in `src/components/ui/button.tsx` |

### What does NOT change (v1.2)

- The `votes` SELECT policy — no change (voters always see own vote, admins bypass)
- The `polls` SELECT policy — no change (all authenticated users read polls; `results_visibility` column is on the polls row, not a secret)
- `submit-vote` EF — no change (one vote per user per poll; visibility mode has no bearing on vote submission)
- All 15 other EFs — no change
- Upstash rate limiting — no change
- Auth flow, ConsentContext, Sentry, PostHog — no change
- `polls_effective` view — the new `results_visibility` column will automatically appear in the view's `SELECT *` when migration 10 runs (Postgres views don't enumerate columns in their stored definition when using `SELECT *` in a non-materialized view — verify with explicit column list if the view uses explicit columns).
