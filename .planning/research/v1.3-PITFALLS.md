## Critical Pitfalls

### Pitfall 1: `search_path = ''` Applied to Functions That Use Unqualified Table Names

**What goes wrong:**

The Supabase security advisor flags 7 pre-Phase-11 `SECURITY DEFINER` functions as `0011_function_search_path_mutable` WARNs. The fix is `SET search_path = ''` (empty, per Supabase's recommended form). However, the 7 functions in `00000000000002_triggers.sql` and `00000000000003_guild_membership.sql` use a mix of qualified (`public.profiles`, `public.choices`, `public.votes`, `public.vote_counts`, `public.admin_discord_ids`) and unqualified references.

Inspecting the migrations:
- `validate_vote_choice` references `public.choices` and `public.votes` â€” fully qualified, safe.
- `increment_vote_count` references `public.vote_counts` â€” fully qualified on the update SET clause (`public.vote_counts.count`), but the INSERT target is bare `vote_counts`. With `search_path = ''`, the bare `vote_counts` in `INSERT INTO vote_counts` resolves to nothing and the function throws `ERROR: relation "vote_counts" does not exist`.
- `handle_new_user` references `public.admin_discord_ids`, `public.profiles` â€” mixed. The `INSERT INTO public.profiles` is qualified; the `SELECT 1 FROM public.admin_discord_ids` is qualified. BUT: `auth.uid()` is not a `public.` reference â€” it lives in the `auth` schema. With `search_path = ''`, `auth.uid()` continues to work because it is schema-qualified. However, `auth.users` in the trigger declaration (via `AFTER INSERT ON auth.users`) is a DDL-level reference, not a search_path-dependent one.
- `profile_self_update_allowed` and `update_profile_after_auth` in the `00000000000003` rewrite use `public.profiles` (qualified) and `auth.uid()` (schema-qualified). These are safe with `search_path = ''`.
- `rls_auto_enable` (if it exists via a policy migration) â€” must be checked individually.

The critical risk: **`increment_vote_count` is on the hot path** â€” it fires on every vote INSERT. If `search_path = ''` breaks this trigger, every vote submission fails with a 500 from `submit-vote`. The trigger is called server-side from the Postgres trigger stack, so it bypasses Edge Function error handling; the failure surfaces as a Postgres error on the `votes` INSERT.

**Why it happens:**

The Supabase advisor recommendation is `SET search_path = ''` (empty string, not `public`). The Supabase docs say: "If you use an empty search path, you must explicitly state the schema for every relation." Developers apply the ALTER FUNCTION mechanically without auditing every table reference inside the function body.

The specific tricky case: the INSERT target in `increment_vote_count` is written as:
```sql
INSERT INTO vote_counts (poll_id, choice_id, count)
```
not `INSERT INTO public.vote_counts`. With `search_path = public`, this resolves correctly. With `search_path = ''`, it does not.

**How to avoid:**

Before issuing `ALTER FUNCTION public.<fn> SET search_path = ''`:
1. Audit every unqualified table/function reference in the body. For each one, add the `public.` prefix.
2. For trigger functions, the `AFTER INSERT ON public.votes` in the `CREATE TRIGGER` DDL is fine â€” that is resolved at DDL time, not at function execution time. Only references inside the `$$` body matter.
3. The correct form is `SET search_path = ''` (empty) paired with fully qualified body references, matching the form already used in `is_current_user_admin` (Migration 9) and `validate_vote_choice` (Migration 8).
4. The safest migration pattern: write a new migration that does `CREATE OR REPLACE FUNCTION ... SET search_path = ''` with the corrected body, rather than `ALTER FUNCTION ... SET search_path = ''` (which does not update the body).
5. Test each function individually in a local Supabase stack before pushing to production.

**Warning signs:**

- `submit-vote` Edge Function returns 500 or Postgres error code `42P01` (undefined_table) on vote submission after the migration runs.
- Supabase dashboard Function Logs show `ERROR: relation "vote_counts" does not exist`.
- Integration test `npm run test:integration` (when local ES256 bug is resolved) fails on the vote path.

**Phase to address:**

DB hygiene phase (first phase of v1.3, before any perf or observability work). The migration must be tested locally against a fresh `supabase start` before deploying. Deploy only after verifying `submit-vote` still round-trips successfully.

### Pitfall 2: `is_current_user_admin()` Semantic Drift From `search_path` Change

**What goes wrong:**

`is_current_user_admin()` is the critical-path gate for all admin RLS policies. It was rewritten in Migration 9 (`00000000000009_admin_integrity_rls.sql`) with `SET search_path = public` (not empty). If v1.3 changes it to `SET search_path = ''` (empty), the body `SELECT ... FROM public.profiles WHERE id = auth.uid()` continues to work because `public.profiles` is already qualified. However, the semantic contract â€” `is_admin AND mfa_verified AND guild_member` â€” must not change in the process.

The risk: a developer reads "migrate to `search_path = ''`" and also takes the opportunity to "clean up" the function body (e.g., simplifying the COALESCE, changing column aliases, or reverting to the old one-column check `is_admin` only). Any logic change here immediately affects every admin-gated RLS policy on `polls`, `choices`, `votes`, `vote_counts`, `categories`, `admin_discord_ids`, and `audit_log`.

The distinction between `SET search_path = public` (current, sufficient) and `SET search_path = ''` (Supabase advisor target) matters only if there are unqualified references in the body. The current body has none â€” it references `public.profiles` explicitly. The Supabase advisor flags it because `public` is in the search path but not hardened to `''`. Changing to `''` is functionally equivalent here (body already fully qualified), but the roadmapper should note this as a no-logic-change migration.

**Why it happens:**

Migrations that look like "just a search_path fix" tempt developers to bundle other improvements. The issue is that `is_current_user_admin()` is called by every RLS USING clause on admin-sensitive tables â€” a behavior change propagates to all of them simultaneously.

**How to avoid:**

- The migration for `is_current_user_admin()` must be a body-identical rewrite with only the `search_path` value changed from `public` to `''` (or verified as already `''` â€” check Migration 9 line 16: it says `SET search_path = public`, not `''`).
- After deploying, verify the Supabase advisor no longer flags it AND run the TEST-11 12-cell RLS matrix from Migration 10 (the vote_counts visibility test covers `is_current_user_admin()` indirectly via the admin-JWT sentinel cell).
- Add a comment to the migration: `-- search_path hardened to '' (empty); body unchanged from Migration 9; logic contract preserved`.

**Warning signs:**

- An admin user can no longer access `/admin` routes (RLS blocks even valid admin sessions).
- The Playwright admin-create `@smoke` spec fails at the `/admin` navigation step.
- Supabase Function Logs show auth errors on `is_current_user_admin` calls for known-good admin sessions.

**Phase to address:**

DB hygiene phase. Treat `is_current_user_admin()` as highest-risk within the 7-function set â€” ship it in its own migration file (or as the last function in the batch migration) with an explicit pre/post test step.

### Pitfall 3: `ALTER FUNCTION` vs `CREATE OR REPLACE FUNCTION` for Live Traffic

**What goes wrong:**

There are two ways to change a function's `search_path` property in Postgres:
- `ALTER FUNCTION public.fn_name SET search_path = ''` â€” changes only the `search_path` setting, does not touch the body. Does not require body qualification; suitable only if the body is already fully qualified.
- `CREATE OR REPLACE FUNCTION ... SET search_path = '' AS $$ ... $$` â€” replaces the entire function definition including the body. Required if the body has unqualified references that need qualifying.

For the 7 pre-Phase-11 functions, the correct approach is `CREATE OR REPLACE` (full rewrite) for `increment_vote_count`, `validate_vote_choice`, `handle_new_user`, `profile_self_update_allowed`, and `update_profile_after_auth` because their bodies need audit and possible qualification. `ALTER FUNCTION` alone for `is_current_user_admin` and `profile_self_update_allowed` (Phase 3 rewrite â€” already qualified) is risky because it assumes the body is already clean.

The race condition: `ALTER FUNCTION` takes an `AccessExclusiveLock` on the function's pg_proc row. This is brief (milliseconds), not a data lock, so live traffic sees the new setting immediately on the next call. Because trigger functions and RLS functions are resolved at query-plan time, a concurrent `submit-vote` call that already has a query plan in flight when the ALTER lands may use the old search_path setting for that in-flight call. On Supabase free tier with low concurrency (20-30 users), this window is negligible.

**Why it happens:**

The risk is confused framing: "ALTER FUNCTION is safe because it doesn't touch the body." True â€” but it also doesn't fix an unqualified body. Developers apply ALTER FUNCTION and assume it fixes the problem. The function then fails at runtime, not at migration time.

**How to avoid:**

- Use `CREATE OR REPLACE FUNCTION` for all 7 functions to ensure both the `search_path` and the body are consistent.
- Test each function in the local Supabase stack before production deploy: for trigger functions (`increment_vote_count`, `validate_vote_choice`, `handle_new_user`, `profile_self_update_allowed`), run an integration test that exercises the trigger path (insert a vote, insert a user, update a profile).
- For the `rls_auto_enable` function (if it exists via an earlier migration â€” grep migrations for it before the phase starts), check whether it references pg_catalog functions unqualified: `pg_catalog.pg_class`, `pg_catalog.pg_namespace`. Those are fine with `search_path = ''` because they are schema-qualified.

**Warning signs:**

- Supabase advisor still flags the function after `ALTER FUNCTION ... SET search_path = ''` but the body has an unqualified reference â€” the advisor clears, but the function will fail at runtime.
- `supabase db diff` after the migration shows no body change, only the `search_path` attribute change â€” that's correct for ALTER FUNCTION, but wrong if the body needed qualification.

**Phase to address:**

DB hygiene phase. Pre-flight: grep every function body for unqualified table names before choosing ALTER vs CREATE OR REPLACE.

### Pitfall 4: Sentry Dedupe Collapses `onCaughtError` + `ErrorBoundary.onError` Into One Event

**What goes wrong:**

The current `src/main.tsx` wires both `createRoot` error hooks (`onUncaughtError`, `onCaughtError`, `onRecoverableError` via `taggedHandler`) AND `<Sentry.ErrorBoundary onError={...}>`. For a render-phase throw caught by the ErrorBoundary, React 19 fires **both** `onCaughtError` (at the root) AND the ErrorBoundary's `onError` callback. The code intentionally pins `dedupeIntegration()` explicitly for this reason (comment in `main.tsx`: "we rely on Dedupe to collapse them").

Issue #17's core problem: render-phase throws under SDK v10 + React 19 may not reach the Sentry transport at all, or arrive as duplicates that Dedupe drops the wrong one. The OBSV-01 comment in `main.tsx` says "may emit up to three events per render-phase throw (ErrorBoundary auto-capture + onError belt + onCaughtError â†’ reactErrorHandler)." The Dedupe integration collapses these by stack trace. But Dedupe collapses events **within a single page lifecycle** â€” if a test or verification scenario triggers two different render-phase throws in sequence, the second one may be dropped if Dedupe treats it as a duplicate of the first.

**Why it happens:**

React 19's `onCaughtError` fires for errors that an ErrorBoundary catches. Sentry's `reactErrorHandler` (called from `taggedHandler`) also fires for the same error. Both paths call `Sentry.captureException`. The `dedupeIntegration` compares fingerprints (stack trace hash) and drops subsequent duplicates **per event.** If the stack trace is identical (same component, same throw), the second event is dropped. This is correct behavior for the same error firing twice â€” but it means that a verification test that throws the same error twice to check capture will only see one Sentry event.

Additionally: the `ErrorBoundary.onError` handler in `main.tsx` calls `Sentry.captureException` **explicitly** (not via `reactErrorHandler`). This creates a third event path. Whether Dedupe collapses all three depends on whether the stack frames match exactly.

**How to avoid:**

- For issue #17 investigation: separate verification into two test scenarios â€” (a) event-handler throw, (b) render-phase throw â€” and verify each independently using the `?debug=auth` DEV overlay or a dedicated test boundary.
- Do NOT use the same error message/stack in both tests. Use distinct error messages so Dedupe cannot conflate them.
- The Phase 6 D-08 resolution (pivoted from render-phase throw to event-handler throw for primary D-08 verification) is still the correct primary path. Issue #17's resolution in v1.3 is to confirm whether render-phase throws now ship correctly under the current triple-hook setup, not to re-engineer the hook architecture.
- The `withScope` + `setTag('boundary', 'app-root')` pattern in `main.tsx` ensures the ErrorBoundary event carries context even if Dedupe keeps it; the key verification is whether an event with `boundary: app-root` appears in Sentry for a render-phase throw.
- If Sentry free-tier rate limits apply (50 events/hour), a test suite that repeatedly throws may exhaust the quota before verification succeeds. Test in a dedicated Sentry project or use the `beforeSend` hook to log events locally without transmitting.

**Warning signs:**

- Sentry dashboard shows zero events for a render-phase throw in a DEV session where `VITE_SENTRY_DSN` is set.
- The Sentry SDK's `beforeSend` hook receives the event (meaning it was captured) but Dedupe drops it before transport (confirm by checking `sdk.integrations` in `Sentry.getClient()?.getOptions()`).
- `Sentry.lastEventId()` returns a non-null ID immediately after a render-phase throw â€” this means the event was captured even if the transport hasn't flushed yet.

**Phase to address:**

Observability hygiene phase (issue #17). This phase should be structured as: (1) diagnose current capture behavior with a minimal throw harness, (2) confirm or deny the render-phase drop, (3) if confirmed, adjust the hook wiring to ensure one event survives Dedupe (e.g., remove the `ErrorBoundary.onError` explicit `captureException` call and rely solely on the `onCaughtError` hook path). Do not ship a fix without a reproducible before/after verification.

### Pitfall 5: Rolldown `keepNames` Applies to Named Declarations But Not All Arrow Functions

**What goes wrong:**

`vite.config.ts` already has `rolldownOptions: { output: { keepNames: true } }` in `build`. This was added to address issue #19 (Sentry shows `$M` instead of `fireSentrySmoke`). However, `keepNames` in Rolldown preserves the `name` property via an injected `__name` helper that explicitly sets the `.name` property on renamed functions and classes. The Rolldown docs confirm it handles named functions and classes, but arrow functions assigned to variables get `name` from the LHS variable assignment â€” if that variable is also renamed by the minifier, `keepNames` may not restore it.

Specifically: arrow functions defined as `const fireSentrySmoke = () => {}` â€” after minification, if the variable is renamed to `$M`, Rolldown's `keepNames` injects `__name($M, "fireSentrySmoke")` to restore `.name`. This works for the `.name` property at runtime. But Sentry's stack frame reconstruction uses the function name from the **source map**, not the runtime `.name` property. If the sourcemap correctly maps `$M` back to `fireSentrySmoke` (via `hidden` sourcemaps and the Sentry plugin uploading them), the stack frame shows `fireSentrySmoke` regardless of `keepNames`.

The actual issue #19 problem: `build.sourcemap: 'hidden'` produces sourcemaps that are uploaded to Sentry but not served publicly. If the Sentry plugin upload fails (e.g., `SENTRY_AUTH_TOKEN` not set in the Netlify deploy environment), Sentry cannot reconstruct the stack frame and shows the mangled `$M`. The `keepNames` option is a belt-and-suspenders measure for runtime `.name` introspection, not the primary fix for stack frames in Sentry.

**Why it happens:**

`keepNames` and sourcemap upload solve different parts of the same problem. Developers conflate them: "I set `keepNames: true`, so Sentry should show real names." That is only true for runtime `.name` introspection (e.g., React component display names). For stack frame reconstruction in Sentry error reports, the sourcemap upload is the critical path.

**How to avoid:**

- Verify the sourcemap upload is happening in the Netlify production build: check the Netlify build log for `sentry-vite-plugin: source maps uploaded` lines or the Sentry UI "Releases" â†’ "Artifacts" tab for the current release.
- Confirm `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, and `SENTRY_PROJECT` are all set as Netlify environment variables (these are non-`VITE_` prefixed â€” they are build-time only, never exposed to the browser).
- The `disable: mode !== 'production'` guard in `vite.config.ts` means the plugin only uploads in production mode. For local diagnosis, set `disable: false` temporarily and run `npm run build` with the env vars set.
- `keepNames: true` in the `build.rolldownOptions.output` path (not `optimizeDeps.rolldownOptions`) is correctly placed for the production bundle. Do not confuse it with the `optimizeDeps` equivalent (which only applies to the dep pre-bundling pass).

**Warning signs:**

- Sentry "Releases" â†’ current release â†’ "Artifacts" shows zero `.map` files â€” the upload failed.
- Netlify build log contains `[sentry-vite-plugin] ERROR: failed to upload source maps` or no upload lines at all.
- A production Sentry event shows `$M` in the stack frame but `fireSentrySmoke` in the Sentry source map viewer (meaning the upload happened but the frame-name extraction failed â€” a different problem).
- Sentry shows `[PLUGIN_TIMINGS] sentry-vite-plugin: >50% of build time` â€” this is a cosmetic Rolldown warning about plugin overhead being a large fraction of total time (Rolldown is fast, so the Sentry network upload dominates the percentage). It does NOT indicate a functional failure.

**Phase to address:**

Observability hygiene phase (issue #19). Separate diagnosis: (1) confirm whether sourcemaps are uploading (check Sentry Artifacts for the current release SHA), (2) confirm whether `keepNames: true` is already set (yes â€” `vite.config.ts` already has it), (3) if upload is confirmed working and names still mangle, investigate whether the Sentry plugin version is compatible with Vite 8's Rolldown sourcemap output format.

### Pitfall 6: Sentry Plugin + Image Optimization Plugin Build Order

**What goes wrong:**

If v1.3's perf-budget pass adds a Vite image optimization plugin (e.g., `vite-plugin-image-optimizer` using sharp to convert PNG/JPG to WebP/AVIF), the plugin must be ordered **before** `sentryVitePlugin` in the `plugins` array. The Sentry plugin processes the final build output (picks up sourcemaps from `dist/`) as a `closeBundle` hook. If an image plugin's output modifies the asset manifest after the Sentry plugin runs, the source map upload may reference paths that don't match the final asset names.

More concretely: the Sentry plugin uses `filesToDeleteAfterUpload: './dist/**/*.map'` to clean up sourcemaps post-upload. If an image plugin regenerates asset content-hashes after the Sentry plugin's `closeBundle` fires (unusual, but possible with some plugin orderings), the uploaded sourcemap for a JS file that imported the old image URL may not match the live URL.

The existing `vite.config.ts` comment already documents "Sentry plugin must be LAST â€” tree-shaking landmine." That ordering should be preserved. Any new plugin (image optimizer, critical CSS extractor, etc.) goes **before** Sentry in the array.

**Why it happens:**

Vite plugin hooks run in array order for most lifecycle hooks (`transform`, `load`). But `generateBundle` and `closeBundle` are more nuanced. Sentry's plugin uses `writeBundle` (post-output-write) hooks, which fire in registration order. An image plugin that also uses `writeBundle` to write optimized images could conflict if added after Sentry.

**How to avoid:**

- Add image optimization and critical CSS plugins at array index 0â€“2 (before `tanstackRouter`, `react`, `tailwindcss`, `sentryVitePlugin`).
- Test the full production build (`npm run build`) locally with the new plugins before deploying. Check `dist/` for the expected `.map` files and the expected `.webp`/`.avif` files.
- Run `du -sh dist/assets/*.js.map` to confirm sourcemaps were generated and then deleted (they should not be in `dist/` after the build completes if `filesToDeleteAfterUpload` worked).

**Warning signs:**

- `dist/` contains `.map` files after `npm run build` completes â€” Sentry upload may have failed, leaving them behind.
- Build log shows `[sentry-vite-plugin] Unexpected error during source map processing` after an image plugin was added.
- Image files in `dist/` are unoptimized (still PNG) after the build â€” image plugin may have run out of order or failed silently.

**Phase to address:**

UIDN-02 perf-budget phase. Add plugins one at a time and verify build output after each addition. Do not add image optimization, font subsetting, and critical CSS plugins in a single commit â€” makes it impossible to isolate which plugin caused a regression.

### Pitfall 7: autoCodeSplitting Route Chunks and the `pendingMs` Default

**What goes wrong:**

`vite.config.ts` already enables `tanstackRouter({ autoCodeSplitting: true })`. In v1.3's perf-budget pass, if route-level lazy loading is tuned further (e.g., manually splitting the `/admin` route tree), the TanStack Router `pendingMs` default (500ms) causes a visible pending state on route transitions if the lazy chunk takes longer than half a second to load. On slow mobile connections (which is the UIDN-02 target scenario), a 500ms delay before showing even a skeleton is perceivable as navigation lag.

For the SPA (no SSR, no hydration), this is not a hydration mismatch â€” it is purely a UX timing issue. The "FOUC" analog for SPAs is showing a pending/skeleton state instead of the full route content.

**Why it happens:**

`autoCodeSplitting: true` in the TanStack Router Vite plugin automatically splits route files. The split happens at the route boundary; the pending-component delay fires when the chunk is not yet cached (first navigation). On mobile with Lighthouse simulate-throttling, chunk fetches take longer, pushing past the 500ms threshold.

**How to avoid:**

- Keep the admin route lazy-loaded (good for public bundle size) but prefetch it after login is confirmed â€” the admin user flow always goes through `/auth/callback` before landing on `/admin`, which gives a natural prefetch window.
- Do NOT add `<Link preload="intent">` on public navigation links that point to admin routes â€” this fetches the admin JS for every user who hovers over an admin link, leaking the admin chunk to non-admin users.
- Admin route code only contains UI â€” all auth gating is server-side (Edge Functions + RLS). Leaking the admin chunk to non-admins is a UX concern (they'd see the admin forms momentarily before the AdminGuard redirects them), not a security concern. Still, avoid it for polish.
- Set `pendingMinMs: 0` and `pendingMs: 1000` (raise the threshold before showing a spinner) for the admin route to avoid spinner flash on fast cached loads.

**Warning signs:**

- Lighthouse `Total Blocking Time` increases after code splitting â€” the chunk fetch itself blocks the main thread if the chunk is large (admin EF invocation code + shadcn Dialog/Sheet components combined).
- A visible spinner or empty state appears for 500ms+ on `/topics` initial load on a 4G-throttled Lighthouse run.

**Phase to address:**

UIDN-02 perf-budget phase. Measure chunk sizes with `rollup-plugin-visualizer` before deciding which routes to split. The admin route is the best candidate (only 3-5% of users are admins); public routes should not be over-split.

### Pitfall 8: Lighthouse Single-Run Variance Masking a Real Regression From Perf Changes

**What goes wrong:**

The v1.2 UIDN-02 rerun established D-13 (single-run policy â€” run once, accept the numbers). For the v1.3 aggressive perf pass, the same policy applies per D-12 (follow-up tied to next perf-budget change). The risk: a perf-budget change that genuinely makes things worse (e.g., a large WebP image not properly sized, a font subsetting step that adds a render-blocking `@font-face` with `font-display: swap`) can produce a Lighthouse score that looks like single-run variance (Â±5â€“10pp) but is actually a real regression.

From the v1.2 baseline: `/` = Perf 85, `/topics` = 86, `/archive` = 88, `/auth/error` = 85. If the v1.3 perf changes produce Perf 78, 79, 82, 79 (all 7â€“9pp below baseline), that is within the Â±10pp variance band but unlikely to be pure variance on 4 out of 5 routes simultaneously. Correlated drops across multiple routes signal a real regression, not noise.

**Why it happens:**

Lighthouse simulate-throttling applies the same simulated CPU and network conditions to every route in a run. Noise is independent per-run (CPU jitter from other processes, network simulation jitter). A real regression from a code or asset change is correlated across routes (the same JS chunk or CSS load affects all routes that load it). A developer who sees all 5 routes drop by 7pp might blame single-run variance and accept it â€” masking the regression.

**How to avoid:**

- After the perf changes, compare the v1.3 run against the v1.2 baseline **per route** and flag any route where the delta exceeds 5pp negatively. A 5pp drop on one route is plausible variance; a 5pp drop on 3+ routes simultaneously is a signal.
- For the Lighthouse harness (`audit-mobile.sh`), run it in an otherwise-idle shell (no other `npm run dev`, no active Playwright test suite, no background video calls on the same machine). Resource contention is the primary driver of false variance.
- The zsh `$status` collision bug from v1.2 (Plan 13-02) means the exit code capture pattern must use `audit_status` or `script_exit`, not `$status`, in any shell snippet that reads `audit-mobile.sh`'s exit code.

**Warning signs:**

- All 5 routes show a simultaneous negative delta of 5â€“10pp from the v1.2 baseline after a specific code change â€” investigate the change rather than accepting as variance.
- The Lighthouse run finishes in <30 seconds â€” likely the Chrome headless instance crashed or throttling was not applied. A normal run takes 90â€“180 seconds for 5 routes.

**Phase to address:**

UIDN-02 perf-budget phase (the Lighthouse rerun plan). Record the v1.2 baseline numbers in the phase's evidence file before running the v1.3 audit so the delta is explicit.

### Pitfall 9: Font Subsetting Strips Non-Latin Glyphs From Discord Usernames

**What goes wrong:**

The Inter font (used throughout the UI via Tailwind CSS v4) is already loaded from Google Fonts CDN or a self-hosted `@font-face`. If v1.3 adds font subsetting to reduce the Inter bundle (a common perf win), the subset must include the Unicode ranges for characters that appear in Discord usernames. Discord usernames since the username migration allow any Unicode alphanumeric character â€” this includes Cyrillic, Greek, extended Latin (accented characters), and CJK characters used by community members.

A Latin-only Inter subset (`U+0000-00FF`) displayed in the username chip on the suggestion card will silently fall back to the browser's default serif/sans-serif for non-Latin glyphs. The fallback font has different metrics than Inter, causing line-height shifts and CLS on username renders.

**Why it happens:**

Font subsetting guides typically show the Latin-only subset as the example. Developers copy the `unicode-range: U+0000-00FF` example without checking whether the app renders non-Latin user content.

**How to avoid:**

- Only subset Inter to remove symbol ranges (Powerline glyphs, private use area, math symbols) that the app definitely does not use.
- Preserve `U+0000-024F` (Basic Latin + Latin Extended A + B + IPA Extensions) and `U+0400-04FF` (Cyrillic) at minimum.
- If using Google Fonts' API with `text=` subsetting parameter, test with a Discord username containing Cyrillic or accented characters before deploying.
- Prefer the Google Fonts swap delivery (`display=swap`) over self-hosted woff2 subsetting unless the self-hosted subset has been validated for glyph coverage.
- The perf win from font subsetting on Inter is typically 40â€“80KB. Against a Lighthouse mobile target where Perf is stuck at 85â€“88, font subsetting alone is unlikely to push past 90. Prioritize JS bundle reduction (code splitting, lazy EF modules) over font subsetting for measurable Lighthouse impact.

**Warning signs:**

- A Discord username with Cyrillic or accented characters renders in a noticeably different font in the suggestion card avatar area.

### Pitfall 10: Two-Layer Seed Interaction â€” `global-setup.ts` Cleanup Marker Conflicts With `beforeEach` Seed

**What goes wrong:**

`e2e/global-setup.ts` pre-cleans orphaned rows via `description = 'freshPoll fixture row'`. The `poll-fixture.ts` freshPoll fixture inserts with this exact description. The interaction is safe in the current design (global-setup runs once before the worker pool starts; per-test freshPoll runs per-test). However, if a new spec for issues #11/#12/#13 adds a `beforeEach` hook that inserts test data outside the `freshPoll` fixture (e.g., directly inserting a poll with a different description marker), global-setup's pre-clean will not catch those rows on the next run.

The `filter-search.spec.ts` currently does NOT use the `freshPoll` fixture â€” it relies on the static `e2e/fixtures/seed.sql` data (`[E2E SMOKE]` prefixed rows). The `admin-create.spec.ts` creates a poll through the UI with `[E2E] Admin-create ${Date.now()}` â€” no `freshPoll` fixture. If this poll's teardown (the `deletePollById` call) fails (e.g., Playwright browser closes before teardown), the row leaks. Global-setup doesn't clean it because it does not have `description = 'freshPoll fixture row'`.

For issue #11 (admin-create doesn't populate Choice 1/2): the fix likely involves adding explicit choice-fill steps. If the fix also involves a new fixture-style data setup, the marker strategy must be consistent with global-setup's cleanup query.

**Why it happens:**

Two cleanup layers with different markers targeting the same table create a coverage gap. Global-setup only cleans `description = 'freshPoll fixture row'`. Poll rows created through the UI (admin-create spec) use `[E2E]` title prefixes but no description marker. If the UI-driven teardown fails, these rows accumulate and can cause count-based assertions (`toHaveCount(1)`) in other specs to fail when there are unexpectedly more matching cards.

**How to avoid:**

- For issue #12 (browse-respond asserts vote count on zero-vote fixtures): the fix is to assert "N total response(s)" after the test submits a vote â€” which is what the current spec already does. The actual bug is that `freshPoll` initially has zero votes and the spec was asserting on count without voting first. The fix should be in the spec body, not the fixture.
- For issue #13 (filter-search `toHaveCount()` under two-layer seed): the `filter-search` spec relies on the static seed having exactly 3 active polls with `[E2E` prefix. If the test DB has leaked `[E2E] Admin-create` rows from a failed admin-create teardown, the `toHaveCount(1)` assertion in filter-search fails (finds 2+ rows matching `SMOKE`). Prevention: the `deletePollById` call in admin-create teardown must be hardened to run even if the Playwright page context fails mid-test.
- Add a second global-setup cleanup pass that deletes `[E2E]%`-prefixed polls whose `description` is NOT `'freshPoll fixture row'` (to catch UI-created leaks). Use a `LIKE '[E2E]%'` query with `AND description != 'freshPoll fixture row'` to target only UI-created orphans.

**Warning signs:**

- `filter-search.spec.ts` `toHaveCount(1)` fails with count=2 in CI after a prior run's `admin-create.spec.ts` left an orphaned `[E2E] Admin-create` row.
- `global-setup.ts` runs without error but orphaned rows remain (different description marker).
- Running the Playwright suite twice in a row without a DB reset between runs produces different counts on the second run.

**Phase to address:**

Test/E2E hygiene phase (issues #11, #12, #13). Fix each spec independently and confirm isolation by running the full suite twice in sequence without a DB reset between runs.

### Pitfall 11: Parallel Worker Contamination From Shared `fixtureUsers.memberUser` Session

**What goes wrong:**

`browse-respond.spec.ts` and `filter-search.spec.ts` both log in as `fixtureUsers.memberUser` (`11111111-1111-1111-1111-111111111111`). If Playwright runs these two specs in parallel (different workers, different browser processes), they share the same Supabase user session. The votes table has a `UNIQUE(poll_id, user_id)` constraint. If `browse-respond` submits a vote for a `freshPoll` poll while `filter-search` is also rendering the topics page with `memberUser`, there is no direct conflict (different polls). However, the `votes` table having a memberUser vote row for the freshPoll poll ID is a concern only if `filter-search` also tries to vote on that same poll.

The real parallel-worker risk is the `results-visibility.spec.ts` â€” this spec creates a poll through the UI (as admin) and then votes as member. If two workers run this spec concurrently, they create two different polls (unique timestamps), so no `UNIQUE` collision. But they both read the `/topics` list and may see each other's `[E2E]` polls, causing count assertions to fail.

**Why it happens:**

Playwright's default `fullyParallel: false` (file-level parallel only) limits worker count. But CI environments with `--workers=4` and a full suite run can trigger simultaneous spec file execution. The `e2e/playwright.config.ts` should be checked for `fullyParallel` and `workers` settings.

**How to avoid:**

- Do not assert on absolute card counts on the `/topics` page without filtering to the freshPoll's specific title. The `filter-search.spec.ts` already does this (filters by `E2E_TITLE` regex). New specs for issues #11/#12/#13 must follow the same E2E-SCOPE-1 pattern.
- For `browse-respond`: the `freshPoll` fixture already appends `${Date.now()}-${testInfo.workerIndex}` to the title to disambiguate parallel workers. This prevents the `.filter({ hasText: freshPoll.title })` locator from matching a parallel worker's poll.
- The `memberUser` session shared across specs is only an issue if two specs **vote on the same poll**. With the freshPoll fixture inserting a unique poll per test, this cannot happen.

**Warning signs:**

- CI Playwright run shows flaky failures on `filter-search` `toHaveCount()` when run with `--workers=2` but passes with `--workers=1`.
- `browse-respond` fails with "choice-button not found" when the freshPoll card is filtered by title but two cards with the same title appear (parallel worker workerIndex appending failed).

**Phase to address:**

Test/E2E hygiene phase. Run the fixed specs with `--workers=2` in CI to explicitly validate parallel isolation.

### Pitfall 12: shadcn `Card` Migration â€” Role and Label Changes Break Screen Reader Navigation

**What goes wrong:**

The current `AdminsList`, `CategoriesList`, and `PromoteAdminDialog` hand-rolled containers use bare `<div>` elements with no ARIA role. The shadcn `Card` component also renders a `<div>` (not `<article>` or `<section>`). So the migration is `div â†’ div` with the addition of shadcn's `CardHeader`, `CardContent`, etc. subcomponents.

The risk: if the existing containers have any implicit labeling through nearby `<h2>` or `<h3>` elements that screen readers associate with the containing `<div>`, wrapping with `<Card>` may alter that association. The `<Card>` component adds `rounded-lg border bg-card text-card-foreground shadow-sm` classes; it does not add an ARIA `role`. This is fine for decorative containers. However, if a list of admins (`AdminsList`) uses the container as a semantic grouping landmark (e.g., `role="region"` with `aria-label`), the shadcn Card does not preserve that.

More importantly: existing snapshot tests (Vitest + Testing Library) that test `AdminsList`, `CategoriesList`, or `PromoteAdminDialog` will produce different snapshots because the `className` attribute changes (adds tailwind card tokens). If snapshot tests use `.toMatchSnapshot()`, they will all fail and need updating.

**Why it happens:**

shadcn `Card` is a styled `<div>`. The visual result is identical. But tests that snapshot DOM structure will diff on the new Tailwind classes. Developers forget to update snapshots (`vitest --update-snapshots`) and CI fails.

**How to avoid:**

- Use `getByRole`, `getByText`, and `getByLabelText` queries in tests â€” these are immune to className changes. Avoid `.toMatchSnapshot()` on components that will be styled differently.
- After the migration, run `npm run test -- --update-snapshots` to regenerate any snapshot files. Commit the updated snapshots explicitly in the PR, NOT as a separate "fix tests" commit.
- Verify that no existing `AdminsList` or `CategoriesList` container has `role="list"`, `aria-label`, or `aria-labelledby` on the root `<div>` â€” if it does, add the same attribute to the `<Card>` replacement.
- The Playwright screenshot matrix (audit-screenshots.mjs) will show the Card migration visually. For the v1.3 UIDN-02 rerun, the auth-local Pass-B screenshots for `/admin/suggestions/new` and the edit form will capture the updated card styling. Verify the admin pages look as expected before running Lighthouse.

**Warning signs:**

- `npm run test` fails with "snapshot mismatch" after the Card migration. Expected â€” update snapshots.
- An admin page shows a region that was previously highlighted by a screen reader as a grouping no longer has that landmark. Verify with axe-core or manual screen reader testing (low priority for this user base but good hygiene).

**Phase to address:**

Planning-doc + UI hygiene phase (UIDN-03-FOLLOWUP-LIST-CARDS). Treat snapshot updates as expected cost, not a surprise.
