## Feature Landscape

### Table Stakes (Users and Admins Expect These)

Features that must exist for the feature set to feel complete. Missing any of these makes the v1.2 release feel half-baked.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Three visibility modes at creation (`respondents_only` / `public_during` / `public_after_close`) | Every comparable platform (Slido, Polly, Simple Poll, Discourse) exposes at least three tiers: always visible, visible after voting, visible after close. Admins asked for this because different governance situations have different needs. | MEDIUM | New Postgres enum + migration with `respondents_only` default (zero behavior change for v1.0 data). |
| Default mode = `respondents_only` | Backwards compat with all v1.0 suggestions. No admin should be surprised by behavior change on in-flight topics. | LOW | Migration `ALTER TABLE polls ADD COLUMN results_visibility ...  DEFAULT 'respondents_only'`. |
| User-facing conditional results before voting when policy allows | When `public_during`, non-respondents expect to see current aggregate results. Hiding them when the admin explicitly chose transparency is contradictory. | MEDIUM | Touches Phase 2 SuggestionCard + results reveal logic. RLS on `vote_counts` must admit anon reads when `public_during`. |
| Archive honors the policy | A closed suggestion that was `respondents_only` should remain so after close (RSLT-05 preserved for that mode). A suggestion closed with `public_after_close` should now show results to everyone on the archive page. | MEDIUM | RLS policy on `vote_counts` must evaluate both `results_visibility` AND `closed_at` together. |
| Admin visibility selector in creation form | Polly, Slido, and Simple Poll all expose this at creation time, not after. Admins who forget to set it at creation cannot retroactively correct bandwagon dynamics in a live suggestion. | MEDIUM | New field in the suggestion creation form (`SuggestionForm.tsx`). Requires RadioGroup or equivalent. |
| "Reveal results now" button in admin live card | Polly calls this "Show results." Slido calls it "Reveal results." The concept — ad-hoc override while a suggestion is still live — is a standard feature of every presentation-polling platform. | MEDIUM | New admin Edge Function `reveal-poll-results` + button in admin suggestion card. |
| One-way reveal (no toggle-back) | Slido and Polly both treat reveal as one-way during an active session. Reversing a reveal after voters have already seen public results is a trust violation — "you could see it a moment ago, now you can't." The expected behavior is forward-only. | LOW | Edge Function only sets `results_visibility = 'public_during'` and `results_revealed_at = now()`; no toggle-back path exposed. |

### Differentiators (Competitive Advantage)

Features specific to this platform's context that go beyond the generic polling template.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Tooltip explaining bandwagon trade-off at creation | Non-technical esports admins (Tim's profile) need context to make the right call. Generic polling platforms expose the toggle without explaining the stakes. A sentence like "Showing live results may influence later respondents" in the creation UI is meaningfully different from a bare dropdown. | LOW | shadcn `Tooltip` on the visibility selector or a `<p className="text-xs text-muted-foreground">` description under each option. |
| `results_revealed_at` timestamp (audit column) | Simple Poll and Polly don't surface when an admin triggered a mid-flight reveal. For a competitive scene where integrity perception matters, having a timestamped record of when visibility changed is a differentiator. Does not need to be user-facing; admin-internal only. | LOW | Add `results_revealed_at TIMESTAMPTZ` column to `polls`. Set in the `reveal-poll-results` Edge Function. |
| RLS invariant matrix test (per-mode × pre/post-vote × pre/post-close) | Most platforms trust application-layer checks. This platform's security posture is RLS-first. An invariant test matrix that covers all 6+ cells (3 modes × pre/post-vote × open/closed) is a differentiator in correctness confidence — especially given PITFALL-10 (RLS leakage prevention). | MEDIUM | Vitest or SQL-level tests. 6-cell minimum: `respondents_only × pre-vote`, `respondents_only × post-vote`, `public_during × pre-vote`, `public_during × post-vote`, `public_after_close × open`, `public_after_close × closed`. |
| User-facing state-change awareness when ad-hoc reveal fires | If a user is actively viewing a suggestion when an admin reveals mid-flight, the 8s HTTP polling cycle will naturally surface results on the next tick — no special notification needed. This is unobtrusive and correct for the use case, while fancier platforms might send a push notification. | LOW | No new code required. The existing 8s polling loop picks up the updated `results_visibility` from the RLS-unblocked `vote_counts` read. |

### Anti-Features (Commonly Requested, Often Problematic)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Toggle-back visibility (reveal → hide again) | "What if we made a mistake?" | Voters who already saw revealed results cannot un-see them. Toggle-back creates confusion ("I could see this before, now I can't — was it fake?"). Undermines integrity perception in a competitive community. Slido's presentation mode allows hide/show toggle because a *presenter* controls the moment-to-moment screen; async community voting is different — voters consume on their own schedule. | One-way reveal. If "mistake" scenario is a real concern, closing the suggestion + reopening is the escape hatch. Keep that as a documented support path, not a UI control. |
| Per-choice visibility granularity ("show only option A count") | Technically interesting | Massively complicates the RLS model. A `vote_counts` row is per-choice; partial row disclosure creates ambiguous aggregate signals (seeing "Yes: 40" without seeing "No" count is worse than hiding everything). No polling platform in the research does this at the row level. | Show all choices or none. The three top-level modes cover all real governance patterns. |
| Scheduled reveal ("reveal at 2026-05-15 18:00") | Admin convenience for planned announcements | Adds a job-scheduling complexity that Supabase free tier doesn't support well (no pg_cron without a paid add-on; the existing close-cron is GH Actions-based). For 300-400 respondents, the ad-hoc reveal button is sufficient — admins can coordinate the reveal manually with their Discord announcement. | `reveal-poll-results` button. If scheduled reveal becomes a real request, implement as GH Actions cron writing a config file that the Edge Function reads — defer to v2. |
| Separate results page accessible by shareable link | "Send the results to someone" | The community lives in Discord; direct links to results from non-respondents would bypass the respond-before-see-results design principle entirely, even for `respondents_only` topics. | Public results in `public_during` / `public_after_close` modes are already accessible to anyone who loads the suggestion page. No separate link needed. |
| Admin analytics dashboard for visibility mode effectiveness | "Which mode drives more responses?" | v2 scope per PROJECT.md (ANLT-01/02 deferred). Complexity of cohort analysis is out of proportion for a 300-person community. PostHog is already collecting events. | Use PostHog event explorer when that question arises. No new UI needed in v1.2. |

### Dependency Notes

- **Schema migration required-by RLS rewrite:** The `polls.results_visibility` column must exist before any RLS policy can reference it. Foundation phase must ship schema before UI or test phases.
- **RLS rewrite required-by user UI:** The user-facing "show results pre-vote when public_during" feature is meaningless if the RLS policy still blocks the `vote_counts` SELECT. Shipping the UI without the RLS rewrite produces a blank/error state for `public_during` suggestions.
- **SEED-002 and UIDN-02/03 are independent:** The carry-forward closure work does not depend on SEED-002 and can phase separately. UIDN-02 rerun should run after the main SEED-002 perf budget hit to avoid re-running Lighthouse against an interim build.
- **UIDN-03 fixes required-by Pass-A rerun:** The authenticated Pass-A screenshot for `/topics` and `/archive` (to visually confirm SearchBar's fixed `<Button>`) cannot be captured until `SearchBar.tsx` is patched. The drift fix must land first.

## UI Pattern Recommendations

### Admin Visibility Selector (creation form)

**Recommended: `RadioGroup` with inline description text per option**

Rationale from research:
- For exactly 3 mutually exclusive options that require explanation, radio groups are the universally recommended pattern (beating dropdown for 6 or fewer choices, beating segmented control when options need explanatory text rather than just labels).
- A dropdown compresses the labels and hides descriptions until opened — non-technical admins (Tim's profile) would see only the enum label ("respondents_only") with no context.
- A segmented control (3 pills) has no room for the trade-off copy that makes the choice meaningful. shadcn's `SegmentedControl` equivalent would need a separate help text block anyway.
- shadcn's `RadioGroup` + `RadioGroupItem` + `Label` + per-item `<p className="text-xs text-muted-foreground">` description is the standard pattern for this in forms. It renders vertically, which is fine at max-w-2xl.

**Option labels and descriptions (admin-facing copy — internal terminology allowed):**

| Value | Label | Description |
|-------|-------|-------------|
| `respondents_only` | **After responding** (default) | Community members see results only after they share their opinion. Prevents bandwagoning. |
| `public_during` | **Always visible** | Results are visible to everyone, including before responding. Use for transparency-first topics where live sentiment is useful. |
| `public_after_close` | **After closing** | Results stay hidden while the topic is active, then become visible to everyone once closed. Best for high-stakes decisions where early results shouldn't influence later responders. |

**Implementation note:** Default `value="respondents_only"` in the React controlled form state. No separate "save" needed — it's part of the main form submit.

### "Reveal Results Now" Button + Modal (admin live card)

**Recommended: `Button variant="outline"` → `AlertDialog` with strong irreversibility framing**

This is NOT a destructive action in the delete/data-loss sense, but it IS a one-way commitment. The appropriate pattern is the "consequential but not data-destructive" confirmation:

**Button appearance:**
- `<Button variant="outline" size="sm">` — neutral weight, not destructive-red (this is a positive action for the community, not data loss), but not the primary action on the card either.
- Label: "Reveal results" (not "Show results now" — too casual; not "Make public" — too bureaucratic)
- Icon: `Eye` from lucide-react (`<Eye className="h-4 w-4 mr-2" />`)

**AlertDialog structure:**
- Title: "Reveal results to everyone?"
- Body: "Results will become visible to all community members immediately. This cannot be undone while the topic is active."
- Confirm button: `<Button variant="default">Reveal results</Button>` — standard primary, not red (not destroying data)
- Cancel button: `<Button variant="outline">Cancel</Button>`

**Why AlertDialog (not a simple onClick with window.confirm):**
- shadcn `AlertDialog` is the semantically correct component for "requires explicit user acknowledgement before proceeding." It uses `role="alertdialog"` with correct focus trap and keyboard handling.
- `window.confirm` is synchronous, unstyled, and inaccessible.
- A `Dialog` without the AlertDialog semantics doesn't communicate irreversibility in the accessibility tree.

**Why NOT a countdown (button-with-countdown):**
- This is an async community platform, not a live presentation (unlike Slido's presenter mode). A countdown adds complexity with no benefit — the admin is making a deliberate governance decision, not accidentally hovering over a button.

**Why NOT a text-input friction gate ("type REVEAL to confirm"):**
- Appropriate for permanent data destruction (account deletion, DB wipe). This action reveals information — it doesn't delete anything. Over-engineering the friction for a moderate-severity irreversible action creates fatigue and makes admins route around it.

## Competitor Feature Analysis

| Feature | Slido | Polly (Slack/Teams) | Simple Poll (Slack) | Discourse Polls | Our v1.2 Approach |
|---------|-------|---------------------|---------------------|-----------------|-------------------|
| Per-item visibility mode at creation | Yes — "Immediately" / "On click" toggle per slide | Yes — "Real-time" / "After close" / "Hidden" per polly | Yes — `show-results-after-close` keyword | Yes — `results: always / on_vote / on_close / staff_only` | Yes — RadioGroup 3-option selector in creation form |
| Mid-flight ad-hoc reveal | Yes — eye icon toggle in presentation control bar | Yes — "Show results" button, appears after close | Not in docs | No (results mode is set at creation, no ad-hoc override) | Yes — "Reveal results" button + AlertDialog; one-way |
| Results hidden from non-respondents | Not a native mode (presenter-focused, no respondent gating) | Via "Hidden" mode | Not in docs | `on_vote` mode hides from non-voters | Yes — `respondents_only` (RSLT-05 default) |
| Reveal after close only | Not a primary mode | "After close" mode | `show-results-after-close` keyword | `on_close` mode | Yes — `public_after_close` mode |
| Audit timestamp for reveal action | No | No | No | No | Yes — `results_revealed_at` column (differentiator) |
| Bandwagon trade-off explanation in UI | No | No | No | No | Yes — per-option description text (differentiator for non-technical admins) |

**Naming convention note:** Discourse uses the clearest machine-readable enum values: `always`, `on_vote`, `on_close`, `staff_only`. Polly uses natural language: "Real-time," "After close," "Hidden." Simple Poll uses a hyphenated keyword flag. The SEED-002 proposed names (`respondents_only`, `public_during`, `public_after_close`) are already decided in the spec and accurately describe the semantics from both the admin and voter perspectives — consistent with Discourse's approach of encoding the *trigger* condition in the enum name, not just "public/private."

## Integration Points with Existing Features

### RSLT-05 (v1.0 RLS — respondents_only is the new default)

RSLT-05 hard-coded "respondents only, even after close" as the only mode. In v1.2, `respondents_only` becomes the default value of `polls.results_visibility`, and the RLS policy is rewritten to evaluate the per-row column. The behavioral outcome for any suggestion without an explicit overriding mode is identical to v1.0. No data backfill needed.

**RLS policy change sketch:**

```sql
-- Old (v1.0):
-- vote_counts SELECT allowed only if auth.uid() has a row in votes for this poll_id

-- New (v1.2):
-- USING (
--   p.results_visibility = 'public_during'
--   OR (p.results_visibility = 'public_after_close' AND p.closed_at IS NOT NULL)
--   OR (p.results_visibility = 'respondents_only' AND EXISTS (
--         SELECT 1 FROM votes WHERE poll_id = vote_counts.poll_id AND user_id = auth.uid()
--       ))
-- )
-- where p is a join/subquery to polls on poll_id
```

The join to `polls` inside the USING clause will trigger RLS on `polls` itself — use a `SECURITY DEFINER` helper function to avoid the double-RLS performance hit (per Supabase RLS performance docs recommending SECURITY DEFINER functions for cross-table joins in policies).

### Phase 2 (browsing-responding) — SuggestionCard and results render

The conditional results render (pre-vote for `public_during`) requires a check against the suggestion's `results_visibility` field. The `useSuggestions` hook must include this column in its SELECT. The results display logic currently gates on `hasResponded`; it must also gate on `results_visibility === 'public_during'` as a second unlock path.

### Archive view — `public_after_close` reveal

The archive page currently shows results for all closed suggestions (DESIGN-SYSTEM.md: "Results always visible (suggestion is closed)"). This assumption is no longer correct for `respondents_only` suggestions that are closed. The archive view must query `results_visibility` and only render results for closed suggestions where visibility is `public_after_close` or `public_during`. For `respondents_only` closed suggestions, the results remain gated behind the respondent check (RSLT-05 behavior preserved).
