# Architecture: v1.2 Admin Visibility Controls

## Recommended Architecture

v1.2 extends the existing layered architecture in three independent tracks:

1. **SEED-002 track** (schema-blocked): migration → RLS rewrite → new Edge Function → admin UI → user UI, strict sequential dependency
2. **UIDN-03 track** (parallel): shadcn button cleanup in three files, no schema dependency
3. **UIDN-02 track** (blocked on external condition): Lighthouse mobile rerun must wait for v1.2 performance budget hit + the Plan 02 hydration-wait fix in `audit-screenshots.mjs`

## Data Flow

### Admin sets visibility at creation
```
Admin fills SuggestionForm
  → selects VisibilityPicker value (default: respondents_only)
  → form submits via useCreatePoll
  → create-poll EF validates results_visibility enum membership
  → calls create_poll_with_choices RPC with p_results_visibility
  → polls row inserted with results_visibility column set
  → polls_effective view exposes results_visibility on next read
```

### User sees results based on visibility policy
```
useSuggestions fetches from polls_effective
  → SuggestionWithChoices now includes results_visibility field
  → SuggestionCard receives results_visibility via suggestion prop
  → canSeeResults derived:
      results_visibility = 'respondents_only': canSeeResults = userHasVoted
      results_visibility = 'public_during':    canSeeResults = true (always)
      results_visibility = 'public_after_close': canSeeResults = isClosed
  → if canSeeResults: render ResultBars (even before voting)
    else:             render ChoiceButtons (same as v1.0)
  → useVoteCounts called for all polls where canSeeResults (not just voted polls)
```

### Admin ad-hoc reveal
```
Admin opens SuggestionKebabMenu on an active suggestion
  → clicks "Reveal results now…"
  → RevealResultsDialog opens with irreversibility warning
  → admin confirms
  → useRevealPollResults calls reveal-poll-results EF
  → EF: requireAdmin check → UPDATE polls SET results_visibility='public_during',
    results_revealed_at=now() WHERE id=poll_id
  → EF returns 200 with updated row
  → onChanged() callback triggers AdminSuggestionsTab refetch
```

### RLS enforcement (vote_counts SELECT)
```
Non-voter requests vote_counts for poll P:
  CASE polls.results_visibility:
    'respondents_only': DENIED (no vote row for this user)
    'public_during':    ALLOWED (regardless of vote status)
    'public_after_close': ALLOWED only if polls.closed_at IS NOT NULL
                          or polls.status = 'closed'
                          (must join polls table in policy USING clause)

Voter requests vote_counts for poll P:
  Always ALLOWED (existing voter branch preserved)

Admin requests vote_counts:
  Always ALLOWED (admin bypass branch preserved)
```

## Integration Points: Precise File Map

### Migrations (what to add, what to verify)

**Add: Migration 10** (`00000000000010_results_visibility.sql`)

New file. Contains:
- `CREATE TYPE public.results_visibility_mode AS ENUM ('respondents_only', 'public_during', 'public_after_close');`
- `ALTER TABLE public.polls ADD COLUMN results_visibility public.results_visibility_mode NOT NULL DEFAULT 'respondents_only';`
- `ALTER TABLE public.polls ADD COLUMN results_revealed_at TIMESTAMPTZ;`
- `COMMENT` statements
- `DROP POLICY IF EXISTS "Vote counts visible to voters or admin" ON public.vote_counts;` + `CREATE POLICY` rewrite (see RLS section)
- `CREATE OR REPLACE VIEW public.polls_effective AS ...` rewrite to expose the two new columns (see View section)
- Modified `create_poll_with_choices` function signature (see RPC section)

The enum DDL, the two column additions, the RLS rewrite, the view rewrite, and the RPC update all belong in the same migration so they are atomic at the DB layer. Splitting them across two migrations creates a window where the RLS is updated but the view doesn't expose `results_visibility` yet, which could break in-flight reads.

**Verify: Migration 9** (`00000000000009_admin_integrity_rls.sql`) — no changes needed. It only patches `is_current_user_admin()`. The admin bypass branch in `vote_counts` SELECT policy is patched in migration 5; the v1.2 rewrite supersedes that policy.

**Verify: `polls_effective` view** — currently defined in migration 5. The v1.2 migration must `CREATE OR REPLACE VIEW` to add `results_visibility` and `results_revealed_at` columns. The view's `security_invoker = on` setting must be re-applied after the `CREATE OR REPLACE`.

**polls_effective invariant test impact:** `src/__tests__/admin/polls-effective-invariant.test.ts` scans `src/routes`, `src/hooks`, `src/components` for `from('polls')` direct reads. The new `results_visibility` and `results_revealed_at` columns will appear in the view automatically once added to the base table and the view is updated. No allowlist changes needed in the invariant test — the new columns are additive to the view. The test will continue to pass as long as no new code reads `from('polls')` in the public paths. The `useVoteCounts` modification (new non-voter public reads) reads from `vote_counts`, not `polls` — not caught by the invariant. The `reveal-poll-results` EF reads `from('polls')` server-side (service-role, not in `src/`) — not scanned.

### RLS Policies (which to rewrite, which to leave alone)

**Rewrite: `vote_counts` SELECT policy** (currently "Vote counts visible to voters or admin" from migration 5)

Current USING clause:
```sql
EXISTS (SELECT 1 FROM public.votes WHERE votes.poll_id = vote_counts.poll_id AND votes.user_id = auth.uid())
OR public.is_current_user_admin()
```

v1.2 USING clause:
```sql
EXISTS (SELECT 1 FROM public.votes WHERE votes.poll_id = vote_counts.poll_id AND votes.user_id = auth.uid())
OR public.is_current_user_admin()
OR EXISTS (
  SELECT 1 FROM public.polls p
  WHERE p.id = vote_counts.poll_id
  AND (
    p.results_visibility = 'public_during'
    OR (p.results_visibility = 'public_after_close' AND p.closed_at IS NOT NULL)
  )
)
```

Three branches: voter branch (unchanged), admin bypass (unchanged), public-visibility branch (new). The public-visibility branch joins `polls` to read `results_visibility` and `closed_at`. This is a per-row correlated subquery — acceptable at free-tier scale (20–30 concurrent users, pre-aggregated `vote_counts` table is small).

**Leave alone:** All other policies — `polls` SELECT (`authenticated` open read — polls row including `results_visibility` is already public to authenticated users, no change needed), `votes` SELECT (voter-own + admin bypass — unchanged), `categories`/`profiles`/`choices` SELECT (unchanged).

**Leave alone:** `is_current_user_admin()` function — unchanged. Already checks `is_admin AND mfa_verified AND guild_member` per migration 9.

### Edge Functions (new vs modified vs unchanged)

**New: `supabase/functions/reveal-poll-results/index.ts`**

Standalone function, NOT folded into `update-poll`. Rationale: `update-poll` is locked behind the zero-votes edit gate (returns 409 if any vote exists); reveal-results is specifically for live suggestions with votes. Folding would require removing the vote-count guard from `update-poll` or adding a special bypass — either is a regression risk. A separate EF is cleaner and follows the one-handler-per-file convention.

Structure follows `update-poll/index.ts` template exactly:
1. CORS preflight
2. Auth header extraction + `supabaseUser.auth.getUser()`
3. `requireAdmin(supabaseAdmin, user.id)` from `_shared/admin-auth.ts`
4. Parse `{ poll_id }` from JSON body; UUID regex validation
5. Verify poll exists and is active (not already closed) — return 409 if closed
6. `supabaseAdmin.from('polls').update({ results_visibility: 'public_during', results_revealed_at: new Date().toISOString() }).eq('id', poll_id)` — direct UPDATE via service-role; no RPC needed (single-table, non-transactional)
7. Return `{ success: true, results_visibility: 'public_during', results_revealed_at: <timestamp> }`

**Modified: `supabase/functions/create-poll/index.ts`**

Add `results_visibility` to the body type and parsing block. Validate it is one of `['respondents_only', 'public_during', 'public_after_close']`; default to `'respondents_only'` if absent (backwards compat for any old clients). Pass as `p_results_visibility` to `create_poll_with_choices` RPC.

**Modified: `create_poll_with_choices` RPC (in migration 10 via `CREATE OR REPLACE FUNCTION`)**

Add `p_results_visibility public.results_visibility_mode DEFAULT 'respondents_only'` parameter. Update the `INSERT INTO public.polls` statement to include `results_visibility`. The existing `SECURITY DEFINER` + `STABLE`-equivalent and choice-count validation remain unchanged.

**Unchanged:** `update-poll/index.ts` — no visibility field in the edit flow. Edit is locked after first vote anyway. Visibility changes post-creation go through `reveal-poll-results` (ad-hoc) only.

**Unchanged:** All other 14 EFs — `submit-vote`, `close-poll`, `close-expired-polls`, `pin-poll`, `set-resolution`, `delete-poll`, `create-category`, `rename-category`, `delete-category`, `promote-admin`, `demote-admin`, `search-admin-targets`, `get-upload-url`.

## Patterns to Follow

### Pattern 1: Admin EF Skeleton (from update-poll)
Every admin EF follows: CORS preflight → auth header check → `supabaseUser.auth.getUser()` → `requireAdmin(supabaseAdmin, userId)` → body parse + validate → mutation via service-role client → structured JSON response.

`reveal-poll-results` must follow this exactly. The `requireAdmin` import from `../_shared/admin-auth.ts` handles all three integrity checks (is_admin + mfa_verified + guild_member).

### Pattern 2: One-way irreversible action with confirmation (new)
`RevealResultsDialog` uses `<AlertDialog>` (not `<Dialog>`) to signal destructive/irreversible intent. Pattern: AlertDialogTrigger (external open via state) → AlertDialogContent with explicit irreversibility copy → AlertDialogCancel + AlertDialogAction (calls mutation).

### Pattern 3: Derived boolean visibility gate in SuggestionCard
Centralize the three-mode gate in a single `canSeeResults` boolean computed from `suggestion.results_visibility`, `userChoiceId`, and `isClosed`. Do not scatter the three-way check across `ResultBars`, `ChoiceButtons`, or parent components.

---

## Anti-Patterns to Avoid

### Anti-Pattern 1: Folding reveal into update-poll
`update-poll` is locked behind the zero-votes edit gate. Adding a visibility-only mutation path to it creates a conditional bypass of the gate logic. Keep `reveal-poll-results` separate.

### Anti-Pattern 2: Client-side visibility enforcement without RLS backing
The `canSeeResults` branch in `SuggestionCard` is UX convenience. The real gate is the `vote_counts` SELECT policy. Do not treat the React branch as the security boundary — the RLS rewrite is mandatory.

### Anti-Pattern 3: Reading `polls` base table in public paths for the new column
`results_visibility` is on the base `polls` table but must be consumed via `polls_effective` in all `src/routes`, `src/hooks`, `src/components` paths. The invariant test at `src/__tests__/admin/polls-effective-invariant.test.ts` will catch regressions. `polls_effective` view must expose `results_visibility` and `results_revealed_at` after migration 10.

### Anti-Pattern 4: Querying vote_counts for all suggestions (perf)
When extending `useVoteCounts`, only query counts for polls where the user can actually see results. Sending all suggestion IDs to the `vote_counts` query regardless of visibility wastes bandwidth and could hit PostgREST row limits on large archives.

### Anti-Pattern 5: Making visibility editable in update-poll
`results_visibility` is set at creation; ad-hoc reveal goes through `reveal-poll-results`. The edit page should show the current mode as a read-only badge, not an editable picker, consistent with the one-way reveal principle.
