# Research Summary — v1.2 Admin Visibility Controls

**Project:** WTCS Community Polls
**Domain:** Per-row results-visibility policy retrofit + shadcn/ui drift closure + mobile audit rerun
**Researched:** 2026-05-11
**Confidence:** HIGH

---

## Executive Summary

v1.2 is a focused, low-risk milestone that retrofits per-suggestion results-visibility controls onto an already-shipped voting platform. The new `results_visibility` Postgres enum (`respondents_only` / `public_during` / `public_after_close`) plus a rewritten `vote_counts` RLS SELECT policy are the load-bearing foundation; everything else — a new `reveal-poll-results` Edge Function, admin creation UI selector, and user-facing conditional results render — depends on that foundation landing cleanly. All four research agents converge on the same three-phase build order: Phase 11 (schema + RLS + EF), Phase 12 (admin UI + user UI + UIDN-03 fold-in), Phase 13 (UIDN-02 Lighthouse rerun after production deploy). Zero new npm or Deno dependencies are required; this is a pure schema + code change on the existing v1.1 stack.

The single non-trivial risk is a **P0 RLS leakage** (PITFALLS Pitfall 1): if the new `vote_counts` SELECT policy is written with incorrect boolean logic across the 12-cell matrix (3 visibility modes × voted/not × open/closed), non-voters can query pre-aggregated vote counts directly through the Supabase anon key. The mitigation is mandatory: the full 12-cell invariant test suite must land in the same Phase 11 PR as the migration — blocking merge until all cells pass. The React `canSeeResults` gate in `SuggestionCard` is UX convenience only; RLS is the real security boundary.

Two carry-forward items from v1.1 close alongside SEED-002. UIDN-03 (4 native-`<button>` drift sites in `SearchBar.tsx`, `SuggestionForm.tsx` ×2, `ImageInput.tsx`) folds naturally into Phase 12 since `SuggestionForm.tsx` is already being touched for the `VisibilityPicker` addition. UIDN-02 (Lighthouse mobile rerun with hydration-wait harness fix) is blocked on both the Phase 12 production deploy and the `audit-screenshots.mjs` Plan 02 fix, so it correctly forms Phase 13.

## Key Findings

### Recommended Stack

The existing v1.1 stack is locked and carries forward unchanged. v1.2 adds zero new npm packages and zero new Deno ESM imports — the only new code artefacts are a SQL migration file, one new Edge Function, and a handful of new/modified React components and hooks.

**New for v1.2 only:**

- **Postgres native enum** (`CREATE TYPE public.results_visibility_mode AS ENUM`) — DB-enforced valid values, 4-byte storage, readable strings over Supabase JS client, extensible via `ALTER TYPE … ADD VALUE`. Chosen over `TEXT CHECK (… IN (…))` because a closed three-value set warrants a type object; chosen over state-machine because the feature is a single column + a one-way transition.
- **Migration 10** (`00000000000010_results_visibility.sql`) — one atomic file containing the enum DDL, two column additions (`results_visibility`, `results_revealed_at`), `DROP/CREATE POLICY` on `vote_counts`, `CREATE OR REPLACE VIEW polls_effective`, and updated `create_poll_with_choices` RPC signature. Atomicity is required: splitting across two migrations creates a window where the RLS references a column that doesn't exist in the view yet.
- **`reveal-poll-results` EF (#17)** — standalone function following the exact 16-EF skeleton (`requireAdmin` from `_shared/admin-auth.ts`, service-role client, CORS from `_shared/cors.ts`). Standalone over extending `update-poll` because `update-poll` is locked behind the zero-votes edit gate; the semantics are inverted.
- **`waitForLoadState('networkidle')` in `audit-screenshots.mjs`** — fixes the Plan 02 hydration-wait defect (6 F6 DOM-assertion warnings, byte-identical unauth PNGs). No Playwright version change; this API exists since Playwright 1.0.

**What NOT to add:** `zod`/`yup` for EF validation (existing manual TypeScript guards pattern), `@lhci/cli` (score variance + local-vs-prod gap makes hard CI gates unreliable at $0), React Query/SWR (stack drift for one new query type), toggle-back reveal (trust violation), `<button type="submit">` to `<Button>` migrations without explicit `type` attribute.

### Expected Features

v1.2 ships three work streams. All research classifies every item in these streams as P1 table stakes — there is no "nice to have" in this milestone.

**Must have (SEED-002 table stakes):**
- Three visibility modes at creation with `respondents_only` default — backwards compat with all v1.0/v1.1 data; every comparable platform (Slido, Polly, Simple Poll, Discourse) exposes at least three tiers.
- RLS rewrite on `vote_counts` SELECT policy covering the full 12-cell matrix — the security gate; must land before any UI is wired.
- Admin creation form `RadioGroup` selector with human-readable labels and per-option description copy (not raw enum values) — non-technical esports admin audience requires context.
- `reveal-poll-results` EF + admin "Reveal results now" button with `AlertDialog` confirmation (irreversibility-framed, names the suggestion by title).
- User-facing conditional results render pre-vote when `results_visibility = 'public_during'`.
- Archive honors the policy: `respondents_only` closed suggestions stay gated; `public_after_close` + `public_during` closed suggestions show results to all authenticated users.
- RLS invariant matrix test suite (12 cells) — non-optional, blocks Phase 11 merge.

**Must have (carry-forward):**
- UIDN-03: 4 native-`<button>` drift fixes with correct `variant=` and explicit `type=` attributes.
- UIDN-03: authenticated Pass-A screenshots of `/topics` and `/archive` after fixes land.
- UIDN-02: `audit-screenshots.mjs` hydration-wait fix + Lighthouse mobile rerun.

**Should have (differentiators in v1.2):**
- `results_revealed_at TIMESTAMPTZ` audit column — no polling platform in the competitive analysis surfaces a timestamped reveal record; cheap to add, high future value.
- Bandwagon trade-off tooltip on `public_during` option — non-technical admins need context that generic platforms omit.
- "Reveal results" button replaced with read-only "Revealed [time ago]" badge after successful reveal — prevents re-click confusion.

**Defer to v2+:**
- Toggle-back visibility reveal — anti-feature (trust violation for async community voting).
- Scheduled reveal — needs pg_cron or GH Actions scheduling; out of scope at $0.
- Admin audit log UI for `results_revealed_at` — useful but not blocking v1.2.
- `UIDN-03-FOLLOWUP-LIST-CARDS` (AdminsList/CategoriesList/PromoteAdminDialog hand-rolled list containers) — no FAIL cells in the audit; purely cosmetic; STACK research explicitly calls out that v1.2 ROADMAP must decide whether to include or continue deferring. **Recommendation: defer to v1.3.** The four mandated UIDN-03 drift fixes already close the Key Decision row without requiring the list container cleanup.

**Unresolved: three open questions that need user resolution before plan-phase (see Gaps section).**

### Architecture Approach

v1.2 extends the existing five-layer architecture (Presentation → Routing → State/Data → Backend/EF → Database/RLS) with strictly additive changes. The schema migration is the root of the entire dependency graph — nothing else can proceed until migration 10 is deployed and verified. The SEED-002 track has a strict sequential dependency: migration → RLS rewrite → EF deploy → admin UI → user UI. The UIDN-03 track is parallel (no schema dependency). The UIDN-02 track is externally blocked (must wait for Phase 12 production deploy + harness fix).

**Major new/modified components:**

1. **Migration 10** (`00000000000010_results_visibility.sql`) — enum DDL, two columns, RLS rewrite, view rewrite, RPC update; single atomic file
2. **`reveal-poll-results` EF** (`supabase/functions/reveal-poll-results/index.ts`) — admin-gated one-way reveal; #17 in the EF fleet
3. **`VisibilityPicker` component** (`src/components/suggestions/form/VisibilityPicker.tsx`) — labeled RadioGroup for creation form
4. **`RevealResultsDialog` component** (`src/components/admin/RevealResultsDialog.tsx`) — AlertDialog confirmation for one-way reveal
5. **`useRevealPollResults` hook** (`src/hooks/useRevealPollResults.ts`) — single mutation hook; `reveal(pollId)` → EF invoke
6. **`SuggestionCard` modification** — `canSeeResults` derived boolean replaces binary `userChoiceId` gate
7. **`useVoteCounts` extension** — new `publicPollIds` parameter for non-voter public reads; bounded to polls where user can actually see results (anti-pattern: querying all suggestion IDs regardless of visibility)
8. **`polls_effective` view rewrite** — must explicitly add `results_visibility` and `results_revealed_at` to column list in `CREATE OR REPLACE VIEW` (view uses explicit columns, not `SELECT *`)

**Key patterns to follow:**
- Centralize the three-mode visibility gate as a single `canSeeResults` boolean in `SuggestionCard` — do not scatter across `ResultBars`, `ChoiceButtons`, or parent components.
- All public-path reads of `results_visibility` go through `polls_effective`, never `from('polls')` directly — the `polls-effective-invariant.test.ts` will catch regressions.
- RLS is the authoritative security gate; React conditional render is UX convenience only.
- `update-poll` is NOT extended for visibility changes — `reveal-poll-results` is standalone.
- `results_visibility` is set at creation and is display-only in edit mode (informational badge, not an editable picker).

### Critical Pitfalls

1. **P0 — RLS leakage under new per-row policy** — If the new `vote_counts` USING clause has incorrect boolean logic (wrong operator precedence, missing JOIN to `polls`, `OR` where `AND` is required), non-voters can read pre-aggregated counts directly via the Supabase anon key. **Prevent:** write all 12 matrix cells explicitly in the policy; land the 12-cell invariant test suite in the same Phase 11 PR; block merge until all 12 pass. Emergency recovery if discovered post-merge: `DROP POLICY / CREATE POLICY` via `supabase db push` — no data corruption (vote_counts is read-only), but integrity window exists.

2. **Deploy-window race on enum default** — Migration and EF deploy pipelines are not atomic. If the EF is deployed before the migration, it will fail. If the migration lands but the EF still passes `results_visibility: undefined`, the row silently takes the DB default. **Prevent:** deploy migration first, verify `column_default = 'respondents_only'` via `information_schema`, then deploy EF; EF must validate the value against the allowlist in its body parser.

3. **`polls_effective` view not updated** — The view currently uses explicit column names (not `SELECT *`). If migration 10 adds columns to `polls` but doesn't include a `CREATE OR REPLACE VIEW polls_effective` rewrite, `results_visibility` will not appear in the view and all public-path reads will get `undefined`. **Prevent:** migration 10 must include the `CREATE OR REPLACE VIEW` with the two new columns; re-apply `security_invoker = on` after the view rewrite.

4. **Type=submit lost in UIDN-03 refactor** — shadcn `<Button>` defaults to `type="button"`, not `type="submit"`. Replacing a native `<button>` inside a `<form>` without explicit `type="submit"` breaks form submission silently. The 4 drift sites are all non-submit actions, but verify each site's DOM context before replacing. **Prevent:** always pass `type` explicitly on every Button replacement; run the test suite after each individual site's fix, not only at the end.

5. **Lighthouse rerun before harness fix** — Re-running `audit-mobile.sh` without fixing the `audit-screenshots.mjs` hydration-wait defect will reproduce the same 6 F6 DOM-assertion warnings and byte-identical unauth PNGs. The gate exits non-zero on F6 warnings. **Prevent:** treat the `audit-screenshots.mjs` fix as a prerequisite task inside Phase 13; do not run the Lighthouse rerun script until the fix is committed.

### Phase Ordering Rationale

- **Schema before UI:** Every v1.2 component depends on the `results_visibility` column existing. The RLS policy depends on the column. The view depends on the column. The EF depends on the column. Reversing this order produces runtime errors at every layer.
- **RLS before React gate:** The `canSeeResults` boolean in `SuggestionCard` is UX. The real access control is the `vote_counts` SELECT policy. If Phase 12 shipped without Phase 11 the application would render a "show results" UI that always receives an empty array.
- **UIDN-03 in Phase 12, not a separate phase:** `SuggestionForm.tsx` is already being modified. The two native-button drift sites at lines 140/163 are inside the same file. A separate phase for 4 small button replacements would add planning overhead disproportionate to the change size.
- **UIDN-02 last:** The Lighthouse audit measures the production bundle. Running it before Phase 12 is deployed gives a score that's immediately stale. The harness fix is also prerequisite — no point addressing the external-condition blocker before the code blocker is resolved.
- **All three phases are v1.2:** Nothing here is deferred to v1.3. The three-phase structure is a sequencing decision for safe deployment, not a scope-reduction decision.

## Contradictions and Cross-Agent Conflicts

**No material contradictions found.** Cross-checking the four files against the seven mandated cross-check items:

1. **Phase split consensus** — All four agents name the same three phases in the same order (Phase 11 schema/RLS/EF → Phase 12 UI + UIDN-03 → Phase 13 UIDN-02). Confirmed.

2. **P0 RLS leakage with 12-cell matrix** — PITFALLS names this Pitfall 1 and provides the full 12-cell table. STACK provides the corrected USING clause SQL. ARCHITECTURE provides the shorter additive form of the same policy. FEATURES calls the invariant test suite a "differentiator in correctness confidence." **Minor notation difference:** PITFALLS counts a 12-cell matrix (3 modes × voted/not × open/closed), while STACK and FEATURES describe a 6-cell matrix (3 modes × voted/not, collapsing open/closed into a single state per mode). The 12-cell count is more precise (open vs closed matters for `public_after_close`); the 6-cell count refers to the minimum surface. Use 12 cells in the invariant test implementation. Both are consistent with each other; no contradiction.

3. **Single-migration discipline** — ARCHITECTURE explicitly states: "all belong in the same migration so they are atomic at the DB layer. Splitting them across two migrations creates a window where the RLS is updated but the view doesn't expose `results_visibility` yet." STACK echoes this with "Migration 10 contains…" listing all components together. Confirmed; no contradiction.

4. **Zero new dependencies** — STACK states this explicitly: "v1.2 ships zero new npm or Deno dependencies." No other agent introduces a new package. Confirmed.

5. **RadioGroup + AlertDialog UI choices** — FEATURES provides the RadioGroup rationale (3 mutually exclusive options needing description text), the exact label strings, and the AlertDialog copy. ARCHITECTURE references `VisibilityPicker` using `<RadioGroup>` and `RevealResultsDialog` using `<AlertDialog>`. PITFALLS adds that the reveal button should use `variant="destructive"` (red background signals irreversibility). **Minor tension:** FEATURES recommends `variant="default"` (not destructive red) for the AlertDialog confirm button because "this is a positive action for the community, not data loss." PITFALLS recommends `variant="destructive"` on the trigger button in the kebab menu. These are different elements — the trigger button vs the confirm button inside the modal — and the recommendations are not contradictory: the trigger button uses destructive styling to signal irreversibility; the confirm button inside the AlertDialog uses default/primary styling because at that point the user has read the warning and is confirming. No contradiction; both are correct in context.

6. **Three open questions needing user resolution** — Surfaced by FEATURES; not contradicted by other agents. See Gaps section.

7. **UIDN-03-FOLLOWUP-LIST-CARDS scope** — STACK notes the finding and says "v1.2 ROADMAP should decide whether to include it or continue deferring." No other agent introduces it as required scope. Recommendation above: defer to v1.3.

### Gaps to Address

These are not blockers for starting Phase 11 (schema work), but need resolution before Phase 12 plan-phase:

1. **Does the anon role (unauthenticated visitors) see `public_during` results, or only authenticated-non-voters?**
   FEATURES describes the user-facing conditional results as relevant to "community members" (implied: authenticated). The RLS policy in both STACK and ARCHITECTURE targets `TO authenticated`. If unauthenticated visitors should also see `public_during` results, the RLS policy needs an `anon` role clause and a separate policy. If the answer is "authenticated only," no change needed. **Resolve before Phase 11 plan-phase.**

2. **Does `requireAdmin` need updating, or reuse as-is?**
   Migration 9 updated `is_current_user_admin()` to include `guild_member` check. STACK and ARCHITECTURE both say `requireAdmin` from `_shared/admin-auth.ts` is reused as-is for `reveal-poll-results`. No agent inspected `_shared/admin-auth.ts` to confirm it calls the updated `is_current_user_admin()` rather than a stale check. **Resolve by reading `_shared/admin-auth.ts` at Phase 11 start; flag if it bypasses the migration 9 guild_member gate.**

3. **Does `results_visibility` need to be projected through `polls_effective` for user-facing reads, or is it automatically available?**
   ARCHITECTURE explicitly calls this out: the `polls_effective` view uses explicit column names (not `SELECT *`), so migration 10 must include a `CREATE OR REPLACE VIEW` rewrite. STACK also flags this. This is confirmed architecture — but neither agent verified that the current view definition in migration 5 lists columns explicitly vs uses `SELECT *`. **Resolve by reading migration 5's view DDL at Phase 11 start before writing migration 10.**
