# v1.1 Research: Mobile audit + shadcn polish closure (issue #18)

## TL;DR
Close UIDN-02 and UIDN-03 with two evidence artifacts: a Lighthouse mobile run (CLI one-off, archived JSON+HTML in repo) plus a 6-breakpoint screenshot matrix for UIDN-02; and a per-route shadcn/Maia/Neutral compliance checklist for UIDN-03. Use `lighthouse` CLI with `--preset=desktop`-inverse (default form-factor is mobile) against the production URL `https://polls.wtcsmapban.com`, target Performance >= 90 / Accessibility >= 95 / Best Practices >= 95 / SEO >= 90. Do NOT wire Lighthouse CI into a recurring GitHub Action this milestone -- one-off CLI runs committed under `.planning/closure/` keep us inside the $0 budget and the Netlify legacy free tier with no minutes burn. Both evidence files end in a dated sign-off line that flips PROJECT.md Key Decisions from "Revisit" to "Good".

## Lighthouse audit setup
**Command (one-off, run locally, commit JSON+HTML):**
```bash
mkdir -p .planning/closure/artifacts
npx -y lighthouse@latest https://polls.wtcsmapban.com \
  --form-factor=mobile \
  --throttling-method=simulate \
  --screen-emulation.mobile=true \
  --only-categories=performance,accessibility,best-practices,seo \
  --chrome-flags="--headless=new --no-sandbox" \
  --output=html --output=json \
  --output-path=.planning/closure/artifacts/lh-mobile-home
# Repeat for /topics, /archive, /admin (as guest -> redirect captured), each suggestion detail.
```
Targets (set as warn/error if we ever promote to Lighthouse CI later):
- categories:performance >= 0.90 (warn 0.85)
- categories:accessibility >= 0.95 (error 0.90)
- categories:best-practices >= 0.95
- categories:seo >= 0.90
- LCP < 2.5s, TBT < 200ms, CLS < 0.1, tap-targets passing, viewport audit passing (mobile-first signals).

**CI vs one-off:** One-off this milestone. Free-tier GH Actions minutes are scarce and Netlify deploy previews would need an LHCI server or temporary public storage; Lighthouse CI is a v1.2 candidate (`@lhci/cli@0.15.x` + `treosh/lighthouse-ci-action`) once we have a real perf budget to defend.

## shadcn Maia/Neutral compliance checklist
Run per route. Each item PASS / FAIL / N/A with one-line note.

1. `components.json` declares `baseColor: "neutral"` and `cssVariables: true` (currently TRUE; note `style` is `"new-york"` -- reconcile vs DESIGN-SYSTEM.md "Maia" claim before sign-off).
2. `src/index.css` defines the full Neutral token set (`--background`, `--foreground`, `--primary`, `--secondary`, `--muted`, `--accent`, `--destructive`, `--border`, `--input`, `--ring`, `--card`, `--popover`, sidebar + chart vars) for both `:root` and `.dark`.
3. Every surface uses `bg-background` / `text-foreground` / `bg-card` / `bg-muted` -- no raw `bg-white`, `bg-gray-*`, `text-black`, hex literals, or `bg-[#...]` arbitrary values.
4. All buttons render through `<Button>` from `@/components/ui/button` using documented `variant` (`default|secondary|destructive|outline|ghost|link`) and `size` props -- no ad-hoc `<button className="...">`.
5. Inputs/textareas/selects/labels use `@/components/ui/{input,textarea,select,label}`; no native `<input>` styled inline.
6. Cards use `<Card>/<CardHeader>/<CardTitle>/<CardContent>`; no hand-rolled card divs with custom rounded/shadow combos.
7. Spacing uses Tailwind scale (`p-2/p-4/p-6/gap-4/space-y-6`) -- flag any `p-[13px]`, `m-[27px]`, `w-[331px]` arbitrary values.
8. Radius is consistent: `rounded-md` for inputs/buttons, `rounded-lg` for cards/sheets/dialogs (Maia warmth = generous radius; verify no `rounded-sm`/`rounded-none` drift).
9. Typography: Inter is loaded and applied via body `font-family` (confirmed in `index.css`); headings use Tailwind type scale (`text-xl/2xl/3xl font-semibold`) consistently across `archive.tsx`, `topics.tsx`, `index.tsx`, admin tabs.
10. Icons come from `lucide-react` only; sized via `h-4 w-4` / `h-5 w-5` Tailwind classes; no inline SVG one-offs in `Navbar`, `MobileNav`, kebab menus, dialogs.
11. Dark-mode parity: every route renders cleanly under `.dark` (toggle via `next-themes`); contrast >= WCAG AA (Lighthouse `color-contrast` audit confirms).
12. No prop overrides that drift from preset (e.g. `<Button className="bg-blue-500">` overriding `--primary`); only layout/spacing utility additions allowed via `cn()`.
