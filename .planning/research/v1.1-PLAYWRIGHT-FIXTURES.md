# v1.1 Research: Playwright fixture/seed isolation (issues #11, #12, #13)

## TL;DR

Adopt one convention: **every E2E spec scopes its list locator to `[E2E]`-prefixed cards via `Locator.filter({ hasText: /^\[E2E/ })`** (Playwright's canonical filter primitive — see Sources). That single rule fixes #11 and #13 today and inoculates future specs against canonical-seed drift. #12 is a different bug class (missing precondition, not isolation) — fix by **casting a vote in-test** before asserting the post-vote string. Keep SQL seeds for static reference data; layer Playwright **test-scoped fixtures** for per-test mutable state. Enforce the convention with a 10-line ESLint custom rule in `eslint.config.js`.

## The convention (preventive)

> **Rule E2E-SCOPE-1**: Any locator that resolves a list of `data-testid="suggestion-card"` (or any other shared-DB row collection) MUST be filtered to `[E2E]`-prefixed entries before being counted, indexed, or asserted on. Use `.first()` only on a pre-filtered locator.

Enforcement (cheap, no new deps — uses ESLint flat config already in repo):

```js
// eslint.config.js — append rule (no-restricted-syntax is a built-in ESLint rule; no plugin import needed)
{
  files: ['e2e/tests/**/*.ts'],
  rules: {
    'no-restricted-syntax': ['error', {
      selector: "CallExpression[callee.property.name=/^(toHaveCount|first|nth)$/] > :not(CallExpression[callee.property.name='filter'])",
      message: 'E2E-SCOPE-1: filter to [E2E] prefix before counting/indexing list locators. See .planning/research/v1.1-PLAYWRIGHT-FIXTURES.md',
    }],
  },
}
```

This catches `cards.toHaveCount(N)` and `cards.first()` when not preceded by `.filter(...)`. Pure built-in rule, $0 cost — no extra deps.

### #12 browse-respond.spec.ts:45

Root cause is missing precondition (zero seeded votes), not seed-layering. The test name says "responds, sees live results" — the vote is the test, not the assertion. Fix: scope to a fixture card, vote, assert `>=1`:

```diff
-  const firstCard = page.getByTestId('suggestion-card').first()
+  // E2E-SCOPE-1: only operate on fixture-seeded cards.
+  const firstCard = page
+    .getByTestId('suggestion-card')
+    .filter({ hasText: /^\[E2E/ })
+    .first()
   await expect(firstCard).toBeVisible()
   // ... existing collapsedTrigger + firstChoice.click() flow ...
   await firstChoice.click()
-  await expect(firstCard.getByText(/\d+\s+total response/i)).toBeVisible({ timeout: 10_000 })
+  // After our own vote, count must be >= 1. Tightens 0-vote false positive.
+  await expect(firstCard.getByText(/[1-9]\d*\s+total response/i)).toBeVisible({ timeout: 10_000 })
```

The `firstChoice.click()` already submits a vote — the only change to behavior is the regex (`[1-9]\d*` instead of `\d+`), which forces a non-zero count and proves the vote round-tripped.

## Should we migrate to Playwright fixtures?

**Hybrid: keep SQL for static reference data, add a Playwright test-scoped fixture for per-test mutable state.** Do not delete `e2e/fixtures/seed.sql`.

Rationale:
- **Categories, admin profile, fixture users**: shared, read-only, expensive to recreate per-test → keep in SQL seed (already idempotent via `ON CONFLICT`).
- **Polls/votes specs assert on**: per-test, mutable, source of #12 and #13 drift → migrate to a `freshPoll` test-scoped fixture that `INSERT`s + `DELETE`s around `await use()`. Playwright's [fixture composition](https://playwright.dev/docs/test-fixtures) is the canonical pattern; test-scoped means automatic cleanup, no cross-test pollution.

Sketch (do in v1.1 phase 2, after the convention lands):

```ts
// e2e/fixtures/poll-fixture.ts
import { test as base } from '@playwright/test'
import { adminClient } from '../helpers/admin-client'

export const test = base.extend<{ freshPoll: { id: string; title: string } }>({
  freshPoll: async ({}, use, testInfo) => {
    const title = `[E2E] ${testInfo.title} ${Date.now()}`
    const { data } = await adminClient.from('polls').insert({ title, /* ... */ }).select().single()
    await use({ id: data.id, title })
    await adminClient.from('polls').delete().eq('id', data.id)
  },
})
```

Cost: ~1 day. Benefit: kills the entire bug class (#12 + #13 family) and removes the `[E2E]` filter requirement for fixture-using specs.

## Open questions

1. Should the `[E2E]` prefix become a DB CHECK constraint on a `polls.is_test` column to make the convention machine-enforceable on the seed side too? (Defer — adds schema churn.)
2. Worker-scoped fixture for `freshPoll` to amortize INSERT cost? Risk: cross-test contamination defeats the point. Recommend test-scoped only.
3. Should `e2e/fixtures/seed.sql` move all mutable poll rows out, leaving only categories + auth users? (Yes, after step-2 fixture migration.)
