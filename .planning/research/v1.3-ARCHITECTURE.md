### Work-Stream 1: UIDN-02 Aggressive Perf-Budget Pass

#### Route-level code splitting

**Current state**: `vite.config.ts` uses `tanstackRouter({ autoCodeSplitting: true })`. This flag instructs the TanStack Router Vite plugin to automatically emit `createLazyFileRoute` wrappers for every route file at code-generation time ŌĆö it transforms `createFileRoute(...)({ component: Foo })` into a lazy boundary automatically. All 10 route files use `createFileRoute`, not `createLazyFileRoute`, but `autoCodeSplitting: true` handles the split transparently.

**What this means for v1.3**: Route-level splitting is ALREADY WIRED. The `autoCodeSplitting: true` plugin option is the integration point; no per-route `createLazyFileRoute` conversion is needed. The `tsr generate` step writes `routeTree.gen.ts` with the lazy wrappers baked in.

**Verify the split is working**: `npm run build` output should show separate chunks per route (e.g. `admin-index-<hash>.js`, `topics-<hash>.js`). If chunks are not separate, the issue is in `routeTree.gen.ts` (regenerate with `npm run generate`).

**Admin route lazy-load**: The `/admin/*` subtree (three routes: `admin/index.tsx`, `admin/suggestions/new.tsx`, `admin/suggestions/$id.edit.tsx`) already splits from the public routes via `autoCodeSplitting`. No manual intervention needed unless the bundle visualizer shows admin code leaking into the main chunk ŌĆö in that case, confirm `tanstackRouter` plugin is running before `react()` in the plugin array (it is: line 15 of `vite.config.ts`).

**`routeTree.gen.ts`**: Auto-generated, never manually edited. Regenerated on each build. Not a v1.3 touch point except to verify post-build output.

#### Image optimization tooling

**Current state**: No image optimization plugin in `vite.config.ts`. One application image: `src/assets/wtcs-logo.png` (PNG source). User-uploaded images are served from Supabase Storage or external URLs ŌĆö not in the build pipeline.

**Integration point**: `vite.config.ts` plugin array (slot before `sentryVitePlugin`, after `tailwindcss()`).

**Recommended approach at $0 budget**: `vite-imagetools` (MIT, zero build-time cost) transforms `?format=webp` query imports at build time. Alternative: manual one-time conversion of `src/assets/wtcs-logo.png` ŌåÆ `wtcs-logo.webp` + `wtcs-logo.avif` using a local `sharp` invocation, then swap the `<img>` `src` to the WebP path with a PNG fallback `<picture>` element.

**Plugin ordering constraint**: Any image plugin must appear BEFORE `sentryVitePlugin` in the plugins array. The existing comment in `vite.config.ts` ("Sentry plugin must be LAST ŌĆö tree-shaking landmine") applies equally to any asset-transform plugin. The safe order is: `tanstackRouter` ŌåÆ `react` ŌåÆ `tailwindcss` ŌåÆ `[image-plugin]` ŌåÆ `sentryVitePlugin`.

**Source of truth for images**: `src/assets/` for app-owned images (currently just `wtcs-logo.png`). `public/` for passthrough assets (favicons). Supabase Storage for admin-uploaded poll images ŌĆö these are CDN-served, not in the build. No build-time transforms needed for Storage images; WebP negotiation happens via the Storage CDN headers if enabled.

**CDN-time transforms**: Supabase Storage does not offer automatic format negotiation on the free tier. At $0 budget, build-time conversion of the one `src/assets/` image is the only lever.

#### Font subsetting

**Current state**: `src/index.css` line 133 declares `font-family: Inter, ui-sans-serif, system-ui, sans-serif` as the body font. Inter is referenced by name but there is NO `@font-face` declaration in `src/index.css` and NO `<link rel="preload">` or Google Fonts `<link>` in `index.html`. **Integration points for font subsetting**:

- `src/index.css`: `@font-face { font-family: 'Inter'; src: url('/fonts/inter-subset.woff2') format('woff2'); font-display: swap; }` if self-hosting
- `public/fonts/`: destination for subset WOFF2 file
- Build-time subset script: one-time, not wired into Vite ŌĆö use `pyftsubset` or `glyphhanger` offline, commit the resulting WOFF2

**Compose with Sentry plugin**: Font subsetting is a static-asset concern, not a JS transform. No interaction with `sentryVitePlugin` ordering.

#### Critical CSS extraction

**Current state**: Tailwind CSS v4 via `@tailwindcss/vite` plugin. Tailwind v4 generates atomic utility classes; it does not produce a large monolithic CSS bundle. The `@tailwindcss/vite` plugin handles PurgeCSS-equivalent dead-class elimination at build time. No separate critical-CSS extraction plugin is in use.

**Feasibility assessment**: True critical-CSS inlining (extracting above-the-fold CSS into a `<style>` tag in `index.html`) requires a tool like `critters` or `vite-plugin-critical`. These tools are complex to configure for SPAs (they render a static snapshot and extract used CSS, which misses JS-driven class additions). At this scale (Tailwind generates only ~20-40 KB gzipped CSS for this app), the render-blocking CSS penalty is low. The highest-yield action is ensuring the CSS chunk has a `<link rel="preload">` ŌĆö which Vite does automatically for extracted CSS chunks.

**Recommendation**: Skip a separate critical-CSS extraction plugin in v1.3. The Tailwind v4 + Vite bundle already minimizes CSS size. If Lighthouse flags render-blocking CSS, the intervention is `<link rel="preload" as="style">` in `index.html` ŌĆö but Vite's output HTML injection already handles this for the extracted CSS chunk. Verify in build output before adding tooling.

**Compose with existing plugins**: If a critical-CSS plugin IS added, it must run AFTER `react()` and `tailwindcss()` but BEFORE `sentryVitePlugin`. The constraint is firm: Sentry last.

**Integration point**: `src/routes/__root.tsx` or per-route files. TanStack Router supports `<Link preload="intent">` and `router.preloadRoute()`. No prefetch tuning is currently wired.

**No new files required**: Prefetch hints are props on `<Link>` components (already used in Navbar/nav components) or on `createRouter({ defaultPreload: 'intent' })` in `src/main.tsx`. The `createRouter` call is at line 42 of `src/main.tsx` ŌĆö this is the single hook point for global prefetch behavior.

**Current state ŌĆö ALREADY RESOLVED**: `vite.config.ts` already contains `rolldownOptions: { output: { keepNames: true } }` at lines 33-35. Issue #19 is shipped. **Compose concern**: `sentryVitePlugin` with `sourcemaps.filesToDeleteAfterUpload: './dist/**/*.map'` will also delete any `.map` files produced by the visualizer or image plugins ŌĆö but neither `rollup-plugin-visualizer` nor image plugins emit `.map` files. No collision.

## Patterns to Follow

### Pattern 1: Vite plugin ordering (anti-footgun)
Sentry vite plugin must always be the LAST plugin in the `plugins` array. The existing comment in `vite.config.ts` documents this. Any new plugin (visualizer, image optimizer) must be inserted before `sentryVitePlugin`. Safe order: `tanstackRouter` ŌåÆ `react` ŌåÆ `tailwindcss` ŌåÆ `[visualizer]` ŌåÆ `[imageOptimizer]` ŌåÆ `sentryVitePlugin`.

### Pattern 2: Migration idempotency (CREATE OR REPLACE)
All 7 SECURITY DEFINER patches use `CREATE OR REPLACE FUNCTION`, not `DROP FUNCTION` + `CREATE`. This preserves function OID stability (important for triggers that reference the function by OID internally), avoids breaking `CREATE TRIGGER ... EXECUTE FUNCTION` references, and is safe to re-run.

### Pattern 3: E2E fixture scoping (E2E-SCOPE-1)
Any locator that counts or indexes a collection of `suggestion-card` elements must be filtered to `[E2E]`-prefixed titles before count/index operations. `browse-respond.spec.ts` uses the `freshPoll` fixture title for exact match (stronger than prefix). `filter-search.spec.ts` uses the `E2E_TITLE` regex. Both patterns satisfy the constraint.

### Pattern 4: React 19 error hooks + deduplication
`createRoot` receives all three error hooks. Each hook calls `Sentry.reactErrorHandler(callback)`. `Sentry.dedupeIntegration()` is pinned in `Sentry.init` to suppress triple-fire events. `Sentry.ErrorBoundary` has an explicit `onError` belt as defense-in-depth. This is the complete pattern from v1.1 research ŌĆö already implemented in `src/main.tsx`.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Adding `createLazyFileRoute` manually
`autoCodeSplitting: true` in the TanStack Router Vite plugin handles lazy route conversion automatically. Manually converting routes to `createLazyFileRoute` would be redundant and could conflict with the plugin's code-generation output in `routeTree.gen.ts`.

### Anti-Pattern 2: Moving sentryVitePlugin before other plugins
The sourcemap upload hook in `sentryVitePlugin` must fire after all code transforms are complete. Any plugin reordering that moves Sentry earlier will produce incomplete sourcemaps. The existing comment guards this.

### Anti-Pattern 3: Dropping SECURITY DEFINER functions instead of replacing
`DROP FUNCTION` + `CREATE FUNCTION` on trigger functions breaks the trigger's internal function reference. Use `CREATE OR REPLACE FUNCTION` exclusively.

### Anti-Pattern 4: Adding route-level `errorComponent` as a substitute for #17 fix
Route-level `errorComponent` in TanStack Router provides per-route recovery UI ŌĆö it does not fix the Sentry capture transport gap. The fix is in `createRoot` hook wiring, not in route configuration. Both can coexist but are independent concerns.

### Anti-Pattern 5: Treating #11, #12, #13 spec fixes as requiring new fixture infrastructure
The `poll-fixture.ts` `freshPoll` fixture is already wired and working. The fixes for #11, #12, #13 are either already applied (based on reading the current spec files) or require only comment additions. No new fixture files, no new seed SQL, no new helpers.
