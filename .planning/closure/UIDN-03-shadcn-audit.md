## 3. Drift findings + fixes

The matrix above contains **4 FAIL cells, all in item 4 (Button component adoption)**, plus 9 footnotes ([a]–[i]) carrying nuance / inherited-evidence rationale / non-FAIL transparency notes. Findings list, ordered by footnote tag for cross-referencing:

- **[a] Inherited LandingPage evidence on `/topics`, `/archive`, `/admin (unauth)` Pass-A captures.** The Pass-A unauth screenshots for these three routes all redirect to the unauth `/` `LandingPage` (correct `AuthGuard` behavior — supersedes UIDN-02's "loading shell" framing; see Method § "Audit-data-quality correction"). MANIFEST.json sha256 confirms the 4-route × 1-width unauth PNG group is byte-identical to the LandingPage capture at the matching width. Cells inheriting `/`'s evidence: item 3 verdicts PASS (LandingPage uses semantic tokens cleanly); item 6 verdicts N/A (the LandingPage uses the vendored `Card` component; the hand-rolled `divide-y border rounded-md` divs flagged in [e] live in the SuggestionList / admin list components, which require auth to render and were therefore not captured in the unauth Pass-A set). Visual evidence for these three routes' AUTHENTICATED state is not in the 42-PNG corpus; the cells are inferred from `/`'s evidence + targeted src-grep against the route-specific component imports. Confirmation of the authenticated state is queued for the deferred Pass-A rerun (UIDN-02 follow-up rerun, which will capture authenticated `/topics` and `/archive` once a fixture-admin session can be plumbed through the prod-target screenshotting path).

- **[b] item 4 / `/topics` + `/archive` (authenticated state, not captured visually):** `src/components/suggestions/SearchBar.tsx:22` — the clear-search "×" button uses a native `<button className="absolute …">` instead of a `<Button variant="ghost" size="icon">` from `@/components/ui/button`. SearchBar is rendered on both `/topics` and `/archive` when authenticated (they share the SuggestionList + filter UI). The drift is identical between the two routes (one source location, two render contexts) so a single fix resolves both FAIL cells. Visual confirmation in Pass-A is NOT available (SearchBar only renders in the authenticated state, which is not in the captured PNG set per [a]); evidence is `src`-grep-only this audit. **Defer to v1.2** — cosmetic; the clear-X is functional and accessible (it's a real `<button>`, not a `<div onClick>`), but it bypasses the documented shadcn `<Button>` API and so fails item 4's strict rubric. Proposed v1.2 ROADMAP-id: `UIDN-03-FOLLOWUP-NATIVE-BUTTONS`.

- **[c] item 4 / `/admin/suggestions/new` + `/admin/suggestions/$id/edit` (Pass-B authenticated captures available):** TWO native-button drift sites in the admin form trio: (1) `src/components/suggestions/form/SuggestionForm.tsx:140,163` — the "Back to admin" link-style native `<button>` above the form heading (should be `<Button variant="link">` or `<Link>` from TanStack Router with appropriate styling); (2) `src/components/suggestions/form/ImageInput.tsx:108` — the drop-zone trigger native `<button>` (legitimate as a drag-drop affordance, but could be wrapped in `<Button asChild>` to retain the documented Button styling). Both render on both `/admin/suggestions/new` AND `/admin/suggestions/$id/edit` (same components, two render contexts). **Visible in Pass-B PNGs** `bp-375-admin-suggestions-new.png` and `bp-375-admin-suggestions-id-edit.png` — the "← Back to admin" link is the small text link above the form heading and the "Drop an image here, or click to browse" upload card is the drop zone. **Defer to v1.2** — functional (the form submits correctly; the navigation link works; the drop zone accepts files), but bypasses the documented shadcn `<Button>` API. Same v1.2 ROADMAP-id as [b]: `UIDN-03-FOLLOWUP-NATIVE-BUTTONS`.

- **[d] item 5 / `/admin/suggestions/{new,id-edit}` — TWO native `<input>` instances are intentional, NOT drift:** (1) `src/components/suggestions/form/ImageInput.tsx:86` is `type="file"` with `className="hidden"` (idiomatic shadcn file-picker pattern — the visible button is programmatically wired via `useRef` + `fileRef.current?.click()` to delegate to the hidden native file input; this is the documented shadcn pattern for file uploads since shadcn does not ship a `<FileInput>` primitive); (2) `src/components/suggestions/form/TimerPicker.tsx:127` is `type="datetime-local"` styled with shadcn token classes (`rounded-md border bg-background px-3 py-2 text-sm`) because shadcn does not ship a Datetime input primitive — the native `<input type="datetime-local">` is the shadcn-recommended escape hatch with the standard input token classes applied. Both render correctly with new-york tokens. **Verdict: PASS** (not drift; documented escape-hatch usage of native primitives in places where shadcn intentionally has no primitive).

- **[e] item 6 / `/admin/sug/{new,id-edit}` — 4 list-container divs use `divide-y border rounded-md` (hand-rolled list containers; could use `<Card>` wrapper):** `src/components/admin/AdminsList.tsx:91,101`, `src/components/admin/CategoriesList.tsx:197`, `src/components/admin/PromoteAdminDialog.tsx:88`. These render on the admin list TABS (the `/admin` route's authenticated state showing the Suggestions / Admins / Categories tab triple), NOT directly on the form-screen captures `bp-*-admin-suggestions-{new,id-edit}.png` — the form screens delegate to inline form fields, not list components. The Categories popover visible in `bp-375-admin-suggestions-new.png` ("Lineup Changes" Select) uses `<Select>` primitive from `@/components/ui/select` (PASS). **Per-route cell verdict for the form-screen columns is PASS** (no list components rendered there). The AdminsList/CategoriesList/PromoteAdminDialog drift surfaces on the parent `/admin` route's AUTHENTICATED state — which is not in the 42-PNG set per [a]. **Defer to v1.2** — cosmetic; the hand-rolled list containers render identical visuals to a `<Card>`-wrapped list (the only material delta is `rounded-md` vs `rounded-xl` for the card edge and the `<Card>`'s `bg-card` semantic token vs the bare `border bg-background` inheritance). Proposed v1.2 ROADMAP-id: `UIDN-03-FOLLOWUP-LIST-CARDS`.

- **[f] item 7 / spacing scale — 9 `min-h-[44|56|64|72px]` matches in admin components + Navbar are WCAG touch-target / list-item fixed heights, NOT arbitrary-value drift.** Files: `src/components/layout/Navbar.tsx:29` (44px — WCAG AA touch target), `src/components/admin/AdminsList.tsx:95,119` (64px — list-row min-height), `src/components/admin/AdminSuggestionsTab.tsx:165` (72px — suggestion-row skeleton), `src/components/admin/CategoriesList.tsx:177,199,240` (56px — category-row min-height), `src/components/admin/PromoteAdminDialog.tsx:92` (44px — touch target), `src/components/admin/AdminSuggestionRow.tsx:46` (72px — suggestion-row), `src/components/admin/AdminTabs.tsx:35,41,47` (44px — tab buttons). All values are semantic touch-target / list-density specifiers, not arbitrary spacing drift. ONE `p-[3px]` lives in vendored `src/components/ui/tabs.tsx:27` (preserved as a shadcn-source line — the vendored shadcn primitive uses arbitrary value internally in the `tabs-list` recipe; not local drift). `<svg>` icon sizing via `h-5 w-5` Tailwind classes is item-10 lucide convention reused (allowed). **Verdict: PASS with rationale.**

- **[g] item 8 / radius — 11 `rounded-xl` matches all live on Card-style components matching the vendored new-york `src/components/ui/card.tsx:10` default.** Card recipe: `rounded-xl border bg-card py-6 ... shadow-sm`. The plan-body checklist (line 80 of `09-04-PLAN.md`) wrote "rounded-lg for cards/dialogs" as the new-york default; the vendored `ui/card.tsx` in this repo uses `rounded-xl`. **Source-of-truth tension** (also documented in § "Item descriptions" item 8): the vendored source wins — the checklist item-8 description is itself slightly inaccurate (Maia and new-york both ship Card with `rounded-xl` in current shadcn dist; the plan-body prose is what should be updated, not the code). Buttons + inputs use `rounded-md` per `ui/button.tsx` + `ui/input.tsx` (PASS). **Verdict: PASS** — radius is consistent with the vendored primitives.

- **[h] item 10 / lucide-only — ONE inline `<svg>` in `src/components/auth/LandingPage.tsx:18` for the Discord brand mark (24×24, sized via `h-5 w-5` matching lucide convention, `aria-hidden="true"`).** Discord brand glyph is not in `lucide-react` (lucide intentionally excludes proprietary brand marks per their issue #2143 — brand glyphs are vendor-trademark-encumbered and lucide upstreams to the simple-icons project for those). This is a documented brand-icon exception, not drift. The inline SVG is well-formed (correct `viewBox`, single path, correct accessibility attrs). **Verdict: PASS with brand-exception footnote** — the LandingPage's "Sign in with Discord" CTA needs the Discord brand mark to communicate the auth provider, and the inline-SVG-with-shadcn-sizing pattern is the documented escape hatch. Routes inheriting `/`'s evidence per [a] (`/topics`, `/archive`, `/admin` unauth) carry the same brand-mark over their unauth-redirect frames — hence three GLOBAL[h] cells.

- **[i] item 11 / dark-mode parity — verified globally via design-system token plumbing + Lighthouse cross-confirmation.** Token plumbing: `src/index.css:127-135` has `body { @apply bg-background text-foreground; }` and the full token set duplicated in both `:root` (light) and `.dark` (dark) variants (item 2 verified — see `/tmp/uidn03-globals.txt`).
