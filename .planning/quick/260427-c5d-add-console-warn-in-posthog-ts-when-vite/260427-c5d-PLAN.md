# 260427-c5d — Add dev-mode warning when VITE_POSTHOG_KEY missing

## Why

Phase 6 UAT Test 4 surfaced a silent failure: `initPostHog()` short-circuits at `src/lib/posthog.ts:14-15` when `VITE_POSTHOG_KEY` is unset, returning the uninitialized `posthog` stub. Subsequent `posthog.opt_in_capturing()` calls flip an in-memory flag on the stub — no network, no warning. The user spent a UAT cycle reporting this as a bug; it was actually a missing dev env var.

A one-line `console.warn` in dev when the key is missing eliminates this trap.

## Tasks

### Task 1 — emit dev warning when VITE_POSTHOG_KEY missing

**Files:** `src/lib/posthog.ts`

**Action:** In `initPostHog()`, after the existing `if (!key) return posthog` short-circuit, prepend a `console.warn` gated on `import.meta.env.DEV`:

```ts
if (!key) {
  if (import.meta.env.DEV) {
    console.warn('[posthog] VITE_POSTHOG_KEY not set — analytics disabled. Set it in .env.local to enable PostHog in dev.')
  }
  return posthog
}
```

**Verify:**
- Start `npm run dev` with `VITE_POSTHOG_KEY` unset → warning visible in browser console once on first navigation
- Start with key set → no warning
- `npm run build` (production) → no warning emitted (DEV is false)
- `npx vitest run` → all 386 tests still pass (DEV is false in vitest by default; warning won't surface in test output)
- `tsc -b --noEmit` → clean

**Done when:**
- Warning appears exactly once per dev page-load when key is missing
- No regression in test suite or production bundle
- Atomic commit `fix(260427-c5d):` describing the dev-quality improvement
