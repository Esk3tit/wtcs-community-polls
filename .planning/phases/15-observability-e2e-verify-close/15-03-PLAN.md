  <action>
    Probe step (executor runs locally first):
    1. From the repo root, determine whether the production build requires Supabase env vars to be set (i.e., does `src/lib/supabase.ts` throw at module-load when the values are absent?). The outcome dictates the next step.

    **Unset vs empty-string distinction (cross-AI replan, CR Finding 9):** `SENTRY_AUTH_TOKEN= VITE_SUPABASE_URL= ... npm run build` sets the vars to the EMPTY STRING, which is observably different from UNSET (`undefined`) in the module-load check. `src/lib/supabase.ts` may treat `''` as a valid (but invalid-URL) value and throw a different error path than the truly-undefined case. Use `env -u SENTRY_AUTH_TOKEN -u VITE_SUPABASE_URL -u VITE_SUPABASE_ANON_KEY npm run build` instead — `env -u` REMOVES the variable from the environment, so `import.meta.env.VITE_SUPABASE_URL` resolves to `undefined` (the case CI experiences on a fresh runner). On macOS/Linux this is the portable form; on Windows `Remove-Item Env:VITE_SUPABASE_URL` then `npm run build`. After the probe, also `grep -n "VITE_SUPABASE_URL\\|VITE_SUPABASE_ANON_KEY" src/lib/supabase.ts` to confirm which exact env-presence check the module-load throw uses — Case A vs Case B must match the actual code, not an assumption.

    Case A — build succeeds with empty Supabase env vars:
      Insert the two new steps with NO `env:` block on either:
      - Step name `Build (for sourcemap-names verify)` — bare `run: npm run build`
      - Step name `Verify sourcemap function names preserved` — bare `run: node scripts/verify-sourcemap-names.mjs`

    Case B — build fails without Supabase env vars (the module-load throw fires):
      Insert the two new steps; the Build step gets an `env:` block with NON-SECRET placeholders mirroring the e2e job's `Build app` pattern (15-PATTERNS.md `Existing 'Build app' step` block):
      - `VITE_SUPABASE_URL: http://localhost:54321`
      - `VITE_SUPABASE_ANON_KEY: placeholder-key-for-sourcemap-verify-only`
      The verify step runs without any env block.
      (The Sentry sourcemap upload step inside vite.config.ts only runs when `SENTRY_AUTH_TOKEN` is set — leaving it unset in this job ensures the local build does not attempt upload, which is correct: we want to inspect dist/, not publish to Sentry from this job.)

    Insertion location (in either case): AFTER the existing `- run: npm test -- --run` line (currently the third `run` step inside `lint-and-unit`), BEFORE the existing `- name: npm audit (non-blocking)` step.

    Step name comments and step name strings outside `src/` may cite decision IDs — recommend citing `D-08` and `OBSV-04(a)` in a YAML comment line directly above the two new steps. Comment style: mirror the existing `# D-07 — PR gate: ...` and `# D-16 §2 — npm ci only` patterns already present in `.github/workflows/ci.yml`.

    Do NOT add a `SENTRY_AUTH_TOKEN` env entry — the build for the verify step should NOT upload sourcemaps to Sentry (the sourcemap upload is a separate concern handled in production Netlify builds and tracked under OBSV-04(b) via `sentry-cli`, not under OBSV-04(a)).
