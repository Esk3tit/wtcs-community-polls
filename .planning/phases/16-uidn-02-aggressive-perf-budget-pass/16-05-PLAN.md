  <what-built>
    Per RESEARCH § Common Pitfalls > Pitfall 7 and CONTEXT § Deferred Ideas (`vite-imagetools` / `sharp` are explicit anti-features), the PNG → WebP conversion is a one-time manual operator step on the developer machine. This step is the unavoidable human-action in plan 05 — there is no CLI/API surface Claude can drive that the developer can let it run for this kind of asset transformation under the project's explicit anti-feature stance.
  </what-built>
  <how-to-verify>
    1. Confirm `cwebp` is installed: `which cwebp` should return a path. If not present, install via `brew install webp` (macOS) or the OS-equivalent. The Homebrew `webp` formula bundles `cwebp`.
    2. From the project root, run: `cwebp -q 80 -m 6 src/assets/wtcs-logo.png -o src/assets/wtcs-logo.webp`.
    3. Compare sizes: `ls -la src/assets/wtcs-logo.png src/assets/wtcs-logo.webp`. The PNG is 9915 bytes (verified 2026-05-27 per RESEARCH § Component Responsibilities). If the WebP is NOT smaller than the PNG, run the lossless fallback: `cwebp -lossless src/assets/wtcs-logo.png -o src/assets/wtcs-logo.webp` (overwrites the q=80 attempt).
    4. If neither variant produces a smaller file, accept the neutral outcome — RESEARCH § Pitfall 7 is explicit that for very small RGBA images the WebP may match or exceed PNG bytes. The `<picture>` markup is still the right shape; modern browsers benefit from WebP's faster decode path regardless.
    5. Record the chosen `cwebp` flags and the actual WebP file size in the plan SUMMARY for traceability.
    6. Confirm the PNG is NOT deleted: `test -f src/assets/wtcs-logo.png` must still pass. The PNG is the `<picture>` fallback per UI-SPEC § Image Loading Contract.
  </how-to-verify>
  <resume-signal>Type "approved" once `src/assets/wtcs-logo.webp` exists, the PNG is still present, and the actual byte size + cwebp flags used are recorded for SUMMARY capture.</resume-signal>
