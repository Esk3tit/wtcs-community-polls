  <action>
    Edit `vite.config.ts` to extend the existing `build.rolldownOptions.output` object. The object currently contains `keepNames: true` only — Phase 16 plan 04 adds `manualChunks` alongside it (do NOT replace; do NOT relocate to a `rollupOptions` key — Pitfall 8 is explicit).

    **Matcher form: prefer the FUNCTION form up front.** Per the cross-AI review (MEDIUM finding): the object form's behavior under Rolldown is NOT verified in this repo — Rolldown may resolve object-keyed package strings differently than Rollup, and the object form can silently pull transitive deps into the named chunk in ways that are harder to reason about. The boundary-anchored FUNCTION form gives explicit, auditable control over exactly which `node_modules/<pkg>/` paths land in each chunk. Use this as the PRIMARY shape:

    `manualChunks: (id) => { if (/[\\/]node_modules[\\/](react|react-dom|scheduler)[\\/]/.test(id)) return 'vendor-react'; if (/[\\/]node_modules[\\/]posthog-js[\\/]/.test(id)) return 'vendor-posthog'; }`

    The regex anchors on `/node_modules/<pkg>/` boundaries — `id.includes('react')` is FORBIDDEN per Pitfall 2 (it catches @tanstack/react-router, @radix-ui/react-*, @sentry/react, react-error-boundary, @testing-library/react). The `react|react-dom|scheduler` alternation deliberately INCLUDES `scheduler` (React's own internal runtime dependency) so the React family stays in one cache-stable chunk — the review's "vendor-react contains only react + react-dom may be too strict" finding is addressed by treating the React runtime family (react + react-dom + scheduler) as the unit, not literally only two packages. The `posthog-js` package directory contains both `posthog-js` and `posthog-js/react` exports — a single matcher `/node_modules/posthog-js/` covers both.

    The OBJECT form (`manualChunks: { 'vendor-react': ['react', 'react-dom'], 'vendor-posthog': ['posthog-js', 'posthog-js/react'] }`) is acceptable ONLY if the function form proves problematic AND the object form is then empirically verified against the treemap in this repo (Task 2). Do not assume object-form behavior; if used, prove it.
