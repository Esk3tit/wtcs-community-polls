The user explicitly wants direct symbolication evidence to close the launch hardening loop. REQUIREMENTS.md drift is a documentation hygiene problem flagged in the same Phase 5 verifier output.

R-02 fix (REVIEWS.md): The original plan put a `useEffect`-based throw in `AdminTabs.tsx` on main, gated by `?sentry-smoke=1` and AdminGuard. Codex flagged TWO problems:

  (a) **useEffect throws are not reliably caught by React.ErrorBoundary.** Effect errors are asynchronous; the ErrorBoundary contract only guarantees catch for render/lifecycle errors. The throw could bypass `AppErrorFallback` entirely — defeating the verification.

  (b) **Merging deliberate-error code to main, even temporarily and admin-gated, is operationally risky.** Even with rollback, the smoke trigger touches main.

This revision adopts **Option A** (Codex preferred): a Netlify deploy-preview-only smoke route on a feature branch that is DELETED after verification. The smoke trigger uses a render-phase throw inside an admin-only smoke component, NOT a useEffect throw — render-phase throws ARE caught by ErrorBoundary, exercising the same ErrorBoundary path users would see in a real production failure. Sentry sourcemap upload runs on every Vercel/Netlify build with the `sentryVitePlugin` configured (Phase 5 D-13), including deploy previews — so a deploy-preview Sentry issue is symbolicated against the same release-mapped sourcemaps as production would be.

Output: Sentry sourcemap human-verification entry closed with direct evidence captured from a Netlify deploy-preview build; deploy-preview branch deleted; REQUIREMENTS.md status column synced to actual phase artifacts; ZERO new product code, ZERO smoke-trigger residue, on main.
