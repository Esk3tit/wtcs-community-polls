<objective>
Diagnose-first auth bug investigation per D-01 and D-02. Phase 6 is a cleanup phase — we ship instrumentation (Sentry breadcrumbs + ?debug=auth overlay) and a manual reproduction log BEFORE making any guess-driven code change. The targeted fix (if any) is committed in this plan after the overlay surfaces a root cause; if the bug is environmental (extension / 3P-cookie setting / profile state), we document that and close the bucket without a code-level fix.

Purpose: The Discord OAuth login fails in the user's main browser but succeeds in incognito on the same machine. Differential is necessarily client-side state (cookies, localStorage, service workers, extensions). RESEARCH.md Pitfall 4 hypothesizes the strongest signal is PKCE *-code-verifier loss in localStorage. The diagnose-first work makes that signal observable IN THE FAILING ENVIRONMENT (production).

R-01 fix: The bug only reproduces against `polls.wtcsmapban.com` (production), NOT local dev or incognito. A pure `import.meta.env.DEV` gate is FALSE on Netlify production builds, so the original gate would have made the diagnostic tool unreachable in the only environment where the bug manifests. This revision replaces that gate with a multi-condition predicate that allows the overlay on production for an explicitly-toggled browser, while keeping it invisible to public users who have not opted in.

Output: Sentry breadcrumbs across the entire auth lifecycle, a ?debug=auth overlay reachable on production via a per-browser localStorage opt-in, a manual reproduction log file, and either a targeted fix commit or a documented environmental finding.
</objective>
