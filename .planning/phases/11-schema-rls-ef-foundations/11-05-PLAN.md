<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local dev → production Supabase | The deploy commands run by the user/CI cross this boundary; misdirected pushes (e.g., to a staging project) would corrupt the prod schema |
| Migration 10 → prod DB | Single atomic apply; any failure leaves the prod DB in an inconsistent intermediate state if statements are non-transactional |
| EF deploy → prod Functions runtime | New + retrofitted EFs are public-facing once deployed; an EF that fails to import `_shared/audit.ts` would 500 at runtime, breaking admin operations |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-DEPLOY-WRONG-PROJECT | Tampering | `supabase db push` linked project | mitigate | Task 05-02 step 1 explicitly verifies the linked project ref via `supabase projects list` before push. The polls.wtcsmapban.com project is the only valid push target for this phase. |
| T-MIGRATION-PARTIAL-APPLY | Tampering | Migration 10 apply | mitigate | Migration 10 is a single SQL file; Supabase wraps each migration in a transaction (verified in past v1.0 migrations 5, 7, 8, 9). If any statement fails, the whole migration rolls back — no partial state. Task 05-01 local `db reset` proves the full migration applies cleanly first. |
| T-EF-IMPORT-DRIFT | Tampering | 13 EFs depend on `_shared/audit.ts` | mitigate | All 13 deploys run sequentially; a deploy failure on any single EF aborts the loop (`|| exit 1`). The shared helper is deployed implicitly with each EF (Deno bundles imports). Smoke curl confirms the new EF returns 200 (not 500-on-import-error). |
| T-AUTH-LEAK-IN-LOG | Information Disclosure | Smoke curl logs admin JWT | mitigate | Smoke test uses a real admin JWT, which is sensitive. The SUMMARY captures evidence as `Bearer eyJ...` truncated to first 10 chars; full token is NOT pasted into the SUMMARY or commit messages. |
| T-AUDIT-SMOKE-LEAK | Information Disclosure | Smoke audit rows in prod audit_log | accept | The smoke test creates 2 audit rows on the test poll. These rows persist (audit_log is append-only by design). Acceptable because the audit_log is admin-only-visible and the rows are accurately attributed to the admin who ran the smoke. |
| T-ROLLBACK-RISK | Denial of Service | Mis-applied migration 10 | mitigate (procedural) | If smoke test fails: do NOT attempt to manually DROP the new policy / view / table — write a rollback migration 10.1 that reverses the changes (DROP POLICY + restore old policy from migration 5; ALTER TABLE polls DROP COLUMN; DROP TABLE audit_log) and push it. Plan 05 itself does NOT contain a rollback SQL because rollback design depends on the failure mode; surface as a hotfix plan when needed. |
</threat_model>
