<objective>
Retrofit all 12 existing mutation admin EFs (per D-05) so each one emits an `audit_log` row on every state change. The new `toggle-results-visibility` is already audit-emitting from Plan 02; this plan brings the pre-existing surface to parity, giving v1.2 a 13-emitter audit floor. `create-poll` is additionally extended to accept an optional `results_hidden?: boolean` body field per D-08/D-09/D-10.

Purpose: VIS-02 requires every change to write a new `audit_log` row. Phase 11 D-05 expands this to the full mutation surface (12 existing EFs + the new toggle). Without this retrofit, `audit_log` would only capture results-visibility toggles — a gap that defeats the forensics purpose of the table.

Output: 12 EF files modified. Each gets one new import line and one new `writeAudit(...)` call per state-changing path. `create-poll` additionally gets the optional `results_hidden` body field plumbed through to the INSERT/RPC payload, with a conditional second audit row on `results_hidden = true` at creation. Zero changes to response shapes. Zero changes to existing validation, error handling, CORS, or admin-gate code.
</objective>
