    ```yaml
    name: CI

    on:
      pull_request:
        branches: [main]
      push:
        branches: [main]

    concurrency:
      group: ci-${{ github.ref }}
      cancel-in-progress: true

    jobs:
      lint-and-unit:
        runs-on: ubuntu-latest
        timeout-minutes: 10
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: '22'
              cache: 'npm'
          # D-16 §2 — npm ci, never npm install. Fails if lockfile and package.json disagree.
          - run: npm ci
          - run: npm run lint
          - run: npm test -- --run
          # D-16 §4 — non-blocking supply-chain check.
          - name: npm audit (non-blocking)
            run: npm audit --audit-level=high || true

      e2e:
        runs-on: ubuntu-latest
        timeout-minutes: 15
        needs: lint-and-unit
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: '22'
              cache: 'npm'
          - run: npm ci
          - uses: supabase/setup-cli@v1
            with:
              version: 2.92.1
          # M4 — canonical seed flow step 1: `supabase start` bootstraps migrations
          # and applies supabase/seed.sql automatically. NO `supabase db reset`
          # needed; it's implicit on the first start.
          - name: Start Supabase local stack (applies migrations + supabase/seed.sql)
            run: supabase start
          - name: Wait for stack ready
            run: |
              for i in {1..30}; do
                if supabase status 2>/dev/null | grep -q 'API URL'; then
                  echo "Supabase ready after ${i}s"
                  exit 0
                fi
                sleep 2
              done
              echo "Supabase did not become ready in 60s"
              supabase status
              exit 1
          # M5 — derive local keys from `supabase status --output json` instead
          # of storing them as GH secrets. These are deterministic fixed
          # defaults; storing them as secrets is unnecessary complexity.
          - name: Derive local Supabase keys from `supabase status`
            id: supabase-keys
            run: |
              STATUS=$(supabase status --output json)
              ANON_KEY=$(echo "$STATUS" | jq -r '.API.ANON_KEY')
              SERVICE_ROLE_KEY=$(echo "$STATUS" | jq -r '.API.SERVICE_ROLE_KEY')
              DB_URL=$(echo "$STATUS" | jq -r '.DB.URL // .API.DB_URL // empty')
              if [ -z "$DB_URL" ]; then
                # Fallback: parse the text output if the JSON shape differs by CLI version.
                DB_URL=$(supabase status | awk '/DB URL/ {print $3}')
              fi
              echo "::add-mask::$ANON_KEY"
              echo "::add-mask::$SERVICE_ROLE_KEY"
              echo "anon_key=$ANON_KEY"         >> "$GITHUB_OUTPUT"
              echo "service_role_key=$SERVICE_ROLE_KEY" >> "$GITHUB_OUTPUT"
              echo "db_url=$DB_URL"             >> "$GITHUB_OUTPUT"
          # M4 — canonical seed flow step 2: apply the additive E2E fixture seed
          # on top of the base seed that `supabase start` just applied.
          - name: Apply fixture seed (additive, on top of supabase/seed.sql)
            run: psql "${{ steps.supabase-keys.outputs.db_url }}" -f e2e/fixtures/seed.sql
          - name: Install Playwright browsers
            run: npx playwright install --with-deps chromium
          - name: Build app
            env:
              VITE_SUPABASE_URL: http://localhost:54321
              VITE_SUPABASE_ANON_KEY: ${{ steps.supabase-keys.outputs.anon_key }}
            run: npm run build
          - name: Start preview server
            env:
              VITE_SUPABASE_URL: http://localhost:54321
              VITE_SUPABASE_ANON_KEY: ${{ steps.supabase-keys.outputs.anon_key }}
            run: npx vite preview --host 0.0.0.0 --port 4173 &
            shell: bash
          - name: Wait for preview
            run: |
              for i in {1..30}; do
                if curl -sf http://localhost:4173 >/dev/null; then
                  echo "Preview ready"; exit 0
                fi
                sleep 1
              done
              exit 1
          - name: Run Playwright smoke suite
            env:
              PLAYWRIGHT_BASE_URL: http://localhost:4173
              VITE_SUPABASE_URL: http://localhost:54321
              VITE_SUPABASE_ANON_KEY: ${{ steps.supabase-keys.outputs.anon_key }}
              # Service-role key exposed ONLY to the test job env; helper does NOT use it (HIGH #2 in 05-05),
              # but any seeding-style test utilities that do will read it from here.
              SUPABASE_SERVICE_ROLE_KEY: ${{ steps.supabase-keys.outputs.service_role_key }}
            run: npx playwright test --grep @smoke
          - name: Upload Playwright report on failure
            if: failure()
            uses: actions/upload-artifact@v4
            with:
              name: playwright-report
              path: playwright-report/
              retention-days: 7
    ```

<security>
## Threats Mitigated

| Threat ID | Description | Mitigation |
|-----------|-------------|------------|
| T-05-11-cron-secret-in-logs | Secrets echoed to workflow logs | Uses `${{ secrets.* }}` references (auto-masked) + `::add-mask::` on derived keys; no `echo "$SECRET"`; `\|\| true` on `npm audit` avoids noisy log capture that could incentivize grep-to-diagnose workflows |

## Residual Risks
- Local fixed-default keys are derived via `supabase status` at runtime. If a future Supabase CLI version changes the JSON shape, the key-extraction step needs updating. Detected immediately (step fails → job fails → PR blocked).
- `npm audit` non-blocking means a HIGH advisory shows up as a green run. Deliberate per D-16 §4 ("initial threshold is non-blocking warning to avoid breaking launch"). Promotable later.
</security>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GH runner → Supabase local stack | Ephemeral; fixed-default keys derived at runtime |
| GH runner → npm registry | `npm ci` is lockfile-strict |
| Secrets store → workflow env | Auto-masked in logs (native); `::add-mask::` used for derived values |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-05-11 | Information Disclosure | workflow logs | mitigate | `::add-mask::` on derived keys; no echo of values |
| T-05-12 | Denial of Service | CI job flakiness | accept | Retry `supabase start` ready-check; fail fast if 60s elapsed |
</threat_model>
