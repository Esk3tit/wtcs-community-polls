  <action>
    ```yaml
    name: Daily close-expired-polls sweep

    on:
      schedule:
        # 03:00 UTC daily — avoids midnight-UTC queue storm (Pitfall 1).
        # Satisfies INFR-02 (each run is a real DB write via the EF sweep → 7× headroom vs 7-day pause).
        - cron: '0 3 * * *'
      workflow_dispatch:

    concurrency:
      group: cron-sweep
      cancel-in-progress: false

    jobs:
      sweep:
        runs-on: ubuntu-latest
        timeout-minutes: 5
        # NO continue-on-error — we want GH's default email-on-failure to fire (D-03).
        steps:
          - name: Invoke close-expired-polls
            env:
              SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
              CLOSE_SWEEPER_SECRET: ${{ secrets.CLOSE_SWEEPER_SECRET }}
            run: |
              set -euo pipefail
              response=$(curl -sS -w "\n%{http_code}" -X POST \
                -H "Content-Type: application/json" \
                -H "X-Cron-Secret: ${CLOSE_SWEEPER_SECRET}" \
                "${SUPABASE_URL}/functions/v1/close-expired-polls")
              http_code=$(echo "$response" | tail -n1)
              body=$(echo "$response" | sed '$d')
              echo "HTTP ${http_code}: ${body}"
              if [ "${http_code}" != "200" ]; then
                echo "Sweep failed — see body above. 503 = CLOSE_SWEEPER_SECRET not set; 401 = wrong secret; anything else = EF error."
                exit 1
              fi
    ```

    Note: per 04-02 decisions, `close-expired-polls` also requires an
    `Authorization: Bearer <ANON_KEY>` header for Supabase's default EF gateway
    to route the request. Verify by reading the EF source: if the EF expects
    the `authorization` header, extend the curl with:
    ```
    -H "Authorization: Bearer ${{ secrets.SUPABASE_ANON_KEY }}"
    ```
    and add `SUPABASE_ANON_KEY` to the env block + acceptance criteria. If the
    EF is deployed `--no-verify-jwt`, the Authorization header is optional.

    Check `supabase/functions/close-expired-polls/index.ts` and the deploy
    flags being used in Plan 05-06's deploy workflow — if `--no-verify-jwt` is
    NOT set, the Authorization header is required for the curl to pass the
    gateway. Default assumption for this plan: include Authorization for
    robustness (no-op if not required; essential if the EF is JWT-gated).
  </action>
