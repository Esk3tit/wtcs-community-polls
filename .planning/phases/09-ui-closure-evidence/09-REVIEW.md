## Warnings

### WR-01: Misleading comment in .gitignore — un-ignore wording overstates the parent-ignore problem

**File:** `.gitignore:41-47`
**Issue:** The 7-line comment block introduces the `!.planning/closure/artifacts/MANIFEST.json` rule as if the parent is ignored:

```gitignore
# F1 / Decision A: MANIFEST.json IS committed (sha256-pinned record of every
# binary artifact); only the binary subdirs themselves are ignored.
.planning/closure/artifacts/lighthouse/
.planning/closure/artifacts/screenshots/
!.planning/closure/artifacts/MANIFEST.json
```

The comment says "only the binary subdirs themselves are ignored" — which is correct — but then writes a `!`-prefixed un-ignore for a file that is **not** in any ignored directory. `git check-ignore -v .planning/closure/artifacts/MANIFEST.json` matches **no rule** (returns empty), confirming the file would be tracked even without the un-ignore line.

This is not broken behavior — git tolerates redundant un-ignores cleanly. But the rule shape suggests a misunderstanding of the well-known git footgun ("un-ignoring a file inside a directory that is itself ignored does NOT work because git skips the directory entirely"). Future maintainers reading this block may copy the pattern to other contexts where the parent IS ignored, and silently break their own rules.

**Fix:** Either (a) drop the redundant un-ignore line and rely on the leaf-subdir-only ignore semantics, OR (b) keep the line but tighten the comment to explicitly call out that it is defensive/intent-documenting:

```diff
 # Phase 9 audit harness artifacts (per-run binary outputs — not committed).
 # The summary .md evidence files in .planning/closure/UIDN-0[23]-*.md ARE committed.
-# F1 / Decision A: MANIFEST.json IS committed (sha256-pinned record of every
-# binary artifact); only the binary subdirs themselves are ignored.
+# F1 / Decision A: MANIFEST.json (sha256-pinned record of every binary
+# artifact) sits at .planning/closure/artifacts/MANIFEST.json — its parent
+# directory is NOT ignored, so it tracks normally. The !MANIFEST.json line
+# below is defensive (documents intent + survives a future "ignore the whole
+# artifacts/ tree" edit). Note: a `!file` rule cannot rescue a file whose
+# PARENT directory is ignored — git skips ignored dirs entirely.
 .planning/closure/artifacts/lighthouse/
 .planning/closure/artifacts/screenshots/
 !.planning/closure/artifacts/MANIFEST.json
```

Severity rationale: classified as WARNING (not BLOCKER) because the actual ignore behavior is correct and the manifest is verifiably tracked. The defect is purely *documentation accuracy* — a future maintainer extending this pattern to a context with an ignored parent could ship broken rules. Worth fixing but not blocking phase ship.
